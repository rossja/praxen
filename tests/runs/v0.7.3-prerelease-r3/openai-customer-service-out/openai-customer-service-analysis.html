Executive Summary
Agent Remit (as declared)
A customer-facing multi-agent system for an airline that handles FAQs about policies and services, looks up flight details, and updates seat assignments on existing confirmed reservations. A triage agent routes each request to either an FAQ agent — which answers from a curated dataset only — or a seat-booking agent that may call update_seat on the authoritative reservation system. The remit forbids ticket purchases, payments, refunds, rebooking, and any outbound channel beyond the in-session response, and requires identity verification, confirmation-number ownership checks, and durable audit logging of every seat change before mutating reservation state.
Behavior Summary (as observed)

The dominant pattern is framework ships safe primitives; example wires in none of them. The OpenAI Agents SDK provides InputGuardrail, OutputGuardrail, ToolInputGuardrail, ToolOutputGuardrail, FunctionTool.needs_approval, FunctionTool.is_enabled, and Handoff.input_filter — every one is set to its permissive default in examples/customer_service/main.py. The result is a customer-facing seat-mutation tool (update_seat) that accepts any confirmation number the LLM emits and writes it straight into the shared context with no ownership check, an on_seat_booking_handoff hook that fabricates a flight number with random.randint(100, 999) in direct violation of the remit's prohibition on agent-generated identifiers, and zero durable audit log for seat changes despite the remit requiring one for every mutation.

Underneath that, the agents' identity and scope live entirely in prompt strings — the triage agent's "you can use your tools to delegate" instruction is the only constraint on its behaviour, and there is no code-level enforcement that customer-supplied input cannot reach the FAQ tool's keyword switch or override an agent's role.

Scope of Analysis
Python implementation built on the openai-agents SDK, scoped to examples/customer_service/main.py plus the SDK surface the example wires together (Agent, Runner, handoff, function_tool, trace). Three Agent instances (triage_agent, faq_agent, seat_booking_agent) share a single AirlineAgentContext Pydantic model that the LLM-supplied confirmation number is written into directly inside update_seat() with no verification step. The handoff hook on_seat_booking_handoff() generates a fake flight number via random.randint() and writes it into the shared context. faq_lookup_tool is a keyword-substring switch over a hardcoded answer set. The SDK provides InputGuardrail / OutputGuardrail / ToolInputGuardrail / ToolOutputGuardrail / needs_approval / is_enabled / handoff input_filter, and RunConfig.tracing_disabled defaults to False; the example wires in none of the guardrail or approval primitives and depends on the SDK's default tracing for any record of what happened.

CRITICAL PRAX-2026-05-25-001 <code>update_seat</code> mutates reservation state with no authenticated-customer identity check and no ownership verification of the supplied confirmation number.
Implement Zero Trust ASI03 — Identity and Privilege Abuse ASI02 — Tool Misuse and Exploitation LLM06 — Excessive Agency
Policy Rule — R-05, R-10 (Worker Remit):
"Customer identity MUST be confirmed before any action that mutates reservation state. / Reservations or seat assignments for any customer other than the authenticated session user MUST NOT be modified."
examples/customer_service/main.py:67 — <code>@function_tool</code> decorator with default args; <code>needs_approval</code> defaults to <code>False</code> at <code>src/agents/tool.py:328</code> examples/customer_service/main.py:79 — <code>context.context.confirmation_number = confirmation_number</code> writes the LLM-supplied string with no caller-identity check examples/customer_service/main.py:80 — <code>context.context.seat_number = new_seat</code> mutates seat assignment immediately after
Recommended Action
  • Add a customer-identity check before any reservation mutation — load the authenticated principal from a session/auth source and refuse the tool call if absent.
  • Set needs_approval=True (or a callable that fires when the principal cannot be verified) on the update_seat @function_tool and use RunState.approve() / RunState.reject() to gate the mutation.
  • Verify the supplied confirmation_number belongs to the authenticated principal against an authoritative reservation lookup before any write.

Implement Zero Trust ASI02 — Tool Misuse and Exploitation LLM06 — Excessive Agency

"The seat-booking agent updates seat assignments on existing confirmed reservations only. / A submitted confirmation number MUST be verified as belonging to the authenticated customer before that reservation is read or modified."

CRITICAL PRAX-2026-05-25-003 <code>update_seat</code> writes any seat string the LLM produces with no availability or existence check.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation LLM06 — Excessive Agency

"Seat updates MUST be limited to seats that exist on the flight and are available — not already assigned to another passenger."
examples/customer_service/main.py:80 — <code>context.context.seat_number = new_seat</code> assigns without consulting a seat map examples/customer_service/main.py:82 — only post-condition is <code>assert context.context.flight_number is not None</code> examples/customer_service/main.py:83 — returns a success-shaped string regardless of validity

CRITICAL PRAX-2026-05-25-004 Seat changes are not recorded to any durable audit log capturing time, customer identity, confirmation number, old/new seat, and acting agent.
Monitor Continuously ASI02 — Tool Misuse and Exploitation
Policy Rule — R-08 (Worker Remit):
"Every seat change MUST be recorded to a durable audit log capturing the time, the customer identity, the confirmation number, the old and new seat, and the agent that performed the change."
examples/customer_service/main.py:67 — <code>update_seat()</code> body, lines 67-83 — mutates and returns with no log call examples/customer_service/main.py:166 — only telemetry is the SDK's <code>trace("Customer service", group_id=conversation_id)</code> per-turn span

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

SDK default-on tracing produces a per-turn span

main() wraps each turn in `with trace("Customer service", group_id=conversation_id)` and `RunConfig.tracing_disabled` defaults to `False`, so the SDK's tracing backend receives a structured span per turn with a stable conversation grouping. This is the only operational telemetry surface in the example and earns partial credit for Monitor Continuously.

examples/customer_service/main.py:166

Function-tool arguments are strict-mode JSON schemas by default

FunctionTool.strict_json_schema defaults to True and ensure_strict_json_schema runs in __post_init__, so tool argument shapes are validated before invocation. This narrows the model's surface to malformed-argument injection on these specific tools.

src/agents/tool.py:310

Handoff input JSON schema is always strict-mode

handoff() calls ensure_strict_json_schema(input_json_schema) unconditionally (src/agents/handoffs/__init__.py:309-310), so even when a custom input_type is provided the agent-to-agent handoff payload is shape-checked before the next agent runs.
