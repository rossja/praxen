Executive Summary
Agent Remit (as declared)
A natural-language interface to a single relational database, intended to answer read-only analytical questions for internal data consumers by translating user questions into SQL, executing them, and returning natural-language answers. The agent is authorized exactly four tools — sql_db_list_tables, sql_db_schema, sql_db_query_checker, and sql_db_query — against a pre-configured SQLAlchemy connection scoped to one database. Every data- or schema-modifying statement is forbidden outright with no approval path; the remit explicitly states defense-in-depth via a least-privilege read-only database role is an operator responsibility, and that the agent's own logic must not be the sole enforcement point for the prohibition on modifying the database.
Behavior Summary (as observed)

Pattern: policy declared in the prompt, zero code-level enforcement. The remit's read-only, no-DML-or-DDL, no-multi-statement, no-administrative prohibition is communicated to the agent only through the natural-language SQL_PREFIX string in agent_toolkits/sql/prompt.py ("DO NOT make any DML statements (INSERT, UPDATE, DELETE, DROP etc.)") — the execution path in tools/sql_database/tool.py and utilities/sql_database.py accepts any SQL the LLM emits and dispatches it through SQLAlchemy with no statement-class filter, no multi-statement guard, no row-limit enforcement, and no schema-scope check, so a single prompt-injection success in untrusted row content (column values, table comments, retrieved rows treated as context) is one tool-call away from a write or schema change.

The maintainer is unusually explicit about this — the create_sql_agent docstring carries an !!! warning admonition naming arbitrary SQL execution and prescribing operator-side controls (least-privilege role, server-side statement timeouts, query guardrails, schema restriction) — but the library itself ships no in-process safety net, so the agent is exactly as safe as the deploying operator's database role and server-side timeout policy, and silently as unsafe as a default pip install-and-go deployment would be.

Scope of Analysis
Python LangChain integration distributed as langchain-community v0.4.2; the SQL agent surface is three modules — agent_toolkits/sql/{base,toolkit,prompt}.py, tools/sql_database/{tool,prompt}.py, and utilities/sql_database.py. create_sql_agent() in base.py builds a langchain_classic.AgentExecutor with max_iterations=15 and early_stopping_method="force"; SQLDatabaseToolkit.get_tools() returns exactly the four tools named in the remit. QuerySQLDatabaseTool._run() forwards the LLM-emitted query string to SQLDatabase.run_no_throw() which dispatches through _execute() to a SQLAlchemy connection.execute(text(command)) with no statement-class parser, no DML/DDL filter, and no per-statement count check; SQLDatabase.run() truncates each result cell to max_string_length=300 by default. The prompt prefix (SQL_PREFIX) carries the remit's read-only intent and top_k row limit as natural-language instructions to the LLM, not as code-enforced gates, and the create_sql_agent docstring explicitly warns operators that the agent can run arbitrary SQL and that least-privilege roles, statement timeouts, and query guardrails must be applied server-side.
