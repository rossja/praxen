BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is the confirmation gate that protects the CLI's stated
promise is wired to the wrong condition. The anonymous-deploy warning +
Continue? [y/N]: prompt in deploy/commands.py (_deploy(), lines ~283–326) only
fires when frontend.enabled AND auth.provider == "anonymous"; a deploy whose
[auth] provider is anonymous without [frontend].enabled = true silently ships
the same permissive auth.py handler (AUTH_BLOCK_ANONYMOUS in
deploy/templates.py) that overrides LangSmith's default x-api-key requirement
— the API ends up open to anyone with the deploy URL with no warning and no
confirmation, directly violating the Stated Protection. Secondary, MCP
transport validation (_validate_mcp_for_deploy in deploy/config.py) checks
only that the transport is not stdio — it accepts plain http:// URLs and never
asserts TLS, contradicting the remit's "remote MCP servers MUST be reached
over TLS" clause; the same loader propagates whatever URL the project declares
into the bundle with no version-pinning or integrity-checking of the remote
server.
