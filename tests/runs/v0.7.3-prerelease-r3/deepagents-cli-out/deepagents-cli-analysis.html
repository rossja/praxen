Executive Summary
Agent Remit (as declared)
Deep Agents CLI is a local developer command-line tool that scaffolds, bundles, and ships a deepagents agent for deployment, running a local development server (dev) or shipping the bundle to a managed LangGraph deployment platform (deploy). The CLI's central promise is that the bundle it ships faithfully and only reflects the developer's reviewed project — its declared [agent]/[sandbox]/[auth]/[memories]/[frontend] configuration, AGENTS.md system prompt, skills/, mcp.json, and declared subagents — with no surface added or exposed that the developer did not configure. A deploy that would leave the API open to anyone with the URL MUST surface a prominent warning and obtain explicit operator confirmation before it proceeds; remote MCP servers carried into the bundle MUST be reached over TLS and validated before shipping; credentials MUST NOT be written into the project, committed, logged, or printed.
Behavior Summary (as observed)

The dominant pattern is the confirmation gate that protects the CLI's stated promise is wired to the wrong condition. The anonymous-deploy warning + Continue? [y/N]: prompt in deploy/commands.py (_deploy(), lines ~283–326) only fires when frontend.enabled AND auth.provider == "anonymous"; a deploy whose [auth] provider is anonymous without [frontend].enabled = true silently ships the same permissive auth.py handler (AUTH_BLOCK_ANONYMOUS in deploy/templates.py) that overrides LangSmith's default x-api-key requirement — the API ends up open to anyone with the deploy URL with no warning and no confirmation, directly violating the Stated Protection.

Secondary, MCP transport validation (_validate_mcp_for_deploy in deploy/config.py) checks only that the transport is not stdio — it accepts plain http:// URLs and never asserts TLS, contradicting the remit's "remote MCP servers MUST be reached over TLS" clause; the same loader propagates whatever URL the project declares into the bundle with no version-pinning or integrity-checking of the remote server.

Scope of Analysis
Python 3.11+ package deepagents-cli==0.1.2 shipping the deepagents / deepagents-cli console scripts. The entry point (deepagents_cli/main.py) is an argparse façade exposing three subcommands — init, dev, deploy — implemented in deepagents_cli/deploy/commands.py. The bundler in deepagents_cli/deploy/bundler.py reads a fixed project layout (deepagents.toml, AGENTS.md, .env, mcp.json, skills/, user/, subagents/) into a _seed.json payload, copies mcp.json verbatim as _mcp.json, generates deploy_graph.py from string templates in deploy/templates.py, and shells out to langgraph deploy. A ContextHubBackend (deploy/context_hub.py) talks to the LangSmith Hub for memory seeding. Config parsing in deploy/config.py uses strict allow-listed sections/keys, validates required env vars per provider, and rejects stdio transports in mcp.json for deploy. A repo-root .mcp.json declares two LangChain docs HTTP MCP servers used during repository development (not by the CLI's runtime).

HIGH PRAX-2026-05-25-001 Anonymous-auth deploy ships an open API without warning or confirmation when [frontend].enabled is not set.
Implement Zero Trust LLM06 — Excessive Agency ASI03 — Identity and Privilege Abuse
Policy Rule — R-01, R-03, R-08, R-09 (Worker Remit):
"The bundle MUST be assembled solely from the project's declared sources, and a deploy that would expose the agent's API without authentication MUST require explicit operator confirmation before it proceeds. / A deploy configured to leave the deployed API open to anyone with its URL MUST surface a prominent warning and obtain explicit operator confirmation before it proceeds. / The CLI MUST NOT silently mutate, generate, or deploy any agent surface the developer did not declare in the project. / A deploy that opens the deployed API without authentication MUST be confirmed by the operator before it runs; it MUST NOT proceed unattended."
libs/cli/deepagents_cli/deploy/commands.py:287 — is_anonymous = (config.frontend is not None and config.frontend.enabled and config.auth is not None and config.auth.provider == "anonymous") — the gate is conjoined on the frontend being enabled. libs/cli/deepagents_cli/deploy/templates.py:372 — AUTH_BLOCK_ANONYMOUS is generated whenever [auth].provider == "anonymous" regardless of [frontend]; get_current_user returns {"identity": "anonymous"} for every request, making the API open to anyone with the deploy URL.
Recommended Action
  • Decouple the warning + confirmation gate from [frontend].enabled: in commands.py:287-292, set is_anonymous = config.auth is not None and config.auth.provider == "anonymous" so the prompt fires whenever the anonymous auth handler will be shipped.
  • Add a deploy/test_commands.py test that asserts the prompt fires (and aborts on N) for the [auth] anonymous + no-[frontend] configuration, so the gate cannot regress silently.
HIGH PRAX-2026-05-25-002 MCP transport validation rejects only stdio; plain-http:// URLs pass and ship to deploy without TLS.
Implement Zero Trust LLM03 — Supply Chain All remote connections use TLS
Policy Rule — R-04 (Worker Remit):
"Remote MCP servers carried into the bundle MUST be reached over TLS, and their transport configuration MUST be validated before the bundle is shipped."
libs/cli/deepagents_cli/deploy/config.py:293 — transport = server_config.get("type", server_config.get("transport", "stdio")) followed by if transport == "stdio": is the only check; nothing inspects server_config["url"] for https://. libs/cli/deepagents_cli/deploy/templates.py:448 — runtime loader accepts http and sse schemes verbatim into MultiServerMCPClient connections without re-validating the scheme.
Recommended Action
In _validate_mcp_for_deploy, after the transport check, parse server_config.get("url") and append a validation error when the scheme is not https (or wss for SSE) — accept http://localhost / 127.0.0.1 only behind an explicit dev-mode opt-in.
HIGH PRAX-2026-05-25-003 Remote MCP servers are carried into the deployed bundle with no version pin or integrity check.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities Tool definitions are signed and version-pinned
Policy Rule — R-15 (Worker Remit):
"Remote MCP servers declared in the project MUST be pinned to a known-good, integrity-checked version, so the deployed agent does not auto-install an unpinned server afresh."
libs/cli/deepagents_cli/deploy/bundler.py:194 — if mcp_present: shutil.copy2(project_root / MCP_FILENAME, build_dir / "_mcp.json") — verbatim copy with no per-server integrity capture. libs/cli/deepagents_cli/deploy/templates.py:444 — generated _load_mcp_tools reads servers from _mcp.json and instantiates MultiServerMCPClient(connections) with no hash or signature verification.
Recommended Action
Extend _validate_mcp_for_deploy (and the bundler) to require either an integrity SHA-256 / SRI hash per server entry in mcp.json or an explicit pin field that captures a known-good tool-manifest digest, and refuse to bundle when neither is present.
HIGH PRAX-2026-05-25-004 CLI installs no logging handlers — deploy operations leave no durable, structured audit record on the operator's machine.
Monitor Continuously LLM02 — Sensitive Information Disclosure
libs/cli/deepagents_cli/config.py:18 — # The CLI does not configure logging handlers by default, so logger.warning output is invisible to end users. libs/cli/deepagents_cli/deploy/commands.py:572 — print("Deploying to LangSmith Deployments…") + subprocess.run(cmd, …) produces ephemeral stdout only; no structured record of the deploy event lands on disk.
Recommended Action
Add an opt-out file-backed audit logger in cli_main (main.py) that writes a JSON-Lines record per init / dev / deploy invocation (timestamp, subcommand, resolved config path, agent name, auth provider, confirmation outcome) to ~/.deepagents/cli.log, with a --no-audit flag for operators who explicitly opt out.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Bundler skips .env from _seed.json payload

The bundler explicitly does not fold .env contents into the _seed.json memory or skills payload (deploy/bundler.py:200-206); the file is copied alongside the bundle as .env so the LangGraph runtime reads it as environment, never as seeded memory — directly satisfying the remit's "Secret material MUST NOT be embedded into bundle artifacts that are not meant to carry it" clause.

libs/cli/deepagents_cli/deploy/bundler.py:200-206

Strict allow-listed config schema

deploy/config.py rejects unknown top-level sections (_ALLOWED_SECTIONS) and unknown keys within every section up front, so a malformed or hostile deepagents.toml cannot quietly carry undeclared content into the bundle — the operator is told exactly what's wrong before any artifact is generated.

libs/cli/deepagents_cli/deploy/config.py:447-466

Committed pinned lockfile and dependabot

libs/cli/uv.lock (~4031 lines) is committed alongside pyproject.toml's bounded version ranges, and .github/dependabot.yml is configured to keep them current — the remit's "Dependencies MUST be version-controlled with a committed, pinned lockfile" is fully satisfied for the CLI's own dependencies.

libs/cli/uv.lock

stdio MCP transport rejected at deploy time

_validate_mcp_for_deploy (deploy/config.py:281-301) catches stdio MCP transports — which cannot run in the deployed runtime — at validation time, returning a clear error rather than silently producing a broken bundle.

libs/cli/deepagents_cli/deploy/config.py:281-301

Frontend runtime config escapes < to prevent </script> breakout

_build_runtime_config_json JSON-encodes the frontend runtime config and post-replaces < with \u003c before injecting it into the inline <script> tag (deploy/bundler.py:128), defending against a hostile subtitle/appName/prompts string that contains </script>.

libs/cli/deepagents_cli/deploy/bundler.py:127-128

_seed_hub_repo short-circuits on existing commits

Hub seeding uses backend.has_prior_commits() as a gate so a redeploy never silently overwrites user edits in the LangSmith Hub repo (deploy/commands.py:520-522), preserving operator changes across CLI invocations.

libs/cli/deepagents_cli/deploy/commands.py:520-522

Overwriting an existing project folder during init requires --force

_init_project checks project_dir.exists() and not force and aborts with "Error: <name>/ already exists. Use --force to overwrite." before any file is written, satisfying the remit's scaffolding-overwrite approval requirement.

libs/cli/deepagents_cli/deploy/commands.py:197-199
