Executive Summary
Agent Remit (as declared)
The execution tier of a two-agent "generator/executor" pattern — receives a code block from a paired LLM agent and runs it inside a configured environment (local host, Docker container, Jupyter kernel, container-hosted Jupyter, or Azure dynamic-sessions sandbox), returning stdout/stderr/exit status for the agent's next reasoning step. Every executor MUST confine file I/O to a configured work directory, enforce a wall-clock timeout, capture all output, isolate execution from the host caller's environment, and record every execution to an audit log keyed by timestamp / executor kind / language / source agent / work directory / timeout / exit status / code digest. The local executor in production, network egress beyond an allow-list, host-volume mounts outside the work directory, and any raising of CPU/memory/timeout ceilings all MUST require human approval.
Behavior Summary (as observed)
The dominant pattern is sandbox has the shape of isolation but not the substance. The remit declares six MUST clauses every executor must enforce — work-directory confinement, timeout, full output capture, host isolation, session-state non-leakage, and a structured audit log — and code in autogen-ext/code_executors implements only the second (timeout) consistently. LocalCommandLineCodeExecutor copies os.environ wholesale into the child process (local/__init__.py:397), gating safety on a one-shot warnings.warn with no approval path; DockerCommandLineCodeExecutor creates its container with no user=, no read_only=, no cap_drop=, no mem_limit=, no pids_limit=, and no network isolation (docker/_docker_code_executor.py:537-550); create_default_code_executor() silently downgrades to the local path on any Docker failure (code_executors/__init__.py:58-80); DockerJupyterServer chmod 0o777s the bind mount (docker_jupyter/_jupyter_server.py:328); and no executor anywhere in the tree writes an audit-log entry of any kind. Defaults the remit treats as forbidden are the defaults the code ships with.
Scope of Analysis
Two Python packages — autogen-core (the abstract CodeExecutor base in code_executor/_base.py) and autogen-ext (five concrete executors under code_executors/: local, docker, docker_jupyter, jupyter, and azure). Each implements execute_code_blocks(), writing the code to a file in the work directory and invoking it via asyncio.create_subprocess_exec (local), container.exec_run with a timeout shell wrapper (docker), an in-process NotebookClient kernel (jupyter), a containerized Jupyter kernel gateway over WebSocket (docker_jupyter), or an HTTPS POST to an Azure Dynamic Sessions endpoint (azure). A top-level helper create_default_code_executor() in code_executors/__init__.py selects Docker if available and otherwise instantiates the local executor after emitting a UserWarning. There is no audit-log surface, no approval-gate surface, no work-directory enforcement on the Jupyter and Docker-Jupyter executors' code path, no container hardening on the Docker paths, and no environment scrubbing on the local path — every "MUST" in the remit's behavioral block is documentation, not enforced code.

CRITICAL PRAX-2026-05-25-001 LocalCommandLineCodeExecutor copies the parent process's full os.environ into the executed subprocess, leaking credentials and configuration to LLM-generated code.
Implement Zero Trust LLM02 — Sensitive Information Disclosure ASI03 — Identity and Privilege Abuse
Policy Rule — R-04 (Worker Remit):
"The execution environment MUST be isolated from the host caller — a failing or malicious execution MUST NOT be able to read or modify the parent process's environment, credentials, or state."
local/__init__.py:397 — `env = os.environ.copy()` — full parent environment copied into the subprocess with no filtering; the executed Python or shell script can `os.environ.get("OPENAI_API_KEY")` and exfiltrate it. local/__init__.py:399 — lines 399-400 — only modification is prepending the venv `bin_path` to `PATH`; nothing is removed.
Recommended Action
  • Replace the unconditional os.environ.copy() with an explicit allow-list of harmless variables (PATH, HOME, LANG, TERM, USER) and require operators to opt in additional variables via a constructor parameter.
  • Document the allow-list in the executor's docstring and emit a one-time warning when the operator passes additional variables.
CRITICAL PRAX-2026-05-25-002 DockerCommandLineCodeExecutor creates containers with no user, capability, filesystem, memory, pid, or network hardening — the container runs as root with full default capabilities.
Implement Zero Trust ASI05 — Unexpected Code Execution (RCE) ASI03 — Identity and Privilege Abuse
Policy Rule — R-07 (Worker Remit):
"Code MUST NOT be executed with host-level privileges when a less-privileged option achieves the same task."
docker/_docker_code_executor.py:537 — `self._container = await asyncio.to_thread(client.containers.create, ...)` — container creation block (lines 537-550) has no hardening keyword arguments. docker/_docker_code_executor.py:71 — default image is `python:3-slim`, whose `USER` is root.
Recommended Action
  • Pass safe defaults to containers.create: user="1000:1000", read_only=True, cap_drop=["ALL"], mem_limit="512m", pids_limit=256, network_disabled=True, security_opt=["no-new-privileges"]; make each one a constructor parameter so operators can relax individually, not all-or-nothing.
  • Add a constructor parameter network (default "none") that maps to network_mode, and require explicit opt-in for non-isolated networking.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Wall-clock timeout consistently enforced

Every executor enforces a configurable <code>timeout</code> with a floor of 1 second, and the command-line variants terminate the subprocess (or rely on the Docker <code>timeout</code> wrapper) on expiry. The remit's "every code execution MUST be subject to a configured wall-clock timeout" clause is the one MUST that is honoured everywhere.

local/__init__.py:441-449, docker/_docker_code_executor.py:360

Workspace-relative filename check is correctly implemented in _common.get_file_name_from_content

The helper at <code>_common.py:96-111</code> resolves the supplied filename against the workspace and calls <code>Path.relative_to(workspace_path.resolve())</code>, which raises <code>ValueError</code> on parent-directory or absolute-path escape. The command-line executors call it and return a structured error on escape attempts — this is the right shape, and the Jupyter / Azure paths should reuse it.

_common.py:96-111

Cancellation tokens are wired through every executor

Every <code>execute_code_blocks</code> implementation accepts a <code>CancellationToken</code> and uses <code>cancellation_token.link_future(task)</code> so operator-initiated cancel propagates to the running subprocess / container exec / kernel call. This gives the operator application a clean halt path.

local/__init__.py:436, docker/_docker_code_executor.py:297, azure/_azure_container_code_executor.py:469
