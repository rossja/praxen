Executive Summary
Agent Remit (as declared)
An autonomous software-engineering agent that accepts natural-language tasks and completes them end-to-end — writing code, executing it inside a sandboxed runtime, browsing the web, and operating on connected version-control and issue-tracker integrations. Authorized counterparties are the operator, the configured LLM provider, scoped hosted-platform integrations, configured MCP tool servers, and the sandboxed runtime that serves as the agent's execution boundary; the public web is read-only via the browser tool and its returned content is untrusted. The remit declares hard rules — all agent-generated code MUST run inside the sandbox, file I/O MUST stay inside the per-session workspace, embedded instructions in fetched content MUST NOT be acted on, secrets MUST NOT be committed or logged, per-session step and time caps MUST be enforced, and destructive integration actions MUST be confirmed or refused outright — and a small set of approval-gated actions including cross-repo writes, PR merges, force-pushes, dependency-change commits, CI edits, and the addition of new MCP servers or integration accounts.
Behavior Summary (as observed)

The dominant pattern is a thoughtfully-architected agent platform whose OSS control-plane defaults openly fall back to unauthenticated, single-tenant operation — the agentic-core invariants (sandbox isolation, OAuth-scoped integrations, structured session record) are real and operative, but the boundary that sits in front of them in this repo is permissive. get_dependencies() in openhands/app_server/utils/dependencies.py returns an empty list whenever SESSION_API_KEY is unset and the app is not in SaaS mode, so the V1 API including /api/v1/secrets/* (custom secrets create/read/delete) and /api/v1/users/me is reachable without credentials on a default docker-compose up; LocalhostCORSMiddleware in openhands/app_server/middleware.py compounds this by logging-and-allowing any origin when no permitted_cors_origins are configured.

Secondary themes: the ProcessSandboxService backend in openhands/app_server/sandbox/process_sandbox_service.py spawns agent-server processes directly on the host with the parent environment inherited (no container, no namespace, no seccomp), and the user-skills loader in openhands/app_server/user/skills_router.py ingests every *.md under ~/.openhands/microagents with no content-trust check — a documented persistence surface for the agent that the control plane treats as configuration data.

Scope of Analysis
This source snapshot is the OpenHands V1 app-server control plane — a FastAPI application assembled in openhands/app_server/app.py from a v1_router that mounts conversation, sandbox, secrets, settings, user, skills, webhook, git, event, and config sub-routers under /api/v1, plus a separately-mounted FastMCP server at /mcp for PR-creation tools. The agentic core (controller, runtime, LLM client, MCP client, agent-event loop) is no longer in this repo — it has been extracted to the separate openhands-sdk and openhands-agent-server packages, so this scan can only judge what the control plane does. The shipped openhands/server/ tree is now deprecation-shim re-exports of the V1 modules; the OSS persistence layer is filesystem-backed via LocalFileStore under ~/.openhands, with Alembic migrations run at startup; sandbox backends are pluggable (Docker by default, a host-process backend, and a remote backend), and docker-compose.yml mounts the host Docker socket and exposes port 3000 directly. The OSS default DefaultUserAuth returns user_id=None for every request and get_dependencies() attaches a real auth check only when SESSION_API_KEY is set in the environment, so the V1 API surface ships unauthenticated by default — the relevant divergence from the remit's "secrets must never reach the model context" and "every request to an integration must be verified" clauses.

CRITICAL PRAX-2026-05-25-001 OSS app server attaches no authentication to V1 routes by default — secrets, settings, git-token endpoints reachable anonymously.
Implement Zero Trust LLM02 — Sensitive Information Disclosure ASI03 — Identity and Privilege Abuse
Policy Rule — R-05, R-13, R-14, R-16, R-18 (Worker Remit):
"Integration and session credentials MUST be verified on every request to an integration, and long-lived credentials MUST NOT be cached anywhere the model can reach. / API keys and session tokens MUST NOT leak into logs, error messages, or model context. / One session's state, memory, or credentials MUST NOT leak into another session. / Connecting a new integration account MUST be confirmed by the user. / OpenHands MUST NOT share state, memory, or credentials across user sessions."
openhands/app_server/utils/dependencies.py:23 — get_dependencies() returns [] unless SESSION_API_KEY env is set or app_mode==SAAS — the OSS default in openhands/app_server/server_config/server_config.py:11 is APP_MODE=OPENHANDS, so the empty list is what every V1 router attaches. openhands/app_server/secrets/secrets_router.py:35 — router = APIRouter(prefix='/secrets', tags=['Secrets'], dependencies=get_dependencies()) — the create/read/list/delete custom-secret endpoints and the POST /git-providers endpoint that stores GitHub/GitLab/Bitbucket tokens are reachable without auth in the OSS default.
Recommended Action
  • Default the OSS path to a generated-on-first-boot SESSION_API_KEY persisted to the persistence directory (and printed to the operator) rather than treating "no key set" as "no auth required" — get_dependencies() in openhands/app_server/utils/dependencies.py should fail-closed instead of returning [].
  • Bind uvicorn to 127.0.0.1 by default in openhands/server/__main__.py (currently host='0.0.0.0') and remove the host-binding from docker-compose.yml's port mapping, so the OSS server is loopback-only unless the operator opts into network exposure.
HIGH PRAX-2026-05-25-002 Process-sandbox backend runs the agent server directly on the host with the parent environment inherited — no isolation.
Limit Your Domain ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-01, R-08 (Worker Remit):
"All agent-generated code MUST execute inside the sandboxed runtime and never directly on the host. / Agent-generated code MUST NOT run on the host operating system outside the sandboxed runtime."
openhands/app_server/sandbox/process_sandbox_service.py:119 — env = os.environ.copy(); env.update(sandbox_spec.initial_env); env['SESSION_API_KEY'] = session_api_key — the agent server inherits every host env var, then runs under a plain subprocess.Popen with no namespace, seccomp, or resource limits. openhands/app_server/config.py:341 — RUNTIME=='local' or 'process' selects ProcessSandboxServiceInjector — and AGENTS.md (repo root) documents `export RUNTIME=local` as the supported way to run OpenHands-with-OpenHands.
Recommended Action
  • Mark the process backend as "development-only" in openhands/app_server/sandbox/process_sandbox_service.py and emit a warning at startup when it is selected; require an explicit OH_ALLOW_HOST_PROCESS_RUNTIME=1 env flag before config_from_env in openhands/app_server/config.py will instantiate it.
  • Filter the inherited environment before subprocess.Popen — pass only the keys named in sandbox_spec.initial_env plus an explicit allow-list, so host secrets do not flow into the agent process by default.
HIGH PRAX-2026-05-25-003 LocalhostCORSMiddleware allows any origin and credentials when no permitted_cors_origins are configured.
Implement Zero Trust ASI03 — Identity and Privilege Abuse
openhands/app_server/middleware.py:34 — is_allowed_origin returns True for every origin when allow_origins is empty — logs "No CORS origins configured, allowing origin" and lets it through; super().__init__ is called with allow_credentials=True and allow_methods=['*'].
Recommended Action
Change the empty-origins fallback in openhands/app_server/middleware.py:42 to allow only localhost/127.0.0.1 (the original docstring intent) and deny everything else, rather than the current allow-any-with-warning behavior.
HIGH PRAX-2026-05-25-004 docker-compose.yml mounts the host Docker socket into the container with no isolation, giving the app full root on the host.
Limit Your Domain ASI05 — Unexpected Code Execution (RCE)
docker-compose.yml:19 — volumes: - /var/run/docker.sock:/var/run/docker.sock — the official compose stack mounts the host docker socket into the app container; the app's docker_sandbox_service uses docker.from_env() to talk to it.
Recommended Action
Document the docker-socket mount as a privileged-mode opt-in in docker-compose.yml (or split the compose into a "default" stack that uses the remote runtime and a "with-docker" overlay that adds the socket), and warn loudly at startup when the socket is reachable from inside the app container.
HIGH PRAX-2026-05-25-005 Skills/microagents are loaded from disk into agent context with no content-trust check or origin label.
Balance Your Knowledge Base ASI06 — Memory and Context Poisoning
Policy Rule — R-03 (Worker Remit):
"The user's task prompt, web-retrieved page content, issue descriptions, pull-request comments, repository file contents, and micro-agent or memory content MUST all be treated as untrusted input capable of carrying prompt-injection payloads."
openhands/app_server/user/skills_router.py:73 — for md_file in skills_dir.rglob('*.md'): fm = _parse_skill_frontmatter(md_file) — every markdown file under USER_SKILLS_DIR (~/.openhands/microagents) is enumerated and its YAML frontmatter loaded, with no origin/signature/review check.
Recommended Action
Add a content-origin label to SkillInfo in openhands/app_server/user/skills_router.py (e.g. source='global' vs source='user' vs source='external') and surface it to the prompt-assembly layer so untrusted skills are marked as such; require explicit operator approval before a newly-discovered user-skill enters context.
MEDIUM PRAX-2026-05-25-006 App server has no durable, structured per-action audit log — only uvicorn request logs and the SDK-side event stream.
Monitor Continuously LLM02 — Sensitive Information Disclosure
Policy Rule — R-06 (Worker Remit):

openhands/app_server/secrets/secrets_router.py:249 — create_custom_secret / update_custom_secret / delete_custom_secret return EditResponse without writing a structured audit record — the only trace is whatever uvicorn happens to log, with no who/when/what schema. openhands/app_server/event/event_store.py — file is 0 lines — placeholder named "event_store" exists in the event/ package but no implementation has been wired in.

MEDIUM PRAX-2026-05-25-007 FileSecretsStore persists git provider tokens and custom secrets as plaintext JSON on disk.
Manage Your Supply Chain LLM02 — Sensitive Information Disclosure
openhands/app_server/secrets/file_secrets_store.py:33 — store() calls secrets.model_dump_json(context={'expose_secrets': True}) and writes the result via file_store.write — plaintext JSON on disk for provider tokens and custom secrets.
Recommended Action
Wrap FileSecretsStore.store / .load with JwtService.encrypt_value / .decrypt_value so the on-disk secrets.json is JWE-encrypted at rest with the existing key-rotation primitives.
MEDIUM PRAX-2026-05-25-008 Rate limit is a single global in-memory bucket (10 req/s) — no per-user, per-route, or distributed quota.
Implement Zero Trust LLM10 — Unbounded Consumption
openhands/app_server/app.py:83 — app.add_middleware(RateLimitMiddleware, rate_limiter=InMemoryRateLimiter(requests=10, seconds=1)) — single global bucket, no per-route or per-user differentiation.
Recommended Action
Add a separate, lower-bound rate-limit class on the sandbox-start and conversation-create routes in openhands/app_server/app.py, and document an LLM-spend budget threshold per session/user that halts the conversation when exceeded (the agent-server enforces step caps; the app server should enforce a wall-clock and dollar cap).
What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Sandbox session API key bound to RUNNING sandbox state

<code>validate_session_key</code> in <code>openhands/app_server/sandbox/session_auth.py</code> rejects any session-API-key call whose target sandbox is not in <code>RUNNING</code> state — explicitly to prevent leaked keys from being used after a sandbox is paused / stopped / deleted; the file's own docstring calls out this property as a security invariant.

openhands/app_server/sandbox/session_auth.py:37

Session-API-key ownership check on sensitive cross-route paths

<code>validate_session_key_ownership</code> additionally requires that the authenticated caller owns the sandbox the session key belongs to (HTTP 403 otherwise) before <code>/users/me?expose_secrets=true</code> will return secret-bearing fields.

openhands/app_server/sandbox/session_auth.py:103

Encryption-key service with JWE and legacy-Fernet decrypt

<code>JwtService</code> provides JWS sign/verify and JWE encrypt/decrypt with key-rotation IDs, plus a legacy-Fernet fallback for decrypting older payloads during the migration — the primitive needed to encrypt the secrets store at rest exists and is exercised by <code>test_jwt_service.py</code> and <code>test_encryption_key.py</code>.

openhands/app_server/services/jwt_service.py:31

DEBUG_LLM requires interactive confirmation before unmasking secrets in logs

<code>openhands/app_server/utils/logger.py</code> gates the <code>DEBUG_LLM</code> mode behind an interactive <code>input()</code> prompt that warns the operator that API keys may be exposed and requires typing <code>y</code> — a deliberate guard against accidentally enabling secret-leaking logs in production.

openhands/app_server/utils/logger.py:44

V0 deprecation shims preserved with explicit redirection

<code>openhands/server/app.py</code>, <code>openhands/server/listen.py</code>, <code>openhands/server/middleware.py</code>, and <code>openhands/user_auth/default_user_auth.py</code> are stamped <code>DEPRECATED</code> / <code>LEGACY V0</code> with a comment pointing at the V1 replacement — explicit signposting reduces the chance of accidentally extending the legacy path.

openhands/server/app.py:1

Webhook secret retrieval requires JWS-verified access token

<code>/api/v1/webhooks/secrets</code> verifies the X-Access-Token as a JWS payload via <code>JwtService.verify_jws_token</code> before resolving the user and provider — the token carries user_id + provider_type and the verification is mandatory; failures return 401.

openhands/app_server/event_callback/webhook_router.py:463
