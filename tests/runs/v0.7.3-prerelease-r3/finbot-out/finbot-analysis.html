Executive Summary
Agent Remit (as declared)
FinBot is CineFlow Productions' autonomous invoice-processing agent: it receives vendor invoices, validates the submitting vendor's registered/approved status, runs fraud detection, and decides whether to auto-approve (triggering payment), auto-reject, or escalate to human review. Authorized counterparties are registered vendors (who supply data but cannot give directives), CineFlow finance admins (who may adjust operational thresholds through an out-of-runtime configuration channel), and the LLM provider (inference only). The remit hard-prohibits runtime mutation of FinBot's goals or decision logic by anyone other than a verified operator deploying out-of-band, forbids treating invoice-content fields as directives, and forbids skipping fraud detection for any reason.
Behavior Summary (as observed)

The dominant pattern is policy declared in prose, zero code-level enforcement, plus an unauthenticated control surface that lets any external caller redefine the agent's goals. Every MUST / MUST NOT in the remit — fraud detection cannot be skipped, runtime messages cannot alter decision logic, invoice-content fields are not directives, the manual-review threshold cannot be bypassed — is implemented entirely as system-prompt text and as runtime config defaults that the /api/admin/finbot/* blueprint exposes for unauthenticated POST modification.

A one-hop compound chain runs from anonymous vendor registration → invoice submission with a crafted description → _analyze_business_context's authority/urgency/production keyword scan → approval with payment_processed=True, with the LLM path additionally susceptible to goal-injection via custom_goals that the admin endpoint persists into get_system_prompt's template at line 71. The confidence_threshold field is declared in FinBotConfig but never consulted by any decision path — the canonical declared-but-never-consulted pattern.

Scope of Analysis
Python Flask + SQLAlchemy application (SQLite or PostgreSQL on Render). A single FinBotAgent class in src/services/finbot_agent.py orchestrates OpenAI function-calling against five tools (get_invoice_details, approve_invoice, reject_invoice, request_human_review, detect_fraud_patterns) with a rule-based _fallback_processing path when the OpenAI client is unavailable. Admin routes are exposed via a Flask blueprint at /api/admin/* (config GET/POST, goals POST, vendor-trust POST, dashboard, CTF flags) with no authentication middleware; vendor and user CRUD routes are likewise unauthenticated. Configuration state — including fraud_detection_enabled, custom_goals, speed_priority, and the auto-approve / manual-review thresholds — lives in a single mutable FinBotConfig row that the system prompt interpolates at every request.

CRITICAL PRAX-2026-05-25-001 Unauthenticated admin endpoint persists arbitrary text into FinBot's system prompt via the custom_goals field.
