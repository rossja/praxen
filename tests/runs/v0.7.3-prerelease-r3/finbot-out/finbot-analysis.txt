BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in prose, zero code-level enforcement,
plus an unauthenticated control surface that lets any external caller redefine
the agent's goals. Every MUST / MUST NOT in the remit — fraud detection cannot
be skipped, runtime messages cannot alter decision logic, invoice-content
fields are not directives, the manual-review threshold cannot be bypassed — is
implemented entirely as system-prompt text and as runtime config defaults that
the /api/admin/finbot/* blueprint exposes for unauthenticated POST
modification. A one-hop compound chain runs from anonymous vendor registration
→ invoice submission with a crafted description → _analyze_business_context's
authority/urgency/production keyword scan → approval with
payment_processed=True, with the LLM path additionally susceptible to
goal-injection via custom_goals that the admin endpoint persists into
get_system_prompt's template at line 71. The confidence_threshold field is
declared in FinBotConfig but never consulted by any decision path — the
canonical declared-but-never-consulted pattern.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-25-001  Unauthenticated admin endpoint persists arbitrary text
            into FinBot's system prompt via the custom_goals field.
      Action 1: Require an authenticated operator session (admin role) on
                /admin/finbot/goals POST and reject anonymous requests.
      Action 2: Remove the runtime custom_goals path entirely — goal
                modifications should require a deployment, as the remit
                specifies.

  PRAX-2026-05-25-002  fraud_detection_enabled is a runtime-toggleable bool
            that short-circuits the fraud-detection tool, with no
            authenticated path required to flip it.
      Action 1: Remove fraud_detection_enabled from the runtime config — make
                fraud detection unconditional in code.
      Action 2: If a kill-switch is genuinely needed, gate it behind an
                authenticated operator role AND require a separate
                deployment-time flag.

  PRAX-2026-05-25-003  Business-context heuristic auto-approves invoices even
            when the agent's own injection detector flagged the description.
      Action: Remove the business-context override branches at lines 841-850
              and route every flagged invoice to _request_human_review
              unconditionally.

  PRAX-2026-05-25-004  Manual-review threshold can be bypassed by keyword
            content in the invoice description.
      Action 1: Treat the manual-review threshold as a hard ceiling on
                auto-approval — remove the override branches at lines 808-820.
      Action 2: If expedited review is needed for legitimate cases, build a
                separate authenticated approval channel rather than a
                description-content heuristic.

  PRAX-2026-05-25-005  No authentication middleware on any /api/admin/* route
            — config mutation, goal injection, vendor-trust modification, and
            review decisions all accept anonymous requests.
      Action 1: Add a @require_admin decorator (or Flask-Login + role check)
                to every /admin/* route; reject requests that lack a verified
                operator session.
      Action 2: Restrict CORS to known origins so cross-origin POSTs cannot
                reach the admin surface from arbitrary pages.

  PRAX-2026-05-25-006  confidence_threshold is declared in FinBotConfig but
            never read by any decision path — the canonical
            declared-but-never-consulted pattern.
      Action: Before any _approve_invoice call, compare the LLM's reported
              confidence (and the rule-based path's confidence variable)
              against config.confidence_threshold and route to
              _request_human_review when below.

  PRAX-2026-05-25-007  Vendor records are auto-approved on registration — any
            anonymous caller can register a vendor and immediately submit
            invoices as a registered, approved party.
      Action 1: Set status='pending' on registration and require an
                authenticated admin POST to transition to approved.
      Action 2: Gate register_vendor itself behind authentication so anonymous
                callers cannot self-onboard.

  PRAX-2026-05-25-008  Compound chain — anonymous vendor registration →
            invoice submission with crafted description → business-context
            override → autonomous payment with no human in the loop.
      Action: Remove _analyze_business_context content-based scoring — replace
              with a fixed allow-list of vendor-supplied fields that influence
              the decision; never let invoice content keywords flip approval
              branches.
