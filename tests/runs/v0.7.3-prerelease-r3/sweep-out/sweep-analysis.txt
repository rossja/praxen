BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the remit, near-zero code-level
enforcement in scope: the remit mandates injection screening on every issue /
comment / file / diff, secret redaction in generated content, a hard
prohibition on executing generated code or shell commands, and a configured
WEBHOOK_SECRET for ingress authentication — but the in-scope code paths
implement none of them. Three search tools (question_answerer.ripgrep,
context_pruning.run_ripgrep_command, dynamic_context_bot.use_ripgrep) take an
LLM-derived query argument, .join(...) it into a shell string, and execute it
under subprocess.run(..., shell=True); the only attempt at neutralisation,
escape_ripgrep in core/context_pruning.py, escapes ( and { and ignores every
shell metacharacter that actually matters (;, |, backtick, $, &, redirects).
Combined with the absence of any instruction-injection filter on issue/comment
content entering model context and the absence of a webhook-signature
verification call site in any in-scope file, an attacker who can open an issue
on a Sweep-installed repo has a direct issue-body to LLM context to
shell-execution chain.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-25-001  Model-derived ripgrep queries executed via
            subprocess.run(shell=True) in three search tools, with no
            effective shell-metacharacter escaping.
      Action 1: In agents/question_answerer.py:270,
                core/context_pruning.py:186, and
                core/dynamic_context_bot.py:44, drop shell=True and pass the
                rg command as an argv list so the OS shell never parses
                model-derived text.
      Action 2: Replace core/context_pruning.escape_ripgrep with shlex.quote
                for every model-derived token, or reject any query that
                contains characters outside a strict allow-list
                ([\w\s\.\-_/:]).

  PRAX-2026-05-25-002  Issue and comment bodies flow into LLM context with no
            instruction-injection screening, violating the remit's mandatory
            untrusted-input policy.
      Action 1: Introduce an injection-screening helper (e.g.
                sweepai/utils/injection_filter.py) that quote-wraps
                issue/comment/file/diff content in a dedicated
                <untrusted>…</untrusted> block before any prompt-builder
                consumes it, and call it from web/event_utils.py and from
                agents/modify.create_user_message.
      Action 2: In core/prompts.human_message_prompt, mark every
                contributor-sourced segment with an explicit untrusted-content
                delimiter the system prompt instructs the model to ignore.

  PRAX-2026-05-25-003  Compound chain — untrusted GitHub issue body reaches
            model context, the model emits a query argument, the search tool
            runs it under shell=True.
      Action: Land both PRAX-2026-05-25-001's argv-list / shlex.quote fix and
              PRAX-2026-05-25-002's untrusted-content wrapper; either alone
              reduces but does not eliminate the chain.

  PRAX-2026-05-25-004  WEBHOOK_SECRET declared in config/server.py but no
            in-scope file verifies inbound GitHub webhook HMAC signatures.
      Action 1: Add a FastAPI dependency in sweepai/web/ that reads the
                request body, computes hmac.new(WEBHOOK_SECRET.encode(), body,
                sha256).hexdigest(), and hmac.compare_digest-matches it
                against the X-Hub-Signature-256 header, rejecting on mismatch.
      Action 2: When WEBHOOK_SECRET is unset and IS_SELF_HOSTED is true, fail
                closed at startup rather than silently disabling verification.
