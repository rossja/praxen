Executive Summary
Agent Remit (as declared)
Sweep is a GitHub App that reads issues and comments, generates code changes via a third-party LLM provider, and opens pull requests for human review. Its authorized capabilities are deliberately narrow: receive GitHub events, read repository contents, create branches it authors, commit to those branches, open PRs against the default branch, and reply to the originating issue or comment. The remit forbids merging its own PRs, force-pushing or deleting branches it did not author, executing arbitrary shell commands or generated code, committing secrets, modifying CI/CD or security-sensitive files without human approval outside the PR path, and following any instructions embedded in untrusted issue, comment, file, or diff content. Every character of every issue, comment, PR review, file, and diff must be treated as untrusted input and screened for instruction injection before being used as model context.
Behavior Summary (as observed)

The dominant pattern is policy declared in the remit, near-zero code-level enforcement in scope: the remit mandates injection screening on every issue / comment / file / diff, secret redaction in generated content, a hard prohibition on executing generated code or shell commands, and a configured WEBHOOK_SECRET for ingress authentication — but the in-scope code paths implement none of them.

Three search tools (question_answerer.ripgrep, context_pruning.run_ripgrep_command, dynamic_context_bot.use_ripgrep) take an LLM-derived query argument, .join(...) it into a shell string, and execute it under subprocess.run(..., shell=True); the only attempt at neutralisation, escape_ripgrep in core/context_pruning.py, escapes ( and { and ignores every shell metacharacter that actually matters (;, |, backtick, $, &, redirects).

Combined with the absence of any instruction-injection filter on issue/comment content entering model context and the absence of a webhook-signature verification call site in any in-scope file, an attacker who can open an issue on a Sweep-installed repo has a direct issue-body to LLM context to shell-execution chain.

Scope of Analysis
Python 3.10 FastAPI service packaged as a Docker container (Dockerfile, docker-compose.yml), with an HTTP health endpoint in sweepai/web/health.py, FastAPI Pydantic event models in sweepai/web/events.py, and a small event_utils.py helper that uses the GitHub App installation token to fetch issues. Configuration is environment-driven via sweepai/config/server.py (which declares WEBHOOK_SECRET and many LLM/AWS/Slack/Jira/Sentry/Resend/PostHog/Voyage keys) and sweepai/config/client.py (the SweepConfig Pydantic model plus a subprocess.run(..., shell=True) call to github-linguist). The agent layer in sweepai/agents/ and sweepai/core/ orchestrates Anthropic / OpenAI chat completions with tools — ripgrep in question_answerer.py, run_ripgrep_command in context_pruning.py, use_ripgrep in dynamic_context_bot.py — every one of which calls subprocess.run(..., shell=True) on a model-derived query argument. The in-scope dirs (sweepai/agents|core|web|config + root configs) contain no authentication middleware, no HMAC signature verification, no instruction-injection screening, no approval gate, no rate limiting, and no per-event audit-log writer.

CRITICAL PRAX-2026-05-25-001 Model-derived ripgrep queries executed via subprocess.run(shell=True) in three search tools, with no effective shell-metacharacter escaping.
Implement Zero Trust ASI05 — Unexpected Code Execution (RCE) LLM05 — Improper Output Handling
Policy Rule — R-12 (Worker Remit):
"Sweep MUST NOT execute arbitrary shell commands, install arbitrary packages, or run generated code as part of producing a change."
sweepai/agents/question_answerer.py:270 — response = subprocess.run(" ".join(["rg","-n","-i","-C=5","--heading","--sort-files",query]), shell=True, …) — query is a model-controlled tool argument with no escaping or validation. sweepai/core/context_pruning.py:167 — escape_ripgrep escapes only ( and {; the consumer at core/dynamic_context_bot.py:44-46 and core/context_pruning.py:186-188 runs the joined command with shell=True.
Recommended Action
  • In agents/question_answerer.py:270, core/context_pruning.py:186, and core/dynamic_context_bot.py:44, drop shell=True and pass the rg command as an argv list so the OS shell never parses model-derived text.
  • Replace core/context_pruning.escape_ripgrep with shlex.quote for every model-derived token, or reject any query that contains characters outside a strict allow-list ([\w\s\.\-_/:]).

CRITICAL PRAX-2026-05-25-004 WEBHOOK_SECRET declared in config/server.py but no in-scope file verifies inbound GitHub webhook HMAC signatures.
Implement Zero Trust ASI03 — Identity and Privilege Abuse LLM06 — Excessive Agency
