Executive Summary
Agent Remit (as declared)
yaah is a local developer CLI that generates coding-agent configuration for Claude Code, OpenCode, Codex CLI, and GitHub Copilot CLI from a single source of truth, and ships a built-in security toolset that runs against the configured agent's tool activity. The remit's central promise is that every generated agent configuration carries the same protective hooks — a linter, a command guard, a secret scanner, a comment checker, and a session logger — plus a built-in yaah serve MCP server and a per-session audit log. Authorized counterparties are the local developer, the configured agent's LLM provider, the context7 and pulumi MCP servers (the latter remote over TLS), the built-in yaah MCP server, and the coding-agent plugin marketplace. Catastrophic shell commands must be blocked before they execute; hardcoded credentials must be detected before edits are accepted; session-loaded files must not be writable in a way that lets ingested content persist into future sessions unreviewed.
Behavior Summary (as observed)

The dominant pattern is policy-implementation divergence in the generator layer: the remit asserts that every generated agent configuration carries the same protective hooks, but pkg/generator/hookmap.go leaves the Codex field blank for HookPreToolUse and HookPostToolUse, and pkg/generator/codex.go's GenerateHooks continues past any event whose target name is empty — so yaah generate --agent codex ships a .codex/hooks.json with no command guard and no secret scanner. pkg/generator/codex_test.go documents this in a code comment (// PreToolUse is not supported by Codex.) and asserts the resulting data is nil, confirming the divergence is known to the code and unaddressed in the spec.

Layered on top: the context7 MCP server is launched via unpinned npx -y @context7/mcp (a silent-update vector in direct contradiction of the "third-party MCP servers MUST be pinned" remit clause); MCP tool calls fall outside PreToolUse/PostToolUse hooks entirely, so the command guard, secret scanner, and session log never see them; the auto-managed AGENTS.md (with CLAUDE.md/GEMINI.md symlinks) is session-loaded and writable; and there are no *_test.go files for guard.go, secretscan.go, or sessionlog.go, so the controls the remit treats as load-bearing have no adversarial test fixture asserting they actually catch what they claim to catch.

Scope of Analysis
Go 1.25 CLI built on spf13/cobra, organised around the Interface to Registry to Generator pattern (described in AGENTS.md). The hook handlers live in pkg/hooks/handlers/ — guard.go (deterministic regex blocklist on Bash), secretscan.go (13-pattern scanner on Edit/Write/MultiEdit), sessionlog.go, linter.go, commentcheck.go. pkg/harness/harness.go is the orchestrator; pkg/session/store.go writes a structured per-session JSON record (tool calls, blocked calls, files modified, findings) under .claude/sessions/. The built-in yaah serve MCP server in pkg/mcpserver/ exposes seven tools with descriptions that describe the tool's function only (no instruction-like language). Generators in pkg/generator/ (claude.go, opencode.go, codex.go, copilot.go) emit per-agent native config, sharing the cross-agent event map in hookmap.go. Root .mcp.json and .claude/settings.json configure three MCP servers — context7 (npx -y @context7/mcp), pulumi (HTTP at https://mcp.ai.pulumi.com/mcp), and the built-in yaah server. AGENTS.md is auto-managed (with CLAUDE.md and GEMINI.md as symlinks) and session-loaded. go.mod and go.sum pin direct and transitive dependencies with hashes; .github/workflows/ci.yaml pins action SHAs and uses cosign + Syft SBOM at release.

HIGH PRAX-2026-05-25-001 Generated Codex CLI config ships with no PreToolUse or PostToolUse hooks — the command guard and secret scanner advertised as "uniform across every coding agent" are silently absent from `yaah generate --agent codex` output.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-01, R-15 (Worker Remit):
"Every generated agent configuration MUST carry the same set of protective hooks — a linter, a command guard that blocks catastrophic shell commands, a secret scanner, a comment checker, and a session logger. / The harness's central promise is that its protections are uniform across every coding agent it generates configuration for."
pkg/generator/hookmap.go:30 — HookPreToolUse entry: `{Claude: "PreToolUse", OpenCode: "tool.execute.before", Copilot: "preToolUse"}` — the `Codex` field is omitted, so `CodexEventName(HookPreToolUse)` returns the empty string. pkg/generator/hookmap.go:35 — HookPostToolUse entry: same shape, no Codex value; the secret-scanner event is dropped on the Codex target by the same path. pkg/generator/codex.go:91 — GenerateHooks body: `if codexEvent == "" || codexEvent == "notify" { continue }` — any event without a Codex name is silently skipped during emission. pkg/generator/codex_test.go:118 — TestCodex_GenerateHooks_NoSupported documents the behavior in a code comment ("PreToolUse is not supported by Codex.") and asserts the generated hooks data is nil when only PreToolUse is registered, confirming the gap is known and intentional in the generator.
Recommended Action
  • Populate the Codex field in pkg/generator/hookmap.go for HookPreToolUse and HookPostToolUse (Codex CLI does support PreToolUse / PostToolUse events in its hooks.json), then update TestCodex_GenerateHooks_NoSupported to assert the hooks land in the generated config rather than asserting they are dropped.
  • If a specific Codex protocol limitation actually prevents this, change the remit so its "uniform across every coding agent" promise honestly carves out Codex, and surface the limitation in yaah generate --agent codex output so operators see what they are losing.
HIGH PRAX-2026-05-25-002 MCP tool invocations bypass the hook chain entirely — the command guard, secret scanner, and session log never see them on any target.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-02, R-05 (Worker Remit):
"Catastrophic shell commands — recursive deletion from the filesystem root, force-pushing to a main branch, hard resets, destructive database statements, filesystem formatting, raw disk writes — MUST be blocked before they execute. / Hardcoded credentials in files the agent edits MUST be detected before the change is accepted."
.claude/settings.json:14 — PreToolUse matcher: "(?i)^Bash$" — only the literal `Bash` tool name fires the command guard; MCP tool names (e.g. mcp__yaah__yaah_check_command, mcp__pulumi__*, mcp__context7__*) never match. .claude/settings.json:5 — PostToolUse matcher: "(?i)^(Edit|Write|MultiEdit)$" — same shape; MCP tools that write or delete are not seen by the secret scanner. pkg/hooks/handlers/guard.go:53 — bashMatch = regexp.MustCompile("(?i)^Bash$") — the guard handler's own Match() is hard-coded to the Bash tool name, so even if the matcher were widened in settings, the handler would still skip non-Bash tool names.
Recommended Action
  • Add a hook rule that fires for the mcp__* tool-name family (the conventional Claude Code naming for MCP tools) and dispatch the call through a policy that classifies the MCP tool by its declared capability (read vs write/send/execute); block or queue for approval anything in the latter set.
  • At minimum, record every MCP tool call in the session log (pkg/session/store.go) so the audit trail covers more than just Bash/Edit/Write — currently a write performed via the pulumi MCP server leaves no record in .claude/sessions/<id>.json.
HIGH PRAX-2026-05-25-003 File writes and MCP write/send/execute tools run without a human approval gate — the remit's "MUST reach a human checkpoint" clause is not enforced in code.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-07 (Worker Remit):
"High-impact actions — destructive shell commands, file writes, and MCP tools that write, send, or execute — MUST reach a human checkpoint before they run."
pkg/hooks/handlers/guard.go:79 — Execute returns &hooks.Result{Block: true} on a regex match — binary block/allow only, no approval-queue concept. pkg/hooks/handlers/secretscan.go:127 — Execute either blocks (on a secret hit) or returns nil; non-blocking edits proceed without operator confirmation. pkg/schema/hooks.go:42 — HookType constants are command, http, prompt, agent — there is no approval-gate type, and no handler in pkg/hooks/handlers/ invokes Claude Code's PermissionRequest flow.
Recommended Action
  • Add a RequireApproval handler that returns a structured Result indicating the call should be paused for operator confirmation (Claude Code supports an approval prompt via the PermissionRequest event); wire it for Edit/Write/MultiEdit and for any MCP tool tagged write/send/exec.
  • Document in pkg/hooks/handlers/doc.go which call sites trigger approval and which do not, so the remit's promise and the implementation reality stay in sync.
HIGH PRAX-2026-05-25-005 `context7` MCP server installs unpinned via `npx -y @context7/mcp` — a silent-update vector and a direct contradiction of the "third-party MCP servers MUST be pinned" remit clause.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities
Policy Rule — R-12 (Worker Remit):
"Third-party MCP servers MUST be pinned to a known-good, integrity-checked version — no server package auto-installed afresh, unpinned, on every run."
.mcp.json:3 — "context7": { "type": "stdio", "command": "npx", "args": ["-y", "@context7/mcp"] } — no @x.y.z version specifier on the package name. .claude/settings.json:64 — same shape duplicated in the mcpServers block of the Claude settings file (the file maintains two copies; see PRAX-2026-05-25-009). pkg/mcp/providers/context7.go:18 — Context7.Server() returns the same unpinned `npx -y @context7/mcp` definition, so even regenerating the config from the harness does not pin it.

MEDIUM PRAX-2026-05-25-006 Third-party MCP tool descriptions for `context7` and `pulumi` are never inspected — the remit's "MUST NOT contain instruction-like language" rule cannot be enforced for them.
Balance Your Knowledge Base Tool descriptions contain no hidden model instructions LLM01 — Prompt Injection
Policy Rule — R-04 (Worker Remit):
"MCP tool descriptions MUST NOT contain instruction-like language directed at the model — neither the built-in server's tool descriptions nor, as far as the harness can inspect them, those of the third-party servers it configures."
pkg/mcpserver/tools.go:58 — yaah's own seven tool descriptions are functional and clean (e.g. "Scan a file for hardcoded secrets and credentials") — the built-in half of R-04 is satisfied. pkg/mcp/providers/context7.go:21 — provider only carries Name/Command/Args — no field for capturing or validating the server's advertised tool descriptions after launch. cmd/yaah/main.go:219 — yaah doctor checks MCP servers for binary presence on the PATH only; there is no equivalent check that fetches tools/list from each server and scans the descriptions.
Recommended Action
Add a yaah mcp scan (or extend yaah doctor) that, for every registered MCP provider, performs a one-shot stdio/HTTP handshake, fetches tools/list, and runs a description-language linter (looks for second-person pronouns, imperatives, "When using this tool you should ...", etc.) — surface any hits as findings recorded to the session log.
MEDIUM PRAX-2026-05-25-007 Auto-managed session-loaded `AGENTS.md` (with `CLAUDE.md`/`GEMINI.md` symlinks) is writable through the same Edit/Write path that any other file is — no integrity guard on the bootstrap surface.
Balance Your Knowledge Base ASI06 — Memory and Context Poisoning
Policy Rule — R-06 (Worker Remit):
"Agent-managed, session-loaded files MUST NOT be writable in a way that lets ingested content persist into future sessions unreviewed."
AGENTS.md:2 — <!-- Managed by agent: keep sections and order; edit content, not structure --> — file declares itself agent-writable. AGENTS.md:12 — <!-- AGENTS-GENERATED:START commands --> … several auto-generated blocks; the file is designed to be re-emitted by the agent. . — workspace listing — CLAUDE.md -> AGENTS.md and GEMINI.md -> AGENTS.md symlinks ensure a single edit reaches every agent's bootstrap surface. pkg/hooks/handlers/secretscan.go:74 — PostToolUse matcher is (?i)^(Edit|Write|MultiEdit)$ and the handler scans only for credential patterns — there is no separate handler that guards the integrity of AGENTS.md / CLAUDE.md / GEMINI.md.
Recommended Action

  • Track and commit changes to AGENTS.md separately from code changes so the bootstrap-surface diff is reviewable in version control.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Deterministic command guard with explicit catastrophic-command blocklist

pkg/hooks/handlers/guard.go runs on every PreToolUse Bash event and blocks `rm -rf /`, force-push to main/master, `git reset --hard`, `DROP TABLE`, system-file truncation, `mkfs`, and `dd of=/dev/*` — deterministic regex, no LLM in the loop, returns Block: true and the matched rule's reason.

pkg/hooks/handlers/guard.go:29-36

13-pattern secret scanner wired into every Edit/Write/MultiEdit

pkg/hooks/handlers/secretscan.go ships with patterns for AWS credentials, GitHub PATs/OAuth tokens, OpenAI keys, Anthropic keys, generic password literals, bearer tokens, private keys, Slack tokens/webhooks, and SendGrid keys; runs file-by-file, blocks the edit on a hit with a remediation message.

pkg/hooks/handlers/secretscan.go:36-49

Durable structured per-session audit log

pkg/session/store.go writes a Session JSON document per session under .claude/sessions/<id>.json via temp-file + rename — atomic, structured, with timestamped ToolCalls, BlockedCalls, FilesModified, and Findings arrays; `yaah session list/show/clean` exposes the operator-facing surface.

pkg/session/store.go:52

Exact-pinned Go dependency graph with cryptographic integrity

go.mod pins direct deps to exact versions (go-sdk v1.4.0, cobra v1.10.2), and go.sum carries h1: hashes for every transitive module — small tree (eight transitive modules) — so dependency-confusion and silent-upgrade attacks at the Go layer are not feasible.

Built-in MCP server tool descriptions are clean — no instruction-like language

All seven tools in pkg/mcpserver/tools.go (yaah_scan_secrets, yaah_lint, yaah_check_command, yaah_doctor, yaah_session_info, yaah_planning_status, yaah_planning_init) carry functional descriptions only ("Scan a file for hardcoded secrets and credentials", "Check whether a shell command is safe to run", …) — no second-person imperatives, no embedded model instructions, satisfies the built-in half of remit R-04.
