Executive Summary
Agent Remit (as declared)
Devika is intended as an autonomous, locally-run software-engineering agent: it takes a natural-language project request from a single operator, decomposes it into a plan, performs web research, generates code, and — when asked — runs that code inside a sandboxed runner confined to a per-project work directory. Authorized counterparties are the operator, the project workspace, an LLM provider, and a search provider; all file I/O is to be confined to the configured per-project directory and all user input and retrieved web content is to be treated as untrusted and screened for instruction injection before reaching LLM context. Every action is required to be recorded to a per-project audit log, runs are required to enforce wall-clock and step-count caps, and control must return to the operator before any package install, write outside a previously-touched file, or unanticipated shell execution. Devika MUST NOT modify its own code/config/memory, exfiltrate operator credentials, or follow instructions embedded in retrieved content.
Behavior Summary (as observed)

The dominant pattern is sandbox-shaped-but-not-substantive: the remit and the architecture document both declare a sandboxed runner as the control that confines LLM-produced code, but the src/sandbox/ directory ships only two 0-byte files (firejail.py, code_runner.py) and the actual Runner in src/agents/runner/runner.py bypasses them entirely — it calls subprocess.run(command.split(" "), cwd=project_path) directly on shell strings the LLM emits, with a 2-retry self-heal loop and zero approval.

Composed with two adjacent gaps — web search results flowing through the formatter into LLM context with no injection screening, and an unauthenticated /api/settings on 0.0.0.0:1337 that rewrites config.toml in place — Devika offers a one-injection-away path from a crafted search result to host shell execution under the operator's user. The remit's required compensating controls (per-action audit log, wall-clock / step-count caps, approval gate for writes outside touched files) are absent from the codebase.

Scope of Analysis
Python Flask + Flask-SocketIO application (devika.py) binding to 0.0.0.0:1337 with no authentication on any route, fronted by a SQLite-backed project store and a multi-agent orchestration tree under src/agents/ (planner, researcher, formatter, coder, action, runner, patcher, feature, reporter, decision). The Agent class in src/agents/agent.py chains planner -> researcher -> web search + page fetch -> formatter -> coder.save_code_to_project; in subsequent turns action dispatches to runner, which calls subprocess.run() directly on shell command strings returned by the LLM with no approval gate, no allowlist, and no sandbox — the only sandbox surface, src/sandbox/firejail.py and src/sandbox/code_runner.py, are 0-byte stub files. The Flask blueprint at src/apis/project.py and the /api/settings / /api/get-browser-snapshot routes in devika.py expose project create/delete/download, a GET that returns plaintext API keys, a POST that overwrites config.toml in place, and a snapshot endpoint that send_file()s an arbitrary user-supplied path.

Project create/delete/download routes use werkzeug.utils.secure_filename on the project name

The Flask blueprint at src/apis/project.py consistently passes request.args.get("project_name") through secure_filename() before resolving paths, which strips traversal and shell metacharacters from the project-name component. This does not fix the path-traversal in save_code_to_project (where the LLM supplies the file portion downstream of the project segment), but it does close the project-name-vector consistently across the blueprint.

ProjectManager.get_project_files resolves base_path = os.path.abspath(os.path.join(os.getcwd(), 'data', 'projects')) and rejects the request when the requested directory's common prefix does not match. The same pattern, applied at save_code_to_project, would close PRAX-2026-05-25-004 — so the team knows the technique, they just have not applied it on the write path.
