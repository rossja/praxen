Executive Summary
Agent Remit (as declared)
Devika is an autonomous software-engineering agent that takes a natural-language project request, decomposes it into a plan, writes code into a per-project workspace, and may run that code to validate it. The remit permits planning, code generation, web research via configured search providers, and inference against operator-configured LLM providers; it scopes all file I/O to the current project's work directory and treats user prompts and retrieved web content as untrusted input that must be screened for instruction injection. Devika MUST NOT execute shell commands outside a sandboxed runner, MUST NOT deploy to production, MUST NOT install packages without explicit user confirmation, and MUST NOT contact external services beyond the LLM and search providers.
Behavior Summary (as observed)

The dominant pattern is sandbox-shaped-but-not-substantive compounded with policy declared in prose, zero code-level enforcement. The architecture document and the remit both describe a sandboxed runner that confines LLM-produced command execution; the actual implementation in src/agents/runner/runner.py executes those commands through raw subprocess.run with no allowlist, no approval prompt, and no isolation, while src/sandbox/code_runner.py and src/sandbox/firejail.py are 1-line stub files. The remit also forbids production deployment and forbids action on instructions embedded in user or web content, yet src/services/netlify.py wires a deploy capability directly into the action loop and not a single agent prompt or code path performs injection screening on the user message or the formatted web-search results before they enter LLM context or drive a tool decision.

A reader opening the workspace should start in src/agents/runner/runner.py, src/sandbox/, src/agents/agent.py (the deploy branch in subsequent_execute), and the WebSocket handler in devika.py: those four files together establish that the agent has an unauthenticated WebSocket ingress, no instruction-injection screening on inbound content, raw shell execution of LLM-produced commands, and an outbound deploy path the remit forbids.

Scope of Analysis
Python Flask + Socket.IO application orchestrating eleven specialized sub-agents (planner, researcher, coder, action, runner, feature, patcher, reporter, decision, formatter, internal_monologue) under src/agents/agent.py, with pluggable LLM providers (Claude, OpenAI, Gemini, Mistral, Groq, Ollama, LM Studio), Playwright-driven browser automation in src/browser/, SQLite-backed project and agent-state stores, and TOML-based configuration in sample.config.toml. The runner agent at src/agents/runner/runner.py executes LLM-produced shell commands via raw subprocess.run on the host. A src/sandbox/ directory exists with files named code_runner.py and firejail.py — both are 1-line stubs. A Netlify deployment integration in src/services/netlify.py is wired into the agent's action flow, and the WebSocket handler at devika.py spawns background agent threads with no authentication.
