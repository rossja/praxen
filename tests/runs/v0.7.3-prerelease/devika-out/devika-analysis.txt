BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is sandbox-shaped-but-not-substantive compounded with
policy declared in prose, zero code-level enforcement. The architecture
document and the remit both describe a sandboxed runner that confines
LLM-produced command execution; the actual implementation in
src/agents/runner/runner.py executes those commands through raw subprocess.run
with no allowlist, no approval prompt, and no isolation, while
src/sandbox/code_runner.py and src/sandbox/firejail.py are 1-line stub files.
The remit also forbids production deployment and forbids action on
instructions embedded in user or web content, yet src/services/netlify.py
wires a deploy capability directly into the action loop and not a single agent
prompt or code path performs injection screening on the user message or the
formatted web-search results before they enter LLM context or drive a tool
decision. A reader opening the workspace should start in
src/agents/runner/runner.py, src/sandbox/, src/agents/agent.py (the deploy
branch in subsequent_execute), and the WebSocket handler in devika.py: those
four files together establish that the agent has an unauthenticated WebSocket
ingress, no instruction-injection screening on inbound content, raw shell
execution of LLM-produced commands, and an outbound deploy path the remit
forbids.
