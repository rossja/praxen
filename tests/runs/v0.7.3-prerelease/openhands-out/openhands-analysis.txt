BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern in this scan is auth-by-environment-variable: every
protected V1 router declares dependencies=get_dependencies(), but
get_dependencies() in app_server/utils/dependencies.py only attaches a real
check when the SESSION_API_KEY env var is set. In OSS mode with no env var
(the default for a developer running docker compose up), the entire /api/v1
surface — sandbox start/delete, secrets CRUD, settings, conversation create —
is reachable with no authentication, and the DefaultUserAuth implementation
returns None for user_id, collapsing every caller into a single shared
identity. The two reinforcing weaknesses are the localhost-CORS middleware in
app_server/middleware.py, which logs a warning and allows any origin when
permitted_cors_origins is empty (the default), and the InMemoryRateLimiter
attached at 10 req/s — a soft mitigation that does not constitute access
control. The remit's authentication, per-session isolation, and approval-gate
clauses (cross-repo writes, PR merge, MCP-server-add) have no code-level
enforcement in the default OSS deployment; the agent relies entirely on the
operator's deployment perimeter to keep these endpoints off the network.
