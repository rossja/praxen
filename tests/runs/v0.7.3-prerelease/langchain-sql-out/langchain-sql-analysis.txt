BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in prompt, zero code-level enforcement
on the agent's exec edge. The remit's hard prohibitions — no DML, no DDL, no
multi-statement queries, no out-of-scope tables, no column wildcards, no
injection-driven instruction-following — are all expressed in the SQL_PREFIX
natural-language system prompt; QuerySQLDatabaseTool._run immediately forwards
whatever string the model emits to db.run_no_throw, which calls SQLAlchemy
text() with no parser, allowlist, or rejection layer. The sql_db_query_checker
tool the prompt tells the model to call is itself an LLM call (not a parser)
and the agent does not gate execution on its result. Combined with schema and
sample-row content from InfoSQLDatabaseTool flowing back into LLM context,
this creates a one-hop injection-to-write chain whenever the connected role
has write privileges.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-23-001  No code-level guard on DML/DDL — the only defence
            against data- or schema-modifying SQL is a sentence in the LLM
            system prompt.
      Action 1: In
                libs/community/langchain_community/tools/sql_database/tool.py,
                add a read-only parser pass in QuerySQLDatabaseTool._run
                before self.db.run_no_throw(query) — parse with sqlglot (or
                equivalent), reject any statement whose type is not
                SELECT/WITH/EXPLAIN/SHOW, and return a structured error to the
                agent instead of executing.
      Action 2: Document the read-only-role expectation as a hard requirement
                (not a recommendation) in the create_sql_agent docstring and
                surface it as a constructor-time warning when the supplied
                SQLDatabase appears to authenticate with a writer-capable
                role.

  PRAX-2026-05-23-002  Multi-statement queries and stored-procedure
            invocations are not detected or blocked at the exec edge.
      Action 1: Parse the model-emitted SQL with sqlglot.parse (or equivalent
                multi-statement-aware parser) in QuerySQLDatabaseTool._run; if
                more than one top-level statement is returned, reject the
                query and surface a structured error to the agent.
      Action 2: As defence-in-depth, document and recommend connection-string
                flags that disable multi-statement support on dialects where
                it is opt-in (e.g., MySQL multi_statements off), and call this
                out in the create_sql_agent security warning.

  PRAX-2026-05-23-003  Schema and sample-row content from the database flows
            back into LLM context with no injection filtering — instructions
            in row/column/table names execute as if from the operator.
      Action 1: Wrap InfoSQLDatabaseTool and ListSQLDatabaseTool outputs in a
                delimiter that the system prompt teaches the model to treat as
                untrusted data (e.g., <db-data>...</db-data>) and update
                SQL_PREFIX to explicitly instruct the model never to follow
                instructions inside those delimiters.
      Action 2: Add an output-content scanner (regex or LLM-judge) that flags
                imperative phrases in returned row content; on flag, redact or
                return a structured warning to the agent rather than the raw
                content.

  PRAX-2026-05-23-004  The 'double-check' step is an unsupervised LLM rewrite,
            not a parser — the checker can rewrite a SELECT into a DML
            statement and execution proceeds anyway.
      Action 1: Replace the LLM-based checker with a deterministic sqlglot
                parse pass that validates statement type (must be
                SELECT/WITH/EXPLAIN/SHOW), confirms no semicolons after the
                first statement, and returns a structured pass/fail to the
                agent — not a rewritten query.
      Action 2: If keeping an LLM rewrite step for ergonomics, gate
                QuerySQLDatabaseTool._run on the checker's output being the
                same statement-kind as the original input, parsed and compared
                before execution.

  PRAX-2026-05-23-005  Compound chain — database-supplied row content can
            hijack the agent and, via the unguarded query tool, drive
            arbitrary writes against the configured connection.
      Action 1: Land the SQL-parser allowlist from PRAX-2026-05-23-001 first —
                it breaks the chain even if PRAX-2026-05-23-003 and
                PRAX-2026-05-23-004 remain unaddressed, because a DML
                statement will never reach the connection regardless of how
                the agent was hijacked.
      Action 2: Document the chain explicitly in create_sql_agent's security
                warning so operators understand that a read-only DB role is
                not optional — it is the only effective control today.
