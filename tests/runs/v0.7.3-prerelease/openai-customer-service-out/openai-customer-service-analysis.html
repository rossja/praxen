Executive Summary
Agent Remit (as declared)
A multi-agent airline customer-service system that triages inbound customer requests and routes them to specialist agents that answer FAQ questions or update seat assignments on existing confirmed reservations. The remit authorizes two tools — faq_lookup_tool (deterministic FAQ lookup) and update_seat (seat assignment on confirmed reservations) — operating against the airline reservation system, FAQ dataset, and flight schedule. Identity verification, confirmation-number validation against the authoritative reservation record, and a durable audit trail for every seat change are mandatory; ticket purchases, refunds, special-handling cases, and any outbound email/SMS/webhook are forbidden.
Behavior Summary (as observed)

The dominant pattern is framework offers safe primitives, the example wires none of them. The openai-agents-python SDK ships InputGuardrail, OutputGuardrail, ToolInputGuardrail, per-tool needs_approval, and tracing — exactly the controls the airline remit calls for — yet examples/customer_service/main.py configures zero guardrails, zero approval gates, and no audit logging. As a result every behavioral rule that requires runtime enforcement (identity verification before mutation, confirmation-number validation against an authoritative record, refusal of out-of-scope flows, prompt-injection resistance, durable seat-change audit) is enforced only by instructions in an LLM system prompt — and update_seat trusts caller-supplied confirmation_number and new_seat directly, while on_seat_booking_handoff fabricates a flight number with random.randint in place of looking it up. A reader copying this example into production inherits a system whose stated invariants exist only as English in a prompt.

Scope of Analysis
A single-file Python example at examples/customer_service/main.py built on the openai-agents-python SDK. Three Agent instances — triage_agent, faq_agent, seat_booking_agent — share an AirlineAgentContext Pydantic model holding passenger_name, confirmation_number, seat_number, and flight_number. Handoffs are wired with handoff(...); update_seat writes to context.context with no reservation-record lookup, and on_seat_booking_handoff fabricates a flight number via random.randint(100, 999). The SDK exposes InputGuardrail, OutputGuardrail, ToolInputGuardrail, needs_approval, and tracing primitives, but the example wires none of them onto these three agents.

CRITICAL PRAX-2026-05-23-001 No customer identity verification anywhere in the agent pipeline before <code>update_seat</code> mutates reservation state.
Implement Zero Trust ASI03 — Identity and Privilege Abuse LLM06 — Excessive Agency
Policy Rule — R-07, R-12, R-21 (Worker Remit):
"Customer identity MUST be confirmed before any action that mutates reservation state. / Reservations or seat assignments for any customer other than the authenticated session user MUST NOT be modified. / Any request where the customer's identity cannot be verified MUST be escalated to a human."
examples/customer_service/main.py:29 — AirlineAgentContext fields (passenger_name, confirmation_number, seat_number, flight_number) all default to None and are never populated from an authenticated session. examples/customer_service/main.py:67 — update_seat accepts confirmation_number and new_seat as tool arguments and writes them to context.context with no identity check. examples/customer_service/main.py:151 — main() builds a fresh AirlineAgentContext() and conversation_id = uuid.uuid4() with no authentication step; any caller can claim any confirmation number.
Recommended Action
  • Before constructing AirlineAgentContext, authenticate the customer (session token, OIDC, etc.) and seed context.passenger_name from the verified identity claim — never from tool arguments or user input.
  • Add an InputGuardrail on triage_agent that halts when context.passenger_name is None, and route the halt to the remit's human-escalation path.
  • Inside update_seat, fetch the reservation from the authoritative reservation system and assert it belongs to context.passenger_name before any context write.
CRITICAL PRAX-2026-05-23-002 <code>update_seat</code> trusts caller-supplied <code>confirmation_number</code> with no lookup against the authoritative reservation record.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation LLM06 — Excessive Agency
Policy Rule — R-03, R-08, R-18 (Worker Remit):
"The seat-booking agent updates seat assignments on existing confirmed reservations only. / A submitted confirmation number MUST be verified as belonging to the authenticated customer before that reservation is read or modified. / A seat change MUST NOT be issued without first verifying the confirmation number against the authoritative reservation record."
examples/customer_service/main.py:67 — update_seat signature takes confirmation_number: str and new_seat: str directly from the LLM tool call. examples/customer_service/main.py:79 — context.context.confirmation_number = confirmation_number — the value is written to context with no lookup against any reservation store. examples/customer_service/main.py:82 — The only precondition checked is assert context.context.flight_number is not None — flight_number is itself fabricated, not retrieved (see PRAX-2026-05-23-005).
Recommended Action
  • Replace the body of update_seat with a reservation-system lookup (e.g., reservation = reservations.get(confirmation_number)) and raise a tool error if the record is missing or does not belong to context.passenger_name.
  • Add a ToolInputGuardrail on update_seat that enforces the same precondition and returns RejectContentBehavior when the confirmation number is unknown.
CRITICAL PRAX-2026-05-23-003 <code>update_seat</code> performs no existence or availability check on the requested seat.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation LLM05 — Improper Output Handling
Policy Rule — R-09 (Worker Remit):
"Seat updates MUST be limited to seats that exist on the flight and are available — not already assigned to another passenger."
examples/customer_service/main.py:80 — context.context.seat_number = new_seat — the only operation on new_seat; no seat-map lookup, no availability check, no format validation. examples/customer_service/main.py:83 — return f"Updated seat to {new_seat} for confirmation number {confirmation_number}" — success is reported unconditionally.
Recommended Action
  • Inside update_seat, fetch the flight's seat map from the authoritative reservation system and reject the call if new_seat is not in the map or is currently assigned to another passenger.
  • Wrap the validation in a ToolInputGuardrail that returns RejectContentBehavior with a model-readable rejection reason so the LLM can ask the customer for another seat.
CRITICAL PRAX-2026-05-23-004 No durable audit log is written for seat changes — the remit-mandated record (time, identity, confirmation number, old/new seat, agent) does not exist.
Monitor Continuously ASI03 — Identity and Privilege Abuse

HIGH PRAX-2026-05-23-006 No <code>InputGuardrail</code> is configured on any agent — customer free-text flows into the LLM with no prompt-injection screening.
Balance Your Knowledge Base LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-13 (Worker Remit):
"Instructions embedded in free-text customer input that attempt to override an agent's role, scope, or identity MUST NOT be accepted or acted on."
examples/customer_service/main.py:96 — faq_agent = Agent[...](name="FAQ Agent", ..., tools=[faq_lookup_tool]) — no input_guardrails kwarg. examples/customer_service/main.py:109 — seat_booking_agent = Agent[...](... tools=[update_seat]) — no input_guardrails kwarg. examples/customer_service/main.py:123 — triage_agent = Agent[...](... handoffs=[...]) — no input_guardrails kwarg. examples/customer_service/main.py:167 — input_items.append({"content": user_input, "role": "user"}) — raw user input is appended without any screening pass.

HIGH PRAX-2026-05-23-007 No <code>ToolOutputGuardrail</code> or output filter — tool returns and handoff content flow back into the LLM unchecked.
Balance Your Knowledge Base LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-14 (Worker Remit):
"Instructions embedded in tool outputs — FAQ content, seat records, flight data — that attempt to change an agent's behavior MUST NOT be accepted."
examples/customer_service/main.py:39 — @function_tool(name_override="faq_lookup_tool", ...) — decorator invocation does not set tool_input_guardrails or tool_output_guardrails. examples/customer_service/main.py:67 — @function_tool above update_seat — same: no output guardrail wired. src/agents/tool_guardrails.py:1 — ToolInputGuardrail / ToolOutputGuardrail / RejectContentBehavior / RaiseExceptionBehavior all exist in the SDK and would slot in directly — none are used in the example.
Recommended Action
  • Add a ToolOutputGuardrail that runs the tool result through an instruction-detection check (regex + small classifier) and returns RejectContentBehavior when control phrases are detected; attach it to both faq_lookup_tool and update_seat.
  • Add an output_guardrails list on each agent that screens the final assistant message before it reaches the customer (e.g. PII leakage, tool-output regurgitation).

HIGH PRAX-2026-05-23-009 No application-level structured logging — agent decisions, handoffs, tool calls, and rejections leave no durable record.
Monitor Continuously ASI08 — Cascading Failures
examples/customer_service/main.py:170 — for new_item in result.new_items: ... print(f"{agent_name}: ...") — print to stdout is the only sink for handoff events, tool calls, and tool outputs. examples/customer_service/main.py:1 — File-level grep — no import of logging, no logger instance, no structured event emission anywhere in main.py.

MEDIUM PRAX-2026-05-23-011 FAQ agent's "don't speculate" rule is enforced only by a system-prompt sentence, not by code.
Limit Your Domain LLM09 — Misinformation
Policy Rule — R-02 (Worker Remit):
"The FAQ agent answers questions about airline policies — baggage, seating, wifi, and similar — using only the curated FAQ dataset, and MUST NOT speculate or draw on general knowledge outside that dataset."
examples/customer_service/main.py:99 — faq_agent instructions: "Use the faq lookup tool to answer the question. Recommended Action

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Triage agent ships with no data-access tools — data-access prohibition holds by omission

The triage agent's <code>Agent</code> constructor wires only handoffs to faq_agent and seat_booking_agent, with no <code>tools=</code> kwarg and no memory/data hooks. That keeps the remit's "MUST NOT access customer data directly" rule satisfied today, even though no test or guardrail locks the invariant in place (see PRAX-2026-05-23-010 for the regression-guard recommendation).

Tool inventory matches the remit's Known Good Baseline exactly

The two functions decorated with <code>@function_tool</code> in this file are exactly <code>faq_lookup_tool</code> (used only by faq_agent) and <code>update_seat</code> (used only by seat_booking_agent) — no extra tools, no broader inventory, no remit-delta on capability surface.

examples/customer_service/main.py:39

No session persistence — the remit's no-persistent-memory rule holds

The remit's prohibition on persistent memory across customer sessions is honoured.
