BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the remit, no code-level
enforcement at the repository boundary or on untrusted file content. The remit
forbids reading/writing outside the repo and demands injection-neutralization
on comments, docstrings, and commit messages, but Aider's path handling
(safe_abs_path, abs_root_path) only resolves paths — it never rejects ones
that escape self.coder.root, and the one outside-repo guard in cmd_add is
gated on auto_commits, while cmd_read_only, cmd_load, and allowed_to_edit have
no equivalent check at all. Untrusted content (file bodies, /web-scraped HTML,
AI! comments picked up by the watcher) flows into the LLM context with zero
injection-detection or secret-redaction, while auto_commits and auto_lint
default to True — so the LLM's edits and the linter's shell invocation both
bypass the per-action confirmation the remit demands. The compound chain is
real: a repository whose file content can steer the LLM, with no
neutralization, into producing edits that are auto-committed and into shell
commands the developer can be social-engineered into approving via the
file-watcher path.
