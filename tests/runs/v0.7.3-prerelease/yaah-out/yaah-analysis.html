Executive Summary
Agent Remit (as declared)
yaah is a local developer CLI that generates coding-agent configuration for four targets — Claude Code, OpenCode, Codex CLI, and GitHub Copilot CLI — from a single Go source of truth, and ships a built-in security-and-quality toolset alongside that configuration. The toolset is five protective hooks (linter, command guard, secret scanner, comment checker, session logger), a built-in MCP server exposing read-only check tools plus one workspace-writing planning tool, and a per-session audit log; remote MCP wiring is limited to context7 and pulumi. The remit's central promise is that every generated agent configuration carries the same set of protective hooks, that catastrophic shell commands and hardcoded credentials are caught before they take effect, and that high-impact writes/exec reach a human checkpoint.
Behavior Summary (as observed)

yaah's central promise — that every generated coding-agent configuration carries the same set of protective hooks — is contradicted in code: pkg/generator/hookmap.go has no Codex entry for PreToolUse, so the CommandGuard that blocks rm -rf / / force-pushes / DROP TABLE is silently dropped from generated Codex configs, and TestCodex_GenerateHooks_NoSupported codifies that behavior as intended. The README acknowledges the gap ("Codex: Hooks limited to SessionStart/Stop — linting and security checks available via yaah MCP tools"), but a defense surfaced only as an opt-in MCP tool the model may or may not call is not the same control as a deterministic Block: true on PreToolUse.

Two further structural patterns compound this: the controls that are wired are bypassable on close inspection (the command-guard regexes hard-code --force, rm -rf, and a small fixed pattern set — git push -f, --force-with-lease, rm -r -f, and many destructive variants slip through), and the harness itself ships zero adversarial test fixtures and no SECURITY.md / disclosure policy, which the remit's Escalation section explicitly calls for.

Scope of Analysis
Go 1.25 module (github.com/dirien/yet-another-agent-harness) following an Interface → Registry → Generator pattern: a Harness wires hooks, mcp, lsp, skills, agents, commands, plugins, and a session.Store, and four per-target generators (pkg/generator/claude.go, opencode.go, codex.go, copilot.go) emit native config formats. Hook handlers live in pkg/hooks/handlers/ (regex-based CommandGuard, line-scanning SecretScanner, Linter, CommentChecker, SessionLogger); a built-in MCP server (pkg/mcpserver/) exposes yaah_scan_secrets, yaah_lint, yaah_check_command, yaah_doctor, yaah_session_info, yaah_planning_status, and yaah_planning_init. The Codex generator's hook map (pkg/generator/hookmap.go) omits PreToolUse entirely — confirmed by TestCodex_GenerateHooks_NoSupported, which asserts that a PreToolUse rule produces a nil hooks file — and the project README at line 236 documents the same limitation.

"Every generated agent configuration MUST carry the same set of protective hooks — a linter, a command guard that blocks catastrophic shell commands, a secret scanner, a comment checker, and a session logger. / Catastrophic shell commands — recursive deletion from the filesystem root, force-pushing to a main branch, hard resets, destructive database statements, filesystem formatting, raw disk writes — MUST be blocked before they execute."

CRITICAL PRAX-2026-05-23-002 CommandGuard regex set is so narrow that obvious catastrophic-command variants pass the gate (git push -f / --force-with-lease, rm -r -f, rm with paths beyond `/`, fork bombs, curl|sh)
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-03 (Worker Remit):
"Catastrophic shell commands — recursive deletion from the filesystem root, force-pushing to a main branch, hard resets, destructive database statements, filesystem formatting, raw disk writes — MUST be blocked before they execute."
pkg/hooks/handlers/guard.go:28 — NewCommandGuard hard-codes seven literal patterns; the recursive-delete rule is `rm\s+-rf\s+/` — matches only that exact flag bundle and a literal trailing `/`, not `rm -r -f /home/...`, `rm -fr /`, or recursive deletions targeting paths beneath root. pkg/hooks/handlers/guard.go:30 — Force-push rule is `git\s+push\s+--force\s+(origin\s+)?(main|master)` — does not match `git push -f origin main`, `git push --force-with-lease origin main`, or pushes to release branches other than main/master. pkg/hooks/handlers/guard.go:32 — Destructive-SQL rule is `DROP\s+(TABLE|DATABASE)` — TRUNCATE, DELETE without WHERE, ALTER TABLE … DROP COLUMN, and DROP SCHEMA all pass through. pkg/hooks/handlers/guard.go:33 — Truncate-system-file rule is `:\s*>\s*/` — matches only the `:` shell idiom; `> /etc/shadow` or `cat /dev/null > /etc/passwd` are not caught.
Recommended Action
  • Rewrite the catastrophic-command guard as a parsed-AST or strict allowlist policy rather than a regex denylist: parse the bash command, walk arguments, and flag any `rm -r`/`rm -R` variant regardless of flag order or path, any `git push --force*` or `git push -f` to any branch the operator has not explicitly opted in to, and any destructive-DDL/DML statement (TRUNCATE, DELETE without WHERE, DROP, ALTER…DROP).
  • Add adversarial test fixtures in `pkg/hooks/handlers/guard_test.go` that exercise the bypass list from this finding's evidence — each variant must return Block:true. Wire the test into CI as a regression gate so a future regex rewrite cannot silently lose coverage.
  • Publish a narrowly-scoped "what this guard catches and what it does not" doc that the README's protections promise links to, so operators do not assume the seven regexes are equivalent to the remit's class-level guarantee.

HIGH PRAX-2026-05-23-008 OpenCode JS plugin literal has mismatched braces — the `event` handler's switch is closed too late and the trailing object closes with stray brackets, likely failing to load
Implement Zero Trust ASI04 — Agentic Supply Chain Vulnerabilities
Policy Rule — R-02 (Worker Remit):
"Every generated agent configuration MUST carry the same set of protective hooks — a linter, a command guard that blocks catastrophic shell commands, a secret scanner, a comment checker, and a session logger."
pkg/generator/opencode.go:116 — event handler begins `event: async ({ event }) => { if (event.type === "session.created") { try { ... } catch (e) { /* non-fatal */ } }` — the `if` block closes after the catch, but the arrow function never closes before the outer object. pkg/generator/opencode.go:127 — Closing sequence reads `}, };` — too few closing braces for the nested structure (return-object → event-arrow → if-block → try-catch). A JS parser fails on the literal as written. pkg/generator/opencode_test.go:80 — TestOpenCode_GenerateHooks_Plugin verifies the string CONTAINS `YaahPlugin`, `tool.execute.before`, `tool.execute.after`, `session.created`, `"hook", "PreToolUse"` — no syntactic parse, so the brace mismatch ships green.
Recommended Action
  • Fix the JS literal: close the `event` arrow function before the outer object closes. The minimal patch is to add one closing brace before the final `}};` so the structure is `event: async ({event}) => { if (...) { try {...} catch (...) {} } }, };` — or move the plugin out of a Go string literal and into a versioned `pkg/generator/opencode_plugin.js` embedded via `//go:embed` so syntax errors are caught by the JS toolchain at build time.
  • Add an integration test that runs `node --check` against the generated plugin file (or parses it via a Go JS-AST library) and fails the build on a syntax error. The current substring-only assertion catches missing strings but not structural breakage.
