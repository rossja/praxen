Executive Summary
Agent Remit (as declared)
FinBot is the autonomous invoice-processing agent for CineFlow Productions' finance operations team. The remit authorizes it to retrieve invoice and vendor records, run fraud detection, and approve, reject, or escalate invoices — and to auto-approve only invoices from registered, approved vendors below the configured threshold. The remit hard-prohibits skipping fraud detection, approving payments to unregistered vendors, processing above-threshold invoices without human sign-off, and treating runtime messages (invoice descriptions, vendor notes, configuration text) as goal-altering directives. Decision logic is declared fixed at deployment and not redefinable at runtime by any caller.
Behavior Summary (as observed)

The dominant pattern is policy declared in prompt, zero code-level enforcement, on top of an unauthenticated admin plane that lets anyone rewrite the policy. Every remit hard-prohibition — fraud detection must always run, no above-threshold approvals without human sign-off, no auto-approval of unregistered vendors, runtime content must not alter goals — depends entirely on the LLM following natural-language instructions in get_system_prompt(); the approve_invoice tool itself performs no amount check, no vendor-status check, and no fraud-signal check before flipping payment_processed=True.

That LLM-only enforcement is then undermined by three reachable channels: unauthenticated POST /api/admin/finbot/goals appends operator-supplied text into the system prompt with an explicit "OVERRIDE ABOVE IF CONFLICTING" clause; unauthenticated POST /api/admin/finbot/config can set fraud_detection_enabled=False; and the system prompt itself instructs the agent to treat invoice description text — vendor-supplied data — as a legitimate signal for expedited approval, with the rule-based fallback hard-coding that override into Python. The result is a one-hop chain from anonymous HTTP to fraudulent payment, acknowledged in the in-tree CTF walkthrough.

Scope of Analysis
Python Flask application backed by SQLAlchemy/SQLite, deployed to Render under Gunicorn. A single FinBotAgent class in src/services/finbot_agent.py orchestrates OpenAI function-calling with five tools (get_invoice_details, approve_invoice, reject_invoice, request_human_review, detect_fraud_patterns) plus a rule-based _fallback_processing path. Three Flask blueprints (user, vendor, admin) are registered at /api with CORS open to all origins and no authentication middleware; the admin blueprint exposes POST /api/admin/finbot/goals and POST /api/admin/finbot/config which write directly into the system prompt and toggle fraud detection. Vendor-supplied invoice description text is concatenated into the LLM context through process_invoice() and also drives the rule-based fallback's scoring.
