BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is a deploy-only bundler that mostly fulfils its remit's
central guarantee — only declared project sources land in the artifact,
secrets are deliberately routed around the _seed.json payload, and a deploy
that would expose the deployed API anonymously hits an explicit colour-coded
warning and a [y/N] confirmation gate in _deploy() before any push. Around
that core, the gaps are concentrated in supply-chain hygiene of declared
remote MCP servers and in the CLI's own observability:
_validate_mcp_for_deploy rejects stdio transports but accepts plain http://
URLs and applies no version-pin or integrity check to the remote server it
ships into the bundle, the generated bundle's pyproject.toml floors deepagents
at 0.5.3 without an upper bound, and the deploy/dev paths log only via Python
logging while the CLI never configures a handler — so almost every deploy
decision (bundle contents, anonymous-auth confirmations, hub-seed outcomes) is
print-only and unrecoverable after the run. Stated SHOULD items in the remit —
a current threat model and a documented disclosure process — are also off: the
threat model present is explicitly marked as pre-0.1.0 and there is no
published disclosure path inside this package.
