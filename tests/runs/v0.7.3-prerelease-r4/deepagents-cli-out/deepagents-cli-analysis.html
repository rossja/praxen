Executive Summary
Agent Remit (as declared)
A local developer command-line tool that scaffolds, bundles, and ships a deepagents agent project to a managed deployment platform (or runs it on a local langgraph dev server). The CLI is the deepagents-cli package and exposes three subcommands — init, dev, and deploy — and must bundle only from the project's declared sources (configuration, system prompt, skills, declared MCP servers, declared subagents) under direct developer supervision. A deploy configured to leave the deployed API open to anyone with its URL MUST surface a prominent warning and obtain explicit operator confirmation before it proceeds; credentials must never be folded into the seeded-memory or skills payload.
Behavior Summary (as observed)
The dominant pattern is a deploy-only bundler that mostly fulfils its remit's central guarantee — only declared project sources land in the artifact, secrets are deliberately routed around the _seed.json payload, and a deploy that would expose the deployed API anonymously hits an explicit colour-coded warning and a [y/N] confirmation gate in _deploy() before any push. Around that core, the gaps are concentrated in supply-chain hygiene of declared remote MCP servers and in the CLI's own observability: _validate_mcp_for_deploy rejects stdio transports but accepts plain http:// URLs and applies no version-pin or integrity check to the remote server it ships into the bundle, the generated bundle's pyproject.toml floors deepagents at 0.5.3 without an upper bound, and the deploy/dev paths log only via Python logging while the CLI never configures a handler — so almost every deploy decision (bundle contents, anonymous-auth confirmations, hub-seed outcomes) is print-only and unrecoverable after the run. Stated SHOULD items in the remit — a current threat model and a documented disclosure process — are also off: the threat model present is explicitly marked as pre-0.1.0 and there is no published disclosure path inside this package.
Scope of Analysis
Python CLI built on argparse (deepagents_cli/main.py), with the entire deploy/dev/init pipeline in deepagents_cli/deploy/ — commands.py orchestrates each subcommand, bundler.py assembles the build artifact, config.py parses and validates deepagents.toml, templates.py carries the per-provider auth/sandbox/MCP code blocks emitted into the generated deploy_graph.py, and context_hub.py is the LangSmith Hub backend vendored alongside hub-backed bundles. The bundler reads only the canonical project layout (AGENTS.md, skills/, user/, mcp.json, subagents/, .env) and shells out to the langgraph CLI for the actual push to LangSmith Deployments. The repo ships a committed uv.lock (4031 lines), a THREAT_MODEL.md (now stale — it covers the pre-0.1.0 interactive REPL that has moved to deepagents-code), and a .mcp.json at the monorepo root holding two public LangChain documentation MCP servers reached over https://.

Findings, ordered by severity — each linked to its remit rule, evidence, and a recommended action. Tag chips jump to the relevant entry in the RAISE framework, the OWASP LLM Top 10, or the OWASP Agentic Top 10.

HIGH PRAX-2026-05-25-001 `_validate_mcp_for_deploy` blocks stdio transport but accepts plain `http://` URLs, so a non-TLS remote MCP server can be carried into the deployed bundle.
Implement Zero Trust LLM03 — Supply Chain All remote connections use TLS
Policy Rule — R-04 (Worker Remit):
"Remote MCP servers carried into the bundle MUST be reached over TLS, and their transport configuration MUST be validated before the bundle is shipped."
libs/cli/deepagents_cli/deploy/config.py:281 — _validate_mcp_for_deploy iterates servers and only flags transport == 'stdio'; the URL scheme is never inspected, so http:// passes through unchanged. libs/cli/deepagents_cli/deploy/bundler.py:194 — Validated mcp.json is copied verbatim to build_dir/_mcp.json (shutil.copy2) and loaded by the generated MCP_TOOLS_TEMPLATE at runtime.
Recommended Action
  • In `_validate_mcp_for_deploy`, after the stdio rejection, parse each remote server's URL and append a validation error when the scheme is not `https` (allowing `http://localhost` only if local dev is explicitly intended).
  • Mirror the same scheme check in the subagent path (`load_subagents` calls `_validate_mcp_for_deploy`) so subagent MCP entries can't bypass it.
HIGH PRAX-2026-05-25-002 Remote MCP servers in `mcp.json` carry no version pin or integrity check — the bundler ships whatever the URL is currently serving.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities Tool definitions are signed and version-pinned
Policy Rule — R-09 (Worker Remit):
"Remote MCP servers declared in the project MUST be pinned to a known-good, integrity-checked version, so the deployed agent does not auto-install an unpinned server afresh."
libs/cli/deepagents_cli/deploy/config.py:281 — _validate_mcp_for_deploy checks only the transport kind; no version, etag, sha, or pin field is read from the server entry. .mcp.json:1 — Root-level config carries `docs-langchain` and `reference-langchain` MCP servers as bare `type=http` + `url` entries, with no version or integrity marker — the pattern users are expected to copy.
Recommended Action
  • Add an optional `version` or `integrity` field to the MCP server schema in `deploy/config.py` and surface a validation warning (or error, behind a flag) when a remote server is declared without one.
  • Document the expected pinning convention in the starter `generate_starter_mcp_json` so projects scaffolded by `deepagents init` start out conformant.
HIGH PRAX-2026-05-25-003 `libs/cli/THREAT_MODEL.md` is marked as describing the pre-0.1.0 surface; the deploy-only package that ships now has no current threat model.

  • Add a brief `SECURITY.md` (or link from the repo root one) covering the disclosure channel for `deepagents-cli` specifically.
HIGH PRAX-2026-05-25-004 Deploy and dev paths produce no durable log of what was bundled, who confirmed the anonymous-auth gate, or which hub repo was seeded.
Monitor Continuously LLM02 — Sensitive Information Disclosure
libs/cli/deepagents_cli/config.py:13 — _stderr docstring explicitly notes: 'The CLI does not configure logging handlers by default, so logger.warning output is invisible to end users.' libs/cli/deepagents_cli/deploy/commands.py:298 — Anonymous-deploy warning + [y/N] confirmation are print()-only; the answer is read into a local and discarded — no audit record persists after the process exits.
Recommended Action
  • Add an optional structured log target (e.g., `DEEPAGENTS_CLI_LOG_FILE`) and emit a single record per `deepagents deploy` invocation capturing config path, agent name, auth provider, anonymous-confirm answer, and the bundle's generated file list.
  • At minimum, install a default `logging.basicConfig` at WARN so the existing `logger.warning` calls reach the operator's terminal.
MEDIUM PRAX-2026-05-25-005 `deepagents init --force` overwrites every existing file in the target directory without per-file confirmation.
Implement Zero Trust ASI09 — Human-Agent Trust Exploitation
Policy Rule — R-08 (Worker Remit):
"Overwriting an existing project folder or its files during scaffolding MUST require an explicit operator opt-in."
libs/cli/deepagents_cli/deploy/commands.py:197 — _init_project guards only on directory existence: `if project_dir.exists() and not force: raise SystemExit(1)`; once --force is set, each file is written unconditionally. libs/cli/deepagents_cli/deploy/commands.py:210 — for filename, content in files: (project_dir / filename).write_text(content, ...) — no per-file existence check, no diff, no backup.
Recommended Action
  • In `_init_project`, when the project directory exists, check each file individually and either prompt per-file (interactive) or refuse to overwrite an existing `.env` even with `--force`.
  • At minimum, write a `.bak` copy of any existing file before overwriting, so `--force` is recoverable.
MEDIUM PRAX-2026-05-25-006 Generated `pyproject.toml` floors `deepagents==0.5.3` with no upper bound, leaving the deployed bundle to pick up future incompatible major versions.
Manage Your Supply Chain LLM03 — Supply Chain
libs/cli/deepagents_cli/deploy/templates.py:1129 — PYPROJECT_TEMPLATE writes `deepagents==0.5.3` followed by `{extra_deps}` items each appended as a bare package name with no version specifier. libs/cli/deepagents_cli/deploy/bundler.py:480 — _render_pyproject builds `deps` as plain package names: `deps.append(_MODEL_PROVIDER_DEPS[provider_prefix])` and `deps.append('langchain-mcp-adapters')` — no version pins emitted.
Recommended Action
  • In `_render_pyproject`, attach a tested version range to each appended dependency (or a generated lockfile shipped alongside the bundle), so `langgraph deploy` builds against a known-good resolution.
  • Bump the `deepagents` pin to the version the CLI is actually tested against (the CLI's own `pyproject.toml` floors at `0.6.1` — the bundle is shipping `0.5.3`).
MEDIUM PRAX-2026-05-25-007 `_run_langgraph_deploy` and `_dev` invoke `langgraph` via `subprocess.run` with `shutil.which` but no version check on the resolved binary.
Manage Your Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities
libs/cli/deepagents_cli/deploy/commands.py:560 — _run_langgraph_deploy: `if shutil.which('langgraph') is None: ...` then `subprocess.run(['langgraph', 'deploy', ...], cwd=str(build_dir), check=False, env=env)` — no path pinning, env is os.environ.copy() plus the analytics-source override. libs/cli/deepagents_cli/deploy/commands.py:417 — _dev does the same shutil.which lookup, then subprocess.run(['langgraph', 'dev', ...], cwd=str(build_dir), check=False) with all parent env inherited.
Recommended Action
  • In `_run_langgraph_deploy` and `_dev`, resolve `langgraph` to its absolute path once and log it (or pin to the version expected at install time), so an unexpected shadowing binary is visible to the operator.
  • Consider scrubbing model/LangSmith API keys from the env passed to `langgraph dev` when they are not needed for the dev server itself.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Anonymous-deploy confirmation gate is real and operator-visible

`_deploy()` detects `[auth].provider == "anonymous"` with `[frontend].enabled` and emits a bold-red ANSI warning plus a `[y/N]` interactive confirmation before any push — exactly what R-03 calls for, and it correctly suppresses the prompt on `--dry-run`.

libs/cli/deepagents_cli/deploy/commands.py:286

Bundle is assembled only from declared project sources

`_build_seed` in `bundler.py` enumerates exactly the canonical layout (`AGENTS.md`, `skills/`, `user/`, `subagents/`) and `.env` is deliberately copied as `.env`, never folded into `_seed.json` — the bundler honours R-02 and R-07 by construction.

libs/cli/deepagents_cli/deploy/bundler.py:328

Committed pinned lockfile for CLI dev dependencies

`libs/cli/uv.lock` (4031 lines) is committed to the repo and pinned with content hashes per wheel/sdist, giving the CLI's own runtime a reproducible dependency tree.

libs/cli/uv.lock:1

Strict TOML schema rejects unknown sections and keys

`_parse_config` allowlists exactly `agent`, `sandbox`, `auth`, `memories`, `frontend` and rejects any other top-level section with a clear migration error — preventing silent migration bugs and reducing the chance of accidental scope expansion through config drift.

libs/cli/deepagents_cli/deploy/config.py:447

Credentials are deliberately routed around the seeded-memory payload

`bundler.py` copies `.env` to `build_dir/.env` separately (R-06/R-07 compliance), and `_build_seed` reads only `AGENTS.md`, `skills/`, and `user/` — environment secrets never enter `_seed.json` which is committed to the LangSmith Hub repo.

libs/cli/deepagents_cli/deploy/bundler.py:200
