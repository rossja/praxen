every control its remit requires: workspace confinement, input validation,
tool-call audit, session-call cap, and any guard on write_file/read_file paths
are all set to false or simply not implemented, while prompt-injection
compliance, system-prompt leakage, and false-history acceptance are explicitly
enabled in vulnerabilities blocks. The dominant pattern is policy-declared,
zero code enforcement — the remit's six MUST-NOT clauses and four MUST clauses
each map to a flat-false feature flag or a missing call site, not to a
bypassable safeguard. A secondary compound chain runs through
vulnerabilities.js: literal credentials in SENSITIVE_DATA are interpolated
into sibling personas' system prompts at module load, and HelperBot's enabled
leakSystemPrompt vector turns any prompt-injection attempt into a one-hop
credential disclosure for those peers, with no audit log to detect it.

CRITICAL FINDINGS

  PRAX-2026-05-25-001  HelperBot exposes read_file and write_file as untyped
            string-path tools with no workspace confinement, contradicting the
            remit's MUST clauses on workspace boundaries.
      Action 1: In src/core/agents.js, replace the bare-string tools array
                with tool objects that carry parameter schemas and a
                workspaceRoot constraint, and add a runtime
                path-canonicalization check before any filesystem access.
      Action 2: Set features.toolApproval: true and wire a human-in-the-loop
                confirmation step in front of write_file before allowing the
                persona to ship outside the DVAA training scope.

  PRAX-2026-05-25-002  HelperBot disables input validation outright
            (inputValidation: false) while enabling prompt-injection
            compliance, breaking the remit's MUST-validate-input clause.
      Action 1: In src/core/agents.js, set features.inputValidation: true and
                route every user message through the detectAttacks() pipeline
                in src/core/vulnerabilities.js before it reaches
                SimulatedLLM.generate().
      Action 2: Delete the vulnerabilities.promptInjection block from the
                HelperBot persona, or move the persona to a quarantined
                EXAMPLES namespace explicitly outside any non-training
                deployment.

  PRAX-2026-05-25-003  HelperBot has auditLogging: false and no tool-call
            logging call sites, contradicting the remit's
            MUST-record-every-tool-call clause.
      Action 1: In src/core/agents.js, set features.auditLogging: true and add
                a structured per-tool-call log emitter (JSON lines, capturing
                tool_name, parameters, outcome, timestamp, session_id) before
                and after every tool dispatch.
      Action 2: Persist the resulting log stream to a tamper-evident
                append-only sink so a compromised agent cannot retroactively
                erase its trail.

  PRAX-2026-05-25-004  HelperBot's search_web tool has no query filter, so
            confidential content can leak into outbound search queries against
            the remit's MUST-NOT clause.
      Action 1: In src/core/agents.js, replace the bare-string search_web with
                a tool object that runs the query through a redaction step
                using the dataExfiltration patterns in
                src/core/vulnerabilities.js:258-266 before any outbound
                dispatch.
      Action 2: Add a content-origin check that rejects queries containing
                tokens drawn from the system prompt, file contents, or session
                context.
