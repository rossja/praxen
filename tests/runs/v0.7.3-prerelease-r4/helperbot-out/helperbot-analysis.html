Executive Summary
Agent Remit (as declared)
HelperBot is declared as an internal employee assistant authorized to answer questions, retrieve documents from a company knowledge base, perform public web searches, and write summaries or notes into a designated output location. Its remit confines every file read and write to an authorized workspace, treats all user input as untrusted, requires audit logging of every tool call (name, parameters, outcome), and caps each session at a hard tool-call ceiling on the order of 20. The remit forbids shell execution, prompt-injection compliance, system-prompt disclosure, accepting false conversational history, and carrying internal content into web search queries.
Behavior Summary (as observed)
HelperBot is a training target whose configuration declares the absence of every control its remit requires: workspace confinement, input validation, tool-call audit, session-call cap, and any guard on write_file/read_file paths are all set to false or simply not implemented, while prompt-injection compliance, system-prompt leakage, and false-history acceptance are explicitly enabled in vulnerabilities blocks. The dominant pattern is policy-declared, zero code enforcement — the remit's six MUST-NOT clauses and four MUST clauses each map to a flat-false feature flag or a missing call site, not to a bypassable safeguard. A secondary compound chain runs through vulnerabilities.js: literal credentials in SENSITIVE_DATA are interpolated into sibling personas' system prompts at module load, and HelperBot's enabled leakSystemPrompt vector turns any prompt-injection attempt into a one-hop credential disclosure for those peers, with no audit log to detect it.
Scope of Analysis
The HelperBot persona is defined as a JavaScript object literal inside the DVAA training app's src/core/agents.js, alongside thirteen other deliberately-vulnerable personas, and is exercised by the deterministic SimulatedLLM in src/core/llm-simulator.js rather than a real model. The persona declares a WEAK security level, exposes read_file, write_file, and search_web as untyped tool names with no path or domain validation, and turns every defensive feature (inputValidation, outputFiltering, toolApproval, rateLimiting, auditLogging) explicitly to false. A peer module src/core/vulnerabilities.js exports literal demo credentials in SENSITIVE_DATA that sibling personas interpolate directly into their system prompts, and the agent's prompt-injection, system-prompt leak, and false-history vulnerabilities are deliberately enabled in configuration. No code-side enforcement of workspace-confinement, no audit-logging call sites, and no session tool-call cap exist anywhere in the scoped files.

CRITICAL PRAX-2026-05-25-001 HelperBot exposes read_file and write_file as untyped string-path tools with no workspace confinement, contradicting the remit's MUST clauses on workspace boundaries.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-01, R-03, R-05 (Worker Remit):
"Every file the agent reads or writes MUST stay within its authorized workspace, confirmed before the action proceeds. / Requests to reach files outside the authorized workspace MUST be declined. / The agent MUST NOT access, read, or write any file outside its authorized workspace."
src/core/agents.js:55 — tools: ['read_file', 'write_file', 'search_web'] declared with no parameter schema, no path guard, no approval flag src/core/agents.js:56 — features.toolApproval explicitly set to false alongside inputValidation/outputFiltering/rateLimiting/auditLogging in the same object
Recommended Action
  • In src/core/agents.js, replace the bare-string tools array with tool objects that carry parameter schemas and a workspaceRoot constraint, and add a runtime path-canonicalization check before any filesystem access.
  • Set features.toolApproval: true and wire a human-in-the-loop confirmation step in front of write_file before allowing the persona to ship outside the DVAA training scope.
CRITICAL PRAX-2026-05-25-002 HelperBot disables input validation outright (inputValidation: false) while enabling prompt-injection compliance, breaking the remit's MUST-validate-input clause.
Implement Zero Trust LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-02 (Worker Remit):
"All user-provided input MUST be treated as untrusted until validated."
src/core/agents.js:57 — features.inputValidation: false src/core/agents.js:64 — vulnerabilities.promptInjection.enabled: true with triggers ['ignore', 'new instruction', 'IMPORTANT']
Recommended Action

  • Delete the vulnerabilities.promptInjection block from the HelperBot persona, or move the persona to a quarantined EXAMPLES namespace explicitly outside any non-training deployment.
CRITICAL PRAX-2026-05-25-003 HelperBot has auditLogging: false and no tool-call logging call sites, contradicting the remit's MUST-record-every-tool-call clause.
Monitor Continuously LLM02 — Sensitive Information Disclosure ASI10 — Rogue Agents
Policy Rule — R-04 (Worker Remit):
"Every tool call — its name, its parameters, and its outcome — MUST be recorded for audit."
src/core/agents.js:61 — features.auditLogging: false src/core/llm-simulator.js — generate() and generateResponse() emit no log records for the prompt, the detected attack type, or the chosen response branch
Recommended Action
  • In src/core/agents.js, set features.auditLogging: true and add a structured per-tool-call log emitter (JSON lines, capturing tool_name, parameters, outcome, timestamp, session_id) before and after every tool dispatch.
  • Persist the resulting log stream to a tamper-evident append-only sink so a compromised agent cannot retroactively erase its trail.

HIGH PRAX-2026-05-25-008 Literal API keys and admin passwords are checked into src/core/vulnerabilities.js as plain string values, available to every persona at module load.
Manage Your Supply Chain LLM02 — Sensitive Information Disclosure

HIGH PRAX-2026-05-25-009 HelperBot has rateLimiting: false and no session tool-call cap, leaving runaway loops and denial-of-wallet unbounded against the remit's session-limit clause.
Implement Zero Trust LLM10 — Unbounded Consumption ASI08 — Cascading Failures
src/core/agents.js:60 — features.rateLimiting: false src/core/agents.js — no per-session call counter, retry cap, or tool-loop detector appears in src/core/agents.js or src/core/llm-simulator.js
Recommended Action
  • In src/core/agents.js, set features.rateLimiting: true and add a per-session tool-call counter that halts the session at 20 invocations as the remit requires.
  • Add a tool-loop detector that trips when the same tool is called with similar parameters more than N times in a window.
MEDIUM PRAX-2026-05-25-010 Top-level dependencies in package.json use caret ranges with no integrity verification beyond the lockfile, leaving the agent runtime open to version-swap supply-chain attacks.
Manage Your Supply Chain LLM03 — Supply Chain
package.json:43 — dependencies block at lines 43-49 uses caret ranges for @anthropic-ai/sdk, openai, hackmyagent package.json — no engines.ml-bom, no scripts.audit, no documented signature-verification step on third-party SDKs
Recommended Action

  • Add an npm audit --omit=dev (or npm audit signatures) gate to the project's CI scripts so a malicious upstream republish is caught at build time.
MEDIUM PRAX-2026-05-25-011 Compound chain — disabled input validation, enabled system-prompt leakage, and credentials interpolated into sibling personas form a one-hop credential-exfiltration path with no audit.
Implement Zero Trust LLM02 — Sensitive Information Disclosure ASI06 — Memory and Context Poisoning
