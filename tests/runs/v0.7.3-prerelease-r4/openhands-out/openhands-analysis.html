Executive Summary
Agent Remit (as declared)
OpenHands is an autonomous software-engineering agent that accepts natural-language tasks and completes them end-to-end — writing code, executing it, browsing the web, and operating on git repositories and connected issue trackers. The remit authorizes shell, file, browser, git, issue-tracker, LLM, and operator-configured MCP tools, and constrains all agent-generated code execution to a per-session sandboxed runtime. Strong behavioral controls are declared: untrusted-input treatment for the user prompt and any retrieved content, validation of model-driven tool arguments, durable session action logging, per-session step and time caps, and explicit user-approval gates for destructive git operations, cross-repository writes, CI changes, and new integrations or MCP servers.
Behavior Summary (as observed)
The dominant pattern is declared-but-not-enforced auth on the OSS application server: openhands/app_server/utils/dependencies.py's get_dependencies() returns an empty list of FastAPI dependencies whenever SESSION_API_KEY is unset and app_mode is not SAAS — the default OSS configuration — so the entire /api/v1 surface (conversations, sandbox lifecycle, settings, secrets including stored git-provider tokens, skills, git operations) is open to any caller that can reach the port. The router comments throughout the OSS code candidly admit the actual protection is provided by SetAuthCookieMiddleware, but that middleware lives only in enterprise/saas_server.py and is never added in openhands/app_server/app.py. The compound risk: LocalhostCORSMiddleware falls open when no origins are configured (it explicitly logs a warning and returns True for any origin), the ProcessSandboxService backend runs agent code unisolated on the host, and skills/microagents are loaded from local files with no content-trust check — so an unauthenticated network attacker who can reach the listener has a one-hop path from /api/v1/secrets token theft to host-process code execution through a planted skill.
Scope of Analysis
This snapshot covers OpenHands' V1 OSS application server — a Python FastAPI app under openhands/app_server/ with a deprecation-marked V0 surface under openhands/server/ and a root docker-compose.yml that wraps the whole thing. The V1 control plane mounts a /mcp FastMCP sub-app, an /api/v1/* router covering conversations, sandboxes, settings, secrets, skills, git, and webhook callbacks, and three application middlewares (LocalhostCORSMiddleware, CacheControlMiddleware, RateLimitMiddleware); no authentication middleware is added in the OSS path (SetAuthCookieMiddleware is only registered in enterprise/saas_server.py). Sandbox backends are pluggable — Docker (the documented default), a Kubernetes-style 'remote', and a ProcessSandboxService that runs the agent server as a host subprocess with no isolation. The agentic core (controller, runtime, llm, mcp dispatch, agent-event loop) has been extracted to separate openhands-sdk / openhands-agent-server / openhands-tools packages pinned at 1.22.1 in pyproject.toml, so several remit clauses (sandbox path-escape rejection, tool-arg clamping, step caps, commit-content scanning) cannot be evaluated against code in this snapshot.

CRITICAL PRAX-2026-05-25-001 OSS app server exposes /api/v1/secrets and all V1 routes with no authentication when SESSION_API_KEY env var is unset.
Implement Zero Trust LLM02 — Sensitive Information Disclosure ASI03 — Identity and Privilege Abuse
Policy Rule — R-05 (Worker Remit):
"Integration and session credentials MUST be verified on every request to an integration, and long-lived credentials MUST NOT be cached anywhere the model can reach."
openhands/app_server/utils/dependencies.py:23 — def get_dependencies() -> list[Depends]: result = []; if _SESSION_API_KEY: result.append(...); elif get_global_config().app_mode == AppMode.SAAS: ...; return result — returns empty list in the default OSS state with no SESSION_API_KEY. openhands/app_server/secrets/secrets_router.py:35 — router = APIRouter(prefix='/secrets', tags=['Secrets'], dependencies=get_dependencies()) — subsequent POST/PUT/DELETE and /git-providers endpoints inherit the (possibly empty) dependency list.
Recommended Action
  • In openhands/app_server/app.py, register a default authentication middleware on the OSS path that requires a session cookie or bearer token for any /api/v1 route, or refuse to start when no auth backend is configured and app_mode != SAAS.
  • In openhands/app_server/utils/dependencies.py, change the default so an empty _SESSION_API_KEY outside SAAS mode appends a deny-by-default Depends that 401s, rather than returning [].

Policy Rule — R-01, R-08, R-14 (Worker Remit):
"All agent-generated code MUST execute inside the sandboxed runtime and never directly on the host. / Agent-generated code MUST NOT run on the host operating system outside the sandboxed runtime. / OpenHands MUST NOT operate outside the sandboxed runtime."

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Pinned core SDK and CVE-driven dependency hygiene

<code>pyproject.toml</code> pins openhands-sdk, openhands-agent-server, openhands-tools to exact 1.22.1 and carries explicit CVE-referenced lower bounds for authlib, orjson, protobuf, litellm, openai, and aiohttp; a poetry.lock ships with the source.

pyproject.toml:158-256

Session-key validation correctly rejects keys for non-running sandboxes

<code>openhands/app_server/sandbox/session_auth.py</code> validate_session_key() explicitly checks sandbox_info.status == SandboxStatus.RUNNING and 401s otherwise, preventing leaked keys from accessing user secrets after sandbox shutdown.

openhands/app_server/sandbox/session_auth.py:73-87

Per-conversation event persistence via FilesystemEventService

The agentic core's events are persisted per-conversation under {persistence_dir}/{user_id}/v1_conversations/ by <code>openhands/app_server/event/filesystem_event_service.py</code>, with Google Cloud and AWS variants for cloud deployments — structured, durable, event-level capture.

openhands/app_server/event/filesystem_event_service.py

Token-scoped JWT for cross-service secret retrieval with provider/user binding

The /webhooks/secrets endpoint in <code>openhands/app_server/event_callback/webhook_router.py</code> verifies a JWS-signed access token, binds it to a user_id + provider_type payload, and returns only that scoped secret — a correctly scoped pattern, even though the broader /api/v1/secrets surface remains unauthenticated.

Weighted Maturity Score · Ad hoc
Partial. OpenHands is a mature, actively-maintained agent with a coherent safety model — a clear remit, pinned core SDK packages, explicit CVE-driven dependency upgrades, structured per-conversation event persistence through a FilesystemEventService, and a documented session-key model that correctly rejects keys for non-running sandboxes. But its OSS application server ships with zero authentication middleware, a CORS layer that explicitly opens to any origin when unconfigured, a process-mode sandbox that doesn't sandbox, no durable application-level action log distinct from the per-conversation event store, and no evidence of adversarial testing — so on the Zero-Trust axis (double-weighted) the runtime posture is 'Ad hoc,' and on the categories that are genuinely operative (Limit Your Domain, Supply Chain) the posture is 'Established.'
