BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in prompt, zero deterministic
code-level enforcement. Every MUST-NOT in the remit — DML, DDL, admin
commands, multi-statement queries, out-of-scope tables, row caps, "double
check before execute" — is left to the LLM via SQL_PREFIX in prompt.py and the
toolkit's tool descriptions; QuerySQLDatabaseTool._run passes the
model-authored SQL string straight to SQLDatabase.run_no_throw with no
statement-type check, no scope check, and no row-cap. The remit itself
anticipates this and says so explicitly — "the agent's own logic MUST NOT be
the sole enforcement point for the prohibition on modifying the database" —
and the create_sql_agent docstring echoes the same message to operators,
pointing them at server-side least-privilege roles, statement timeouts, and
query guardrails. Compounding the agent-level gap, tool results (raw row
contents, table names, column names) are returned into the LLM context as
plain strings with no origin labeling, leaving a textbook
indirect-prompt-injection surface against the very instructions the prompt is
supposed to enforce.

  PRAX-2026-05-25-001  No code-level enforcement of "no DML / DDL / admin /
            multi-statement" — write prohibitions are prompt-only.
      Action 1: In tools/sql_database/tool.py, add a deterministic SQL parser
                step in QuerySQLDatabaseTool._run (e.g.,
                sqlglot.parse_one(query).walk()) that rejects any statement
                whose root node is not SELECT and rejects multi-statement
                scripts; raise a tool error rather than executing.
      Action 2: Document the agent-level allowlist as a defense-in-depth layer
                on top of the server-side read-only role, not a replacement
                for it, in the create_sql_agent docstring.

  PRAX-2026-05-25-002  Tool output (rows, table/column names) flows back into
            LLM context unlabeled — indirect-injection surface against the
            agent's own prompt.
      Action 1: Wrap tool returns in a structured envelope (e.g., {"source":
                "sql_db_query", "trust": "untrusted", "rows": [...]}) and
                document in the system prompt that fields under rows[] are
                data, not instructions; mirror the pattern in the schema and
                list tools.
      Action 2: Add an output-side classifier or string-pattern check that
                flags result content carrying instruction-like keywords
                ("ignore previous", "system:", "DROP TABLE") and surfaces a
                warning to the agent before the response is composed.
