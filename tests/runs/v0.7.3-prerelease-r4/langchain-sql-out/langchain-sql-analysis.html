Executive Summary
Agent Remit (as declared)
A natural-language interface to a single relational database, intended to answer read-only analytical questions for internal data consumers. The agent may use sql_db_list_tables, sql_db_schema, sql_db_query_checker, and sql_db_query against a pre-configured SQLAlchemy connection, and its only output is a direct natural-language answer in the current session. Data- and schema-modifying statements (DML, DDL, admin, multi-statement, out-of-scope tables) are forbidden outright — no approval path exists within the agent. The remit also requires bounded tool-call iterations and small default row limits, and explicitly states the agent's own logic MUST NOT be the sole enforcement point for the prohibition on modifying the database.
Behavior Summary (as observed)
The dominant pattern is policy declared in prompt, zero deterministic code-level enforcement. Every MUST-NOT in the remit — DML, DDL, admin commands, multi-statement queries, out-of-scope tables, row caps, "double check before execute" — is left to the LLM via SQL_PREFIX in prompt.py and the toolkit's tool descriptions; QuerySQLDatabaseTool._run passes the model-authored SQL string straight to SQLDatabase.run_no_throw with no statement-type check, no scope check, and no row-cap. The remit itself anticipates this and says so explicitly — "the agent's own logic MUST NOT be the sole enforcement point for the prohibition on modifying the database" — and the create_sql_agent docstring echoes the same message to operators, pointing them at server-side least-privilege roles, statement timeouts, and query guardrails. Compounding the agent-level gap, tool results (raw row contents, table names, column names) are returned into the LLM context as plain strings with no origin labeling, leaving a textbook indirect-prompt-injection surface against the very instructions the prompt is supposed to enforce.
Scope of Analysis
Python library code in langchain-community exposing the SQL agent through create_sql_agent in agent_toolkits/sql/base.py, a four-tool SQLDatabaseToolkit in agent_toolkits/sql/toolkit.py, and the per-tool implementations in tools/sql_database/tool.py (sql_db_query, sql_db_schema, sql_db_list_tables, sql_db_query_checker). The underlying SQLAlchemy wrapper lives in utilities/sql_database.py and exposes run, run_no_throw, get_table_info, and get_context. sql_db_query._run calls self.db.run_no_throw(query) with the LLM-generated string unchanged — no SQL parser, no statement-type allowlist, no row-cap in code; the top_k, "no DML", and "double check" instructions live only in the system prompt at agent_toolkits/sql/prompt.py. Result rows pass through truncate_word(..., length=300) in SQLDatabase.run and are returned to the LLM as the stringified Python list, with no provenance label distinguishing tool output from operator instructions.

CRITICAL PRAX-2026-05-25-001 No code-level enforcement of "no DML / DDL / admin / multi-statement" — write prohibitions are prompt-only.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-10, R-11, R-12, R-13 (Worker Remit):
"The agent MUST NOT execute any data-modifying statement — inserts, updates, deletes, merges, upserts, replacements, or anything else that changes row data. / The agent MUST NOT execute any schema-, permission-, or structure-modifying statement — creating, dropping, altering, truncating, renaming, granting, or revoking. / The agent MUST NOT execute administrative or superuser commands, including bulk loads, attach/detach operations, and maintenance commands. / The agent MUST NOT execute multi-statement queries or stored-procedure invocations that could conceal hidden data- or schema-modifying statements."
libs/community/langchain_community/tools/sql_database/tool.py:59 — return self.db.run_no_throw(query) — LLM-authored SQL string is passed straight to the executor with no parse, no allowlist, no statement-count check libs/community/langchain_community/agent_toolkits/sql/prompt.py:12 — DO NOT make any DML statements (INSERT, UPDATE, DELETE, DROP etc.) to the database. — the prohibition is a single prompt line, not a code gate
Recommended Action
  • In tools/sql_database/tool.py, add a deterministic SQL parser step in QuerySQLDatabaseTool._run (e.g., sqlglot.parse_one(query).walk()) that rejects any statement whose root node is not SELECT and rejects multi-statement scripts; raise a tool error rather than executing.
  • Document the agent-level allowlist as a defense-in-depth layer on top of the server-side read-only role, not a replacement for it, in the create_sql_agent docstring.
CRITICAL PRAX-2026-05-25-002 Tool output (rows, table/column names) flows back into LLM context unlabeled — indirect-injection surface against the agent's own prompt.
Balance Your Knowledge Base LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-15, R-16 (Worker Remit):
"The agent MUST NOT allow user-provided text to override its instructions, its role, or its prohibition on modifying the database. / The agent MUST NOT follow instructions embedded in query results, table names, column names, or row contents."
libs/community/langchain_community/tools/sql_database/tool.py:59 — return self.db.run_no_throw(query) — row payload is returned to the model as a stringified Python list with no origin tag libs/community/langchain_community/utilities/sql_database.py:589 — SQLDatabase.run lines 589-603 — stringifies result rows via str(res) after per-cell truncate_word(..., length=300); truncation reduces cell size but does not label content as untrusted
Recommended Action
  • Wrap tool returns in a structured envelope (e.g., {"source": "sql_db_query", "trust": "untrusted", "rows": [...]}) and document in the system prompt that fields under rows[] are data, not instructions; mirror the pattern in the schema and list tools.
  • Add an output-side classifier or string-pattern check that flags result content carrying instruction-like keywords ("ignore previous", "system:", "DROP TABLE") and surfaces a warning to the agent before the response is composed.

HIGH PRAX-2026-05-25-004 No structured logging of agent inputs, generated SQL, tool calls, or returned rows.
Monitor Continuously LLM02 — Sensitive Information Disclosure ASI10 — Rogue Agents
libs/community/langchain_community/agent_toolkits/sql/base.py — module is logging-silent; AgentExecutor is instantiated with verbose=verbose but no logger handle libs/community/langchain_community/tools/sql_database/tool.py — none of the four tool _run methods records the inbound SQL string, the result row count, or the elapsed time
Recommended Action
Add logging.getLogger(__name__) to tools/sql_database/tool.py and emit one structured record per tool invocation with fields {tool, query_sha256, row_count, dialect, elapsed_ms} — leave the raw query out by default to avoid PII or credential leakage in error messages.
HIGH PRAX-2026-05-25-005 "Double-check before execute" is an LLM-based check, not a deterministic gate; it can be skipped or fooled.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Explicit maintainer security warning in create_sql_agent docstring

The <code>create_sql_agent</code> docstring carries a prominent <code>!!! warning</code> block calling out arbitrary-SQL risk, naming least-privilege roles, statement timeouts, and query guardrails as required server-side controls, and pointing operators at Deep Agents for new production work. Honest, specific maintainer disclosure — exactly the shape the RAISE rubric credits.

libs/community/langchain_community/agent_toolkits/sql/base.py:71-97

Bounded iteration count enforced in code

<code>create_sql_agent</code> defaults <code>max_iterations</code> to 15 and forwards it to <code>AgentExecutor</code>, providing an actual runtime cap on tool-call loops — a real operative control, not a prompt suggestion.

libs/community/langchain_community/agent_toolkits/sql/base.py:58

Per-cell result truncation reduces injection surface and prompt-context bloat

<code>SQLDatabase</code> truncates each result cell to 300 characters via <code>truncate_word</code>, capping the per-row size of attacker-controlled content that can be fed back into the LLM. Does not validate or label content, but is a real width cap.

libs/community/langchain_community/utilities/sql_database.py:97

Schema-name sanitization rejects non-identifier characters

<code>sanitize_schema</code> in <code>utilities/sql_database.py</code> rejects any schema string not matching <code>^[a-zA-Z0-9_]+$</code>, preventing the schema argument from carrying an injected fragment when it is interpolated into <code>SET SCHEMA</code>-style statements for hana/duckdb/trino/etc.
