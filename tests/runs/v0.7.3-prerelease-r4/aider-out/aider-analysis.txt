BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is a real human-in-the-loop safety model with several
discrete gaps where a single primitive removes the human from the loop.
InputOutput.confirm_ask is wired into every destructive path — new-file
creation, edits to files not yet added, shell-command execution (with
explicit_yes_required=True and allow_never=True), and lint-fix retries — and
the auto-commit, auto-lint, and gitignore-respecting edit flow back this with
a durable on-disk record. The gaps are concentrated: Coder.abs_root_path does
not enforce repo containment so /read-only and /add accept absolute and
~-expanded paths outside the repo (violating the remit's hard "MUST NOT read
files outside the repository root" clause); the --watch-files flow turns an #
ai! comment in any tracked file into an LLM turn with no per-trigger
confirmation, opening a one-hop indirect-injection path from any tool or
process that can write into the working tree; and there is no secret-scanner
over file contents going into LLM context, no instruction-content
neutralization on comments/docstrings, and --git-commit-verify defaults to
False (i.e. git commit --no-verify is the default for aider edits, violating
the remit's "MUST NOT alter version-control internals" intent in spirit even
though hook files are not edited).
