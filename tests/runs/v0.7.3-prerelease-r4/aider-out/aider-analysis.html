Executive Summary
Agent Remit (as declared)
Aider is an interactive AI pair programmer that runs locally in a developer's terminal, reads source files in the current git repository, proposes edits in response to natural-language requests, applies approved edits, and commits them to git with a descriptive message. Authorized capabilities are confined to the current repo and to the developer's configured LLM provider; shell commands, tests, linters, and pushes require explicit developer invocation, and the agent is forbidden from reading or writing outside the repository, executing unconfirmed commands, pushing to remotes, or modifying version-control hooks. Sensitive content — secrets, ignored credential files, and prompt injections embedded in repository content — must be treated as untrusted, redacted from LLM context, and never written into git history.
Behavior Summary (as observed)

The dominant pattern is a real human-in-the-loop safety model with several discrete gaps where a single primitive removes the human from the loop. InputOutput.confirm_ask is wired into every destructive path — new-file creation, edits to files not yet added, shell-command execution (with explicit_yes_required=True and allow_never=True), and lint-fix retries — and the auto-commit, auto-lint, and gitignore-respecting edit flow back this with a durable on-disk record.

The gaps are concentrated: Coder.abs_root_path does not enforce repo containment so /read-only and /add accept absolute and ~-expanded paths outside the repo (violating the remit's hard "MUST NOT read files outside the repository root" clause); the --watch-files flow turns an # ai! comment in any tracked file into an LLM turn with no per-trigger confirmation, opening a one-hop indirect-injection path from any tool or process that can write into the working tree; and there is no secret-scanner over file contents going into LLM context, no instruction-content neutralization on comments/docstrings, and --git-commit-verify defaults to False (i.e. git commit --no-verify is the default for aider edits, violating the remit's "MUST NOT alter version-control internals" intent in spirit even though hook files are not edited).

Scope of Analysis
Python CLI built around aider.main:main invoking a Coder orchestrator (aider/coders/base_coder.py) with a Commands slash-command surface (aider/commands.py) and an InputOutput confirmation/console layer (aider/io.py). Edits run through allowed_to_edit → apply_updates → auto_commit, with confirm_ask calls gating new-file creation, edits to files not yet added to chat, shell commands proposed by the LLM, and lint-fix retries; shell execution is centralized in aider/run_cmd.py. Path resolution for added/read-only files goes through Coder.abs_root_path (base_coder.py:566), which calls Path(root)/path then utils.safe_abs_path (resolve) and does not enforce repo containment. External surfaces include an optional file watcher (aider/watch.py) that triggers an LLM turn when an # ai! / # ai? comment is detected in any tracked file, a web scraper (aider/scrape.py, invoked via /web), PyPI version checks (aider/versioncheck.py), and opt-in PostHog/Mixpanel analytics (aider/analytics.py).

CRITICAL PRAX-2026-05-25-001 abs_root_path does not enforce repo containment — /read-only and /add accept absolute and ~-expanded paths outside the repository.

Policy Rule — R-03, R-08 (Worker Remit):
"Aider MUST NOT read or write any file outside the current repository — including via absolute paths, symlinks, or parent-directory traversal. / All file reads and writes MUST be confined to paths that resolve inside the current repository; absolute paths, escaping symlinks, and parent-directory traversal MUST be rejected."

Policy Rule — R-10, R-14 (Worker Remit):
