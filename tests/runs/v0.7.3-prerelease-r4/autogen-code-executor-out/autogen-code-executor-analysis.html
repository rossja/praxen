Executive Summary
Agent Remit (as declared)
The AutoGen Code Execution Subsystem is the execution tier of a two-agent generator/executor pattern — it accepts code (Python or shell) produced by a paired LLM agent and returns stdout, stderr, and exit status from a controlled environment. Five executor kinds are authorized — LocalCommandLineCodeExecutor, DockerCommandLineCodeExecutor, DockerJupyterCodeExecutor, JupyterCodeExecutor, and ACADynamicSessionsCodeExecutor — with Docker the standard production path and local execution restricted to ephemeral, operator-approved sandboxes. The remit mandates work-directory confinement, wall-clock timeouts, captured stdio, host isolation, per-execution audit logging, and human approval before local-host execution, volume-mount overrides, network egress, or resource-ceiling increases.
Behavior Summary (as observed)
The dominant pattern is a sandbox that has the shape of isolation but not the substance: Docker and Azure are available paths, but the defaults the executor accepts make every path a sharp tool. create_default_code_executor() silently downgrades to LocalCommandLineCodeExecutor on any Docker failure (only a warnings.warn to stderr is interposed), LocalCommandLineCodeExecutor.__init__ itself merely warns — never gates — and copies the host's full os.environ into every child process, and DockerCommandLineCodeExecutor.start constructs the container with no user=, read_only=, mem_limit=, cap_drop=, pids_limit=, or network_mode="none" argument while bind-mounting the work directory as rw. Layered on top, no executor records a per-execution audit entry (timestamp, source agent, code digest, exit status) — logging.info/debug calls only cover infrastructure events like image pulls and container stop — so the remit's audit clause has no implementation anywhere in the subsystem.
Scope of Analysis
A Python library in python/packages/autogen-ext/src/autogen_ext/code_executors/ exposing five concrete subclasses of the CodeExecutor ABC defined in autogen-core/src/autogen_core/code_executor/_base.py. LocalCommandLineCodeExecutor (local/__init__.py) launches asyncio.create_subprocess_exec directly on the host; DockerCommandLineCodeExecutor (docker/_docker_code_executor.py) and DockerJupyterCodeExecutor (docker_jupyter/_docker_jupyter.py + _jupyter_server.py) use docker.from_env() to start containers; JupyterCodeExecutor (jupyter/_jupyter_code_executor.py) runs an in-process NotebookClient; ACADynamicSessionsCodeExecutor (azure/_azure_container_code_executor.py) posts to an Azure pool-management endpoint. A top-level create_default_code_executor() helper in code_executors/__init__.py chooses Docker if docker_client.ping() succeeds and otherwise falls back to LocalCommandLineCodeExecutor after emitting a UserWarning.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Work-directory containment for inline filename hints

<code>_common.get_file_name_from_content</code> resolves the declared filename and rejects anything outside the workspace via <code>path.relative_to(workspace_path.resolve())</code> — a real, code-level containment check, not a prompt-level one.

python/packages/autogen-ext/src/autogen_ext/code_executors/_common.py:96-111

Wall-clock timeouts wired through every executor

Every executor accepts a <code>timeout: int</code> constructor argument (validated <code>>= 1</code>), wraps execution in <code>asyncio.wait_for</code> or invokes the in-container <code>timeout</code> binary at <code>docker/_docker_code_executor.py:360</code>, and surfaces a 124 exit code on overrun — the remit's timeout-and-terminate rule is the most consistently implemented control in scope.

python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:441

Cancellation propagation via CancellationToken.link_future

Each <code>execute_code_blocks</code> implementation links the in-flight asyncio task to the caller's cancellation token, so a caller-side cancel can terminate the subprocess or kernel call cleanly; Docker additionally schedules a <code>pkill -f</code> against the running command via the kept event-loop reference.

python/packages/autogen-ext/src/autogen_ext/code_executors/docker/_docker_code_executor.py:287-325

Container auto_remove and stop_container defaults

<code>DockerCommandLineCodeExecutor</code> defaults both <code>auto_remove</code> and <code>stop_container</code> to <code>True</code>, registering an <code>asyncio_atexit</code> cleanup, so a forgotten executor does not leak a long-lived container.

python/packages/autogen-ext/src/autogen_ext/code_executors/docker/_docker_code_executor.py:553-560
