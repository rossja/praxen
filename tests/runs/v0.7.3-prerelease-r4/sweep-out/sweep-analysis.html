Executive Summary
Agent Remit (as declared)
Sweep is an autonomous GitHub App that reads issues and comments, plans code changes, and opens pull requests for human review. Authorized counterparties are GitHub (read-write on installed repositories), the LLM provider (inference context only), and the issue/PR author whose text is treated as untrusted input. Sweep must never merge its own PRs, force-push, run generated code, follow injected instructions, commit secrets, or modify CI/CD or security-sensitive files without explicit per-issue authorization and out-of-band human approval. Every character of issues, comments, file contents, and diffs is supposed to be screened for instruction injection before reaching model context, and every run is supposed to be logged in a reconstructible record.
Behavior Summary (as observed)
The dominant pattern is policy declared in the remit, never enforced in scope-readable code: every untrusted-input rule in the Behavioral Constraints section (R-08, R-09, R-15) has zero corresponding implementation — issue bodies, PR comments, and file diffs flow straight into Anthropic messages arrays with no neutralization, no quote-wrapping, and no instruction-pattern detector. That gap meets a second pattern of control primitives present in the codebase but wired out: WEBHOOK_SECRET is declared in config/server.py and consumed by verify_signature, but the verifier returns True when the secret is unset (fail-open), and is_blocked() is defined in sweep_bot.py with no caller anywhere in the repo. Compounding both is a row of four subprocess.run(..., shell=True) sites whose arguments include LLM-output or untrusted-issue text — the most direct of these is the ripgrep tool in question_answerer.py, where the model's chosen search query is " ".join-ed into a shell command line with no escaping. Together these form a credible injection-to-shell chain that the remit explicitly forbids (R-14, R-15).
Scope of Analysis
Python 3.10 FastAPI service backed by Redis, MongoDB (for chat-logger persistence), and a third-party LLM provider (Anthropic by default, with OpenAI / Azure / AWS Bedrock alternates). The in-scope agent surface is sweepai/agents/ (modify, question_answerer, search_agent, distill_issue, summarize_*) plus sweepai/core/ (sweep_bot, chat, prompts, context_pruning, dynamic_context_bot, review_utils, pull_request_bot) — orchestration is single-process tool-use over Anthropic streaming with a fixed inventory of code-edit tools (make_change, create_file, submit_task) and read-only search tools (semantic_search, vector_search, ripgrep, view_file). Configuration lives in sweepai/config/server.py (env-var ingest, secret/API-key surface) and sweepai/config/client.py (per-repo SweepConfig and sweep.yaml ingestion). sweepai/web/ defines pydantic event models, a /health endpoint, and a one-line check_sandbox_health stub; webhook routing itself is in the out-of-scope sweepai/api.py. Notable surfaces relative to the remit: four subprocess.run(..., shell=True) sites invoking rg and github-linguist with LLM-derived or repo-derived arguments, a hardcoded PostHog publishable key, a declared-but-never-consulted WEBHOOK_SECRET fail-open default, and an is_blocked() blocked-dirs gate defined in sweep_bot.py that has zero callers.

CRITICAL PRAX-2026-05-25-001 Webhook HMAC verification fails open when WEBHOOK_SECRET is unset — declared-but-defeated counterparty trust control.
Implement Zero Trust LLM06 — Excessive Agency ASI03 — Identity and Privilege Abuse
Policy Rule — R-20 (Worker Remit):
"GitHub — read-write on installed repositories, scoped by the GitHub App's granted permissions."
sweepai/config/server.py:204 — WEBHOOK_SECRET = os.environ.get("WEBHOOK_SECRET", None) — no assert, no production-mode required-flag check sweepai/utils/hash.py:22 — if not WEBHOOK_SECRET: return True (lines 22-24) — explicit fail-open branch in the function that protects every webhook entry
Recommended Action
  • In sweepai/config/server.py, replace the None default for WEBHOOK_SECRET with an assert WEBHOOK_SECRET, "WEBHOOK_SECRET is required." guard at module load (matching the pattern already used for OPENAI_API_KEY and COHERE_API_KEY at lines 210-211), so the service refuses to start without it.
  • In sweepai/utils/hash.py, change the fail-open branch to return False (reject) when WEBHOOK_SECRET is unset rather than True (accept), so a misconfigured deployment fails closed instead of accepting unsigned traffic.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Loop and iteration caps are present

The LLM call loop is bounded by MAX_CALLS=10 at the continuous_llm_calls call sites in sweep_bot.py (lines 564, 602, 628, 879, 917, 943, 1471, 1483, 1525) and the context-pruning agent is bounded by MAX_ITERATIONS=25 at context_pruning.py:28. Runaway-loop risk from a goal-hijack is real but not unbounded.

sweepai/core/sweep_bot.py:564, sweepai/core/context_pruning.py:28

Code-edit tool inventory is narrow

The modify agent's tool inventory is exactly make_change, create_file, and submit_task (declared in modify_utils.py:22-61); no delete_file, send, merge, or exec tool is surfaced to the model. The blast radius of any single misbehaving turn is bounded to non-destructive in-place edits and new-file creation.

sweepai/agents/modify_utils.py:22-61

Dependencies pinned to specific versions

pyproject.toml pins all 50+ runtime dependencies to exact patch versions (e.g. anthropic==0.21.3, openai==1.13.3, fastapi==0.110.0), reducing dependency-confusion and version-swap attack surface.
