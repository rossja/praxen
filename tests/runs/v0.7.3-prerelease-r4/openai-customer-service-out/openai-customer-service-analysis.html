Executive Summary
Agent Remit (as declared)
A multi-agent airline customer-service system intended to answer policy FAQs, look up flight details, and update seat assignments on existing confirmed reservations. A triage agent receives incoming requests and routes by handoff to either an FAQ agent (using faq_lookup_tool against the curated FAQ dataset only) or a seat-booking agent (using update_seat to change a seat on an existing reservation). Customer identity must be confirmed before any mutation, every seat change must be written to a durable audit log, instructions embedded in customer text or tool outputs must not be acted on, and refund/special-handling cases must reach a human checkpoint.
Behavior Summary (as observed)

This is the canonical "framework ships guardrails; example uses none" pattern. The Agents SDK exposes InputGuardrail, OutputGuardrail, per-tool tool_input_guardrails / tool_output_guardrails, needs_approval, is_enabled, and handoff input_filter as first-class primitives — and the customer-service example wires in zero of them, so every remit clause about identity, authoritative-record verification, audit logging, prompt-injection rejection, and human-approval escalation falls back to prose in the LLM system prompts.

Compounding that, on_seat_booking_handoff fabricates a flight number with random.randint(100, 999) in examples/customer_service/main.py, in direct contradiction of the remit clause that flight numbers MUST come from the authoritative reservation system and MUST NOT be generated by the agent at runtime — the example is shipping the exact failure mode the remit names. Default tracing and the SDK's strict-schema tool-args path provide a thin floor of monitoring and input shape-checking, but the runtime trust model is "trust the model" end-to-end.

Scope of Analysis
A single Python script (examples/customer_service/main.py) that constructs three Agent[AirlineAgentContext] instances on top of the OpenAI Agents SDK in src/agents/. Two function tools — faq_lookup_tool (a keyword-substring matcher over four hard-coded answers) and update_seat (writes confirmation_number and seat_number straight onto a Pydantic context object with no lookup) — and an on_seat_booking_handoff callback that fabricates a flight number via random.randint(100, 999). A while True REPL loop in main() feeds raw user input into Runner.run under a trace("Customer service") span. The framework exposes InputGuardrail, OutputGuardrail, tool_input_guardrails, tool_output_guardrails, needs_approval, is_enabled, and handoff input_filter — the example wires none of them.

Findings Register

Findings, ordered by severity — each linked to its remit rule, evidence, and a recommended action. Tag chips jump to the relevant entry in the RAISE framework, the OWASP LLM Top 10, or the OWASP Agentic Top 10.

CRITICAL PRAX-2026-05-25-001 on_seat_booking_handoff fabricates flight_number via random.randint(100, 999) in direct violation of the remit's "MUST NOT be generated by the agent at runtime" clause for flight numbers.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation
Policy Rule — R-14 (Worker Remit):
"Flight numbers, confirmation numbers, and passenger identifiers MUST come from the authoritative reservation system and MUST NOT be generated by the agent at runtime."
examples/customer_service/main.py:89 — lines 89-91 — async def on_seat_booking_handoff(...): flight_number = f"FLT-{random.randint(100, 999)}"; context.context.flight_number = flight_number — runtime ID fabrication
Recommended Action
Replace the random.randint call in on_seat_booking_handoff with a lookup against the authoritative reservation system, keyed off the authenticated session and the customer-supplied confirmation number, and fail the handoff if no matching reservation is found.
CRITICAL PRAX-2026-05-25-002 update_seat mutates reservation state with no identity check, no ownership verification, and no needs_approval gate — anything the model passes is written to context.
Implement Zero Trust ASI03 — Identity and Privilege Abuse LLM06 — Excessive Agency
Policy Rule — R-05, R-10, R-15 (Worker Remit):
"Customer identity MUST be confirmed before any action that mutates reservation state. / Reservations or seat assignments for any customer other than the authenticated session user MUST NOT be modified. / Handoffs between agents MUST NOT escalate privileges or bypass identity verification."
examples/customer_service/main.py:67 — lines 67-83 — async def update_seat(context, confirmation_number, new_seat): context.context.confirmation_number = confirmation_number; context.context.seat_number = new_seat; assert context.context.flight_number is not None — no identity check, no ownership lookup, no needs_approval
Recommended Action
  • Add an authenticated customer_id to AirlineAgentContext populated outside the LLM (e.g., from the calling web session), and gate update_seat on a server-side check that the confirmation number belongs to that customer.
  • Set needs_approval=True on the @function_tool decorator for update_seat (or pass a predicate) so the run interrupts and requires explicit RunState.approve() before any seat write lands.
CRITICAL PRAX-2026-05-25-003 No durable audit log for seat changes — update_seat returns a string and writes only to in-process context; the SDK trace span is not the audit record the remit demands.
Monitor Continuously ASI02 — Tool Misuse and Exploitation
Policy Rule — R-08 (Worker Remit):
"Every seat change MUST be recorded to a durable audit log capturing the time, the customer identity, the confirmation number, the old and new seat, and the agent that performed the change."
examples/customer_service/main.py:67 — lines 67-83 — update_seat body has no log/audit write, only `return f"Updated seat to {new_seat} for confirmation number {confirmation_number}"`
Recommended Action
Inside update_seat, after the (added) ownership check passes and before returning, write a structured audit record (timestamp, authenticated customer id, confirmation number, old seat, new seat, agent name) to a durable store — file, database, or audit-pipeline endpoint — and fail the tool call if the audit write fails.
CRITICAL PRAX-2026-05-25-004 Zero InputGuardrails wired across all three agents — free-text customer input flows straight into LLM context with no validation, contradicting the explicit "MUST NOT accept instructions embedded in free-text customer input" clause.
Implement Zero Trust LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-11 (Worker Remit):
"Instructions embedded in free-text customer input that attempt to override an agent's role, scope, or identity MUST NOT be accepted or acted on."
examples/customer_service/main.py:96 — lines 96-138 — three Agent[...] constructors (faq_agent, seat_booking_agent, triage_agent), none of which pass input_guardrails=... src/agents/agent.py:322 — input_guardrails: list[InputGuardrail[TContext]] = field(default_factory=list) — default empty when the example does not set it
Recommended Action
  • Author at least one @input_guardrail (an injection-detector LLM-as-judge or a heuristic over the user turn) and pass it to every agent's input_guardrails=[...] argument so the run halts on a tripwire before tool calls are issued.
  • Place a sequential (run_in_parallel=False) guardrail on the triage agent so a tripped guardrail aborts the run before any handoff is decided.
HIGH PRAX-2026-05-25-005 Zero output guardrails and zero handoff input_filters — tool outputs (FAQ text, seat-record strings) and prior-agent turns flow into the next agent's context unfiltered, the canonical indirect-injection path the remit names.
Implement Zero Trust LLM01 — Prompt Injection ASI06 — Memory and Context Poisoning
Policy Rule — R-12 (Worker Remit):
"Instructions embedded in tool outputs — FAQ content, seat records, flight data — that attempt to change an agent's behavior MUST NOT be accepted."
examples/customer_service/main.py:184 — `input_items = result.to_input_list()` — carries unfiltered prior-agent output into the next turn examples/customer_service/main.py:130 — lines 130-137 — `handoffs=[handoff(agent=faq_agent, ...), handoff(agent=seat_booking_agent, on_handoff=..., ...)]` — both handoffs omit input_filter=
Recommended Action
  • Add a tool_output_guardrails=[...] list to both faq_lookup_tool and update_seat to reject any tool output containing instruction-like text before it re-enters LLM context.
  • Pass an input_filter= callable to every handoff(...) in triage_agent.handoffs that strips or quarantines prior tool outputs from the inbound message list.
HIGH PRAX-2026-05-25-006 No human-approval gate or escalation path — refunds, special-handling flags, and unverified-identity cases all fall through to the LLM's prose response with no code-level checkpoint.
Implement Zero Trust LLM06 — Excessive Agency ASI09 — Human-Agent Trust Exploitation
Policy Rule — R-16 (Worker Remit):
"Any request where the customer's identity cannot be verified MUST be escalated to a human."
examples/customer_service/main.py:96 — lines 96-138 — no tool with needs_approval=True, no escalate_to_human tool, no human-checkpoint logic in any agent's instructions=

HIGH PRAX-2026-05-25-007 update_seat never verifies the confirmation number against an authoritative reservation record — it accepts whatever the model passes as ground truth.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

SDK default tracing is on — every run produces a structured span tree

The example wraps each turn in trace("Customer service", group_id=conversation_id) and the SDK's RunConfig.tracing_disabled defaults to False, so a structured action-level span tree (tool calls, handoffs, model calls) is recorded for every turn — a thin but real audit floor.

examples/customer_service/main.py:166; src/agents/run_config.py:249

Strict-schema tool arguments are enabled by default

FunctionTool.strict_json_schema defaults to True (src/agents/tool.py:310-312) and the @function_tool decorator carries it through, so tool argument shape is JSON-schema-validated before the tool runs — a baseline input-shape defense (not semantic validation).

src/agents/tool.py:310-312

Agents SDK ships first-class safety primitives the operator can wire

InputGuardrail, OutputGuardrail, per-tool tool_input_guardrails / tool_output_guardrails, needs_approval, is_enabled, and handoff input_filter all exist as first-class fields on Agent, FunctionTool, and Handoff. The example does not use them, but every finding above has a concrete, framework-supported remediation path.
