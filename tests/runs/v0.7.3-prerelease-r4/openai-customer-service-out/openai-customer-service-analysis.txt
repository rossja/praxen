BEHAVIOR SUMMARY
------------------------------------------------------------------------------
This is the canonical "framework ships guardrails; example uses none" pattern.
The Agents SDK exposes InputGuardrail, OutputGuardrail, per-tool
tool_input_guardrails / tool_output_guardrails, needs_approval, is_enabled,
and handoff input_filter as first-class primitives — and the customer-service
example wires in zero of them, so every remit clause about identity,
authoritative-record verification, audit logging, prompt-injection rejection,
and human-approval escalation falls back to prose in the LLM system prompts.
Compounding that, on_seat_booking_handoff fabricates a flight number with
random.randint(100, 999) in examples/customer_service/main.py, in direct
contradiction of the remit clause that flight numbers MUST come from the
authoritative reservation system and MUST NOT be generated by the agent at
runtime — the example is shipping the exact failure mode the remit names.
Default tracing and the SDK's strict-schema tool-args path provide a thin
floor of monitoring and input shape-checking, but the runtime trust model is
"trust the model" end-to-end.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-25-001  on_seat_booking_handoff fabricates flight_number via
            random.randint(100, 999) in direct violation of the remit's "MUST
            NOT be generated by the agent at runtime" clause for flight
            numbers.
      Action: Replace the random.randint call in on_seat_booking_handoff with
              a lookup against the authoritative reservation system, keyed off
              the authenticated session and the customer-supplied confirmation
              number, and fail the handoff if no matching reservation is
              found.

  PRAX-2026-05-25-002  update_seat mutates reservation state with no identity
            check, no ownership verification, and no needs_approval gate —
            anything the model passes is written to context.
      Action 1: Add an authenticated customer_id to AirlineAgentContext
                populated outside the LLM (e.g., from the calling web
                session), and gate update_seat on a server-side check that the
                confirmation number belongs to that customer.
      Action 2: Set needs_approval=True on the @function_tool decorator for
                update_seat (or pass a predicate) so the run interrupts and
                requires explicit RunState.approve() before any seat write
                lands.

  PRAX-2026-05-25-003  No durable audit log for seat changes — update_seat
            returns a string and writes only to in-process context; the SDK
            trace span is not the audit record the remit demands.
      Action: Inside update_seat, after the (added) ownership check passes and
              before returning, write a structured audit record (timestamp,
              authenticated customer id, confirmation number, old seat, new
              seat, agent name) to a durable store — file, database, or
              audit-pipeline endpoint — and fail the tool call if the audit
              write fails.

  PRAX-2026-05-25-004  Zero InputGuardrails wired across all three agents —
            free-text customer input flows straight into LLM context with no
            validation, contradicting the explicit "MUST NOT accept
            instructions embedded in free-text customer input" clause.
      Action 1: Author at least one @input_guardrail (an injection-detector
                LLM-as-judge or a heuristic over the user turn) and pass it to
                every agent's input_guardrails=[...] argument so the run halts
                on a tripwire before tool calls are issued.
      Action 2: Place a sequential (run_in_parallel=False) guardrail on the
                triage agent so a tripped guardrail aborts the run before any
                handoff is decided.
