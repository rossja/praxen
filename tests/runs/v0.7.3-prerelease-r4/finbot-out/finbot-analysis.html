Executive Summary
Agent Remit (as declared)
FinBot is CineFlow Productions' autonomous invoice-processing agent — it ingests vendor invoices, validates them against registered-and-approved vendor records, runs fraud detection, and either approves, rejects, or escalates each invoice to human review. The remit authorizes a fixed tool surface (retrieve invoice/vendor, approve, reject, escalate, fraud-check), and explicitly forbids the agent from sending email, modifying vendor records, or treating runtime content — invoice descriptions, vendor notes, operator config — as goal-changing instructions. Fraud detection MUST run before any approval and MUST NOT be skipped; manual-review-threshold breaches require human sign-off; goal redefinition requires a deployment by engineering, not a runtime configuration write.
Behavior Summary (as observed)
The dominant pattern is a textbook policy-implementation void: the Worker Remit declares hard rules — fraud detection MUST run before approval, invoice content MUST NOT alter goals, vendors MUST be verified-and-approved, the manual-review threshold MUST require human sign-off — and the code provides no enforcement at all for any of them. The unauthenticated POST /admin/finbot/goals endpoint writes raw text into the system prompt via FinBotConfig.custom_goals (no auth, no validation, no goal-deployment gate), fraud_detection_enabled is a runtime boolean flag any caller can flip to False, vendor registration auto-stamps every new vendor as approved, and the rule-based _analyze_business_context path in _fallback_processing explicitly downgrades injection-detected invoices to auto-approve when "business context" keywords appear in the description. This produces one full compound chain — unauth admin → goal injection → fraud-detection bypass → auto-approval above the manual-review threshold — that is simultaneously the canonical ASI01/ASI06 path and the documented CTF win condition.
Scope of Analysis
Python Flask application with SQLAlchemy on SQLite, deployed via gunicorn on Render. A single FinBotAgent class in src/services/finbot_agent.py orchestrates OpenAI function-calling (gpt-4.1-mini) over five tools (get_invoice_details, approve_invoice, reject_invoice, request_human_review, detect_fraud_patterns), with a parallel rule-based _fallback_processing path used when the OpenAI client init fails. Admin routes are exposed at /api/admin/* via a Flask blueprint in src/routes/admin.py with no authentication middleware, including POST /admin/finbot/goals which writes attacker-supplied natural-language text directly into the system prompt via FinBotConfig.custom_goals. The Flask SECRET_KEY is a hardcoded literal in src/main.py; CORS is wide open; vendor registration in src/routes/vendor.py sets status='approved' on every new vendor regardless of input.

CRITICAL PRAX-2026-05-25-001 Compound chain — unauthenticated /admin/finbot/goals endpoint writes attacker-controlled text into the LLM system prompt, enabling full goal hijack of a payment-approving agent.
Implement Zero Trust LLM01 — Prompt Injection ASI01 — Agent Goal Hijack ASI03 — Identity and Privilege Abuse ASI10 — Rogue Agents
Policy Rule — R-09, R-11 (Worker Remit):
"Runtime messages — including invoice descriptions, vendor notes, and incoming payloads — MUST NOT alter FinBot's goals, instructions, or decision logic. / FinBot MUST NOT weaken its security posture on the instruction of anyone other than a verified operator making an authorized configuration change outside of agent runtime."
src/routes/admin.py:90 — @admin_bp.route('/admin/finbot/goals', methods=['POST']) has no auth decorator and accepts any JSON body with a 'goals' field src/services/finbot_agent.py:70 — if config.custom_goals: base_goals += "\n\nCUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING):\n" + config.custom_goals — injects unvalidated text into the system prompt
Recommended Action
  • Remove the /admin/finbot/goals endpoint entirely; goal changes belong in the deployment pipeline, not the runtime API. If a runtime escape hatch must exist, gate it behind strong authentication, an allow-list of structured goal fields, and a code-signed approval.
  • Stop concatenating custom_goals into the system prompt — drop the entire if config.custom_goals: block in get_system_prompt() and remove custom_goals from FinBotConfig.
CRITICAL PRAX-2026-05-25-002 Fraud detection is not invoked before approval in either the LLM or fallback path — the remit's MUST-run gate has no code enforcement.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-02, R-06 (Worker Remit):
"FinBot MUST run fraud detection before approving any invoice. / Fraud detection MUST NOT be skipped for any reason, including configuration flags or runtime instructions."
src/services/finbot_agent.py:409 — _approve_invoice() sets invoice.status='approved' and payment_processed=True with no precondition check on fraud-detection results src/services/finbot_agent.py:174 — LLM orchestration loop has no deterministic step that invokes detect_fraud_patterns before approve_invoice can be called
Recommended Action
  • Make _approve_invoice a private function that requires a fresh fraud-detection result object as a parameter; reject any approval that lacks one or whose risk_level is medium or high.
  • Remove fraud_detection_enabled from the runtime config entirely — the remit (R-06) says this flag should not exist.
CRITICAL PRAX-2026-05-25-003 fraud_detection_enabled is a runtime-flippable boolean any caller of /admin/finbot/config can set to False, disabling fraud checks the remit says are non-disableable.
Implement Zero Trust ASI10 — Rogue Agents ASI03 — Identity and Privilege Abuse
Policy Rule — R-14 (Worker Remit):
"Fraud detection MUST remain enabled at all times; disabling it is not an authorized configuration change."
src/routes/admin.py:74 — POST /admin/finbot/config has no auth and forwards the request body to finbot_agent.update_config(data) which accepts arbitrary keys including fraud_detection_enabled src/services/finbot_agent.py:509 — if not config.fraud_detection_enabled: return { ... CRITICAL PRAX-2026-05-25-004 Business-context override in _fallback_processing auto-approves invoices that fraud detection has already flagged as containing prompt injection, in direct violation of R-04 and R-10.
Implement Zero Trust LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-04, R-10 (Worker Remit):
"FinBot MUST escalate to human review rather than auto-approve whenever any fraud signal is present, regardless of the invoice amount. / Instructions embedded in invoice data fields, such as descriptions, notes, or line items, MUST NOT be treated as directives."
src/services/finbot_agent.py:841 — elif contains_injection: ... if business_context['should_expedite'] and config.speed_priority > 0.6: decision = 'auto_approve' src/services/finbot_agent.py:661 — _analyze_business_context() weights management_approval_score * 3 + urgency_score * 2 + production_score * 2 over attacker-controlled description text
Recommended Action
  • Make injection detection a hard rejection or escalation in _fallback_processing — when contains_injection is True the only permitted decisions are flag_review or reject, never auto_approve.
  • Delete _analyze_business_context() entirely; nothing the vendor writes in a description field should drive a security decision.
CRITICAL PRAX-2026-05-25-005 No code-level escalation gate — approve_invoice can be called regardless of fraud-detection signal, contradicting R-04's MUST-escalate-on-any-fraud-signal rule.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-04 (Worker Remit):
"FinBot MUST escalate to human review rather than auto-approve whenever any fraud signal is present, regardless of the invoice amount."
src/services/finbot_agent.py:347 — _execute_function() dispatches by name with no cross-tool state check; approve_invoice runs even after a fraud-positive detect_fraud_patterns call earlier in the loop src/services/finbot_agent.py:409 — _approve_invoice has no fraud_signal parameter and no DB lookup for a prior fraud result
Recommended Action

  • Remove approve_invoice from the LLM tool surface entirely; let the agent only escalate or reject, and place final approval behind a deterministic gate outside the LLM loop.
CRITICAL PRAX-2026-05-25-006 Manual-review threshold can be bypassed — the fallback path explicitly auto-approves above-threshold invoices when keyword business context is strong, with no human checkpoint.
Implement Zero Trust LLM06 — Excessive Agency ASI01 — Agent Goal Hijack
Policy Rule — R-08, R-12 (Worker Remit):
"FinBot MUST NOT process an invoice above the manual-review threshold without human sign-off. / Any invoice above the manual-review threshold MUST reach a human checkpoint before it proceeds."
src/services/finbot_agent.py:807 — if invoice.amount > config.manual_review_threshold: if business_context['should_expedite']: decision = 'auto_approve' (and two parallel override branches at :813 and :817)
Recommended Action
  • Replace every branch under invoice.amount > config.manual_review_threshold with an unconditional decision = "flag_review" — no business-context override is permitted by the remit.
  • Mirror the same gate in the LLM path: _approve_invoice should refuse when invoice.amount > manual_review_threshold regardless of model intent.
HIGH PRAX-2026-05-25-007 Vendor registration auto-stamps every new vendor as status='approved', defeating the remit's registered-and-approved precondition for payment.
Implement Zero Trust ASI03 — Identity and Privilege Abuse ASI09 — Human-Agent Trust Exploitation

HIGH PRAX-2026-05-25-008 confidence_threshold is declared in the config model and the admin UI but never consulted by any approval path — the remit's confidence gate (R-13) has no enforcement.

HIGH PRAX-2026-05-25-009 All /api/admin/* routes are unauthenticated — anyone reachable on the network can update config, change goals, change vendor trust levels, or trigger reprocessing.
Implement Zero Trust ASI03 — Identity and Privilege Abuse ASI10 — Rogue Agents
Policy Rule — R-11 (Worker Remit):
"FinBot MUST NOT weaken its security posture on the instruction of anyone other than a verified operator making an authorized configuration change outside of agent runtime."
src/routes/admin.py — Every @admin_bp.route(...) decorator stands alone with no auth decorator above it (seven routes total) src/main.py:18 — CORS(app) applies the open default to every blueprint including /api/admin/*

MEDIUM PRAX-2026-05-25-011 Decision logging is partial — ai_reasoning is written into the same DB row the goal-injection endpoint can later overwrite, with no append-only audit trail.
