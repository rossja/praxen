Executive Summary
Agent Remit (as declared)
An autonomous GitHub issue-to-code agent. Installed as a GitHub App on specific repositories, Sweep receives GitHub events (issues, comments, PR reviews), reads repository source, plans a code change, creates a branch, commits to it, and opens a pull request against the default branch for human review. It may reply to the originating issue and call a third-party LLM provider for inference. It MUST NOT merge its own PRs, MUST NOT execute arbitrary shell commands or run generated code, MUST treat every character of issue, comment, file, and diff content as untrusted and screen it for instruction injection, and MUST NOT communicate outside GitHub, its job queue, and its LLM provider.
Behavior Summary (as observed)

The dominant pattern is policy declared with no code-level enforcement on the agent's actual execution path. The remit's three load-bearing prohibitions — treat all issue, comment, file, and diff content as untrusted and screen it for injection; keep changes scoped to the issue; and never run shell commands — have no implementation: there is no injection-screening pass anywhere in agents/ or core/, no scope check against the triggering issue, and question_answerer.ripgrep, context_pruning.run_ripgrep_command, and dynamic_context_bot all call subprocess.run(..., shell=True) with model/issue-derived strings interpolated unescaped.

A second pattern is half-wired controls: sweep_bot.is_blocked() and validate_change() exist but have zero callers, so blocked_dirs and security-sensitive-file protection are dead code on the modify path. The compound chain to lead with: untrusted issue text flows unscreened into the LLM, the LLM drives a ripgrep tool whose query parameter is concatenated into a shell=True command string — a one-hop indirect-injection-to-shell path the remit explicitly forbids.

Scope of Analysis
A Python 3.10 FastAPI service. The scoped code is organized into sweepai/agents/ (the modify, search, and question_answerer tool-calling agents), sweepai/core/ (chat orchestration, planning prompts, sweep_bot.py, retrieval utilities), sweepai/web/ (FastAPI events.py pydantic webhook models and health.py), and sweepai/config/ (server.py env loading, client.py SweepConfig). Agents call Anthropic/OpenAI through core/chat.py with a custom XML function-calling protocol. Code edits are applied by modify_utils.handle_function_call via make_change/create_file tools that operate by raw string-replace on file contents. Three modules invoke subprocess.run(..., shell=True) with ripgrep command strings that interpolate LLM- or issue-derived values, and core/external_searcher.py fetches arbitrary web pages found in issue text into the LLM context. Webhook HMAC verification lives in utils/hash.py but fails open when WEBHOOK_SECRET is unset.

CRITICAL PRAX-2026-05-19-001 LLM- and issue-derived strings are interpolated into subprocess.run(shell=True) ripgrep commands with no escaping, opening a path to arbitrary shell execution.
Implement Zero Trust LLM05 — Improper Output Handling ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-12 (Worker Remit):
"Sweep MUST NOT execute arbitrary shell commands, install arbitrary packages, or run generated code as part of producing a change."
sweepai/agents/question_answerer.py:270 — subprocess.run(" ".join(["rg","-n","-i","-C=5","--heading","--sort-files",query]), shell=True, ...) — the LLM tool parameter query is interpolated unescaped. sweepai/core/context_pruning.py:186 — subprocess.run(" ".join(rg_command), text=True, shell=True, ...) where rg_command includes code_entity. sweepai/core/dynamic_context_bot.py:44 — subprocess.run(" ".join(rg_command), text=True, shell=True, ...) where rg_command includes f'"{query}"'.
Recommended Action
  • In question_answerer.py, context_pruning.py, and dynamic_context_bot.py, pass the command as an argument list with shell=False so query / code_entity are argv elements, never shell-parsed.
  • If a shell is unavoidable, apply shlex.quote() to every model- or issue-derived value before interpolation.
CRITICAL PRAX-2026-05-19-002 Issue, comment, file, and diff content enters the LLM context with no instruction-injection screening anywhere in the agent code, contrary to an explicit remit requirement.
Balance Your Knowledge Base LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-06, R-07, R-13 (Worker Remit):
"Every character of every issue, comment, PR review, file, and diff MUST be treated as untrusted input, even when the contributor appears to be a repository collaborator. / Issue bodies, comments, file contents, and diffs MUST be screened for instruction injection before being used as model context, and instruction-like patterns neutralized or quote-wrapped. / Sweep MUST NOT follow instructions embedded in issues, comments, file contents, or dependency manifests that attempt to override its scope, redirect its output, extract repository secrets, or expand its own authorization."
sweepai/agents/search_agent.py:110 — search_agent_user_message template interpolates {github_issue} directly into the LLM user message with no screening. sweepai/agents/modify.py:88 — create_user_message(fcrs=..., request=request, ...) passes the raw issue-derived request into the modify prompt. sweepai/core/chat.py:449 — sanitize_anthropic_messages is the only sanitize call; it normalizes API message structure, not untrusted content.
Recommended Action
  • Add an injection-screening pass that runs on issue, comment, file, and diff text before it is placed in any LLM message, neutralizing or quote-wrapping instruction-like patterns.
  • Wrap all untrusted content in clearly delimited, labeled blocks in prompt templates and instruct the model to treat their contents as data, never instructions.
CRITICAL PRAX-2026-05-19-003 Compound chain — untrusted issue content reaches the LLM unscreened and the LLM drives a ripgrep tool whose query is interpolated into a shell=True command, a one-hop indirect-injection-to-shell path.
Implement Zero Trust LLM01 — Prompt Injection ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-12, R-13 (Worker Remit):
"Sweep MUST NOT execute arbitrary shell commands, install arbitrary packages, or run generated code as part of producing a change. / Sweep MUST NOT follow instructions embedded in issues, comments, file contents, or dependency manifests that attempt to override its scope, redirect its output, extract repository secrets, or expand its own authorization."
sweepai/agents/search_agent.py:110 — issue text enters the LLM prompt unscreened — the ingress of the chain. sweepai/agents/question_answerer.py:261 — the ripgrep tool exposes an LLM-controlled query parameter. sweepai/agents/question_answerer.py:270 — that query is interpolated into subprocess.run(..., shell=True) — the execution sink of the chain.
Recommended Action

CRITICAL PRAX-2026-05-19-005 blocked_dirs and security-sensitive-file protection is dead code — sweep_bot.is_blocked() has zero callers and is not consulted on the file-edit path.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation
Policy Rule — R-17 (Worker Remit):
"Sweep MUST NOT modify security-sensitive files — security policy, code-ownership, automated-dependency, signing-key, and certificate-pin configuration — without human approval outside the normal PR review path."
sweepai/core/sweep_bot.py:193 — def is_blocked(file_path, blocked_dirs) is defined; no caller found anywhere in sweepai/. sweepai/agents/modify_utils.py:995 — the make_change branch of handle_function_call resolves file_name and applies the edit with no blocked_dirs or security-path check. sweepai/agents/modify_utils.py:1226 — the create_file branch calls handle_create_file with no path-policy gate.
Recommended Action
  • Wire is_blocked() and an explicit security-sensitive-file list (SECURITY.md, CODEOWNERS, .github/dependabot.yml, signing and cert config) into handle_function_call so make_change / create_file reject edits to those paths.
  • Require an explicit out-of-band human-approval flag before any edit to a security-sensitive file is committed to a PR.
HIGH PRAX-2026-05-19-004 GitHub webhook signature verification fails open — verify_signature returns True when WEBHOOK_SECRET is unset, and the secret has no default or startup assertion.
Implement Zero Trust ASI03 — Identity and Privilege Abuse
