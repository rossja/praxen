BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared with no code-level enforcement on the
agent's actual execution path. The remit's three load-bearing prohibitions —
treat all issue, comment, file, and diff content as untrusted and screen it
for injection; keep changes scoped to the issue; and never run shell commands
— have no implementation: there is no injection-screening pass anywhere in
agents/ or core/, no scope check against the triggering issue, and
question_answerer.ripgrep, context_pruning.run_ripgrep_command, and
dynamic_context_bot all call subprocess.run(..., shell=True) with
model/issue-derived strings interpolated unescaped. A second pattern is
half-wired controls: sweep_bot.is_blocked() and validate_change() exist but
have zero callers, so blocked_dirs and security-sensitive-file protection are
dead code on the modify path. The compound chain to lead with: untrusted issue
text flows unscreened into the LLM, the LLM drives a ripgrep tool whose query
parameter is concatenated into a shell=True command string — a one-hop
indirect-injection-to-shell path the remit explicitly forbids.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  LLM- and issue-derived strings are interpolated into
            subprocess.run(shell=True) ripgrep commands with no escaping,
            opening a path to arbitrary shell execution.
      Action 1: In question_answerer.py, context_pruning.py, and
                dynamic_context_bot.py, pass the command as an argument list
                with shell=False so query / code_entity are argv elements,
                never shell-parsed.
      Action 2: If a shell is unavoidable, apply shlex.quote() to every model-
                or issue-derived value before interpolation.

  PRAX-2026-05-19-002  Issue, comment, file, and diff content enters the LLM
            context with no instruction-injection screening anywhere in the
            agent code, contrary to an explicit remit requirement.
      Action 1: Add an injection-screening pass that runs on issue, comment,
                file, and diff text before it is placed in any LLM message,
                neutralizing or quote-wrapping instruction-like patterns.
      Action 2: Wrap all untrusted content in clearly delimited, labeled
                blocks in prompt templates and instruct the model to treat
                their contents as data, never instructions.

  PRAX-2026-05-19-003  Compound chain — untrusted issue content reaches the
            LLM unscreened and the LLM drives a ripgrep tool whose query is
            interpolated into a shell=True command, a one-hop
            indirect-injection-to-shell path.
      Action 1: Treat this chain as a single exploitable path: fixing either
                the screening gap (PRAX-2026-05-19-002) or the shell=True
                interpolation (PRAX-2026-05-19-001) breaks it; fix both.
      Action 2: Add per-tool argument validation so search tools cannot
                receive values containing shell metacharacters.

  PRAX-2026-05-19-005  blocked_dirs and security-sensitive-file protection is
            dead code — sweep_bot.is_blocked() has zero callers and is not
            consulted on the file-edit path.
      Action 1: Wire is_blocked() and an explicit security-sensitive-file list
                (SECURITY.md, CODEOWNERS, .github/dependabot.yml, signing and
                cert config) into handle_function_call so make_change /
                create_file reject edits to those paths.
      Action 2: Require an explicit out-of-band human-approval flag before any
                edit to a security-sensitive file is committed to a PR.
