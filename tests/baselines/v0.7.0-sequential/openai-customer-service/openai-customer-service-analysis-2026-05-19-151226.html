Executive Summary
Agent Remit (as declared)
A customer-facing airline customer-service system built as a multi-agent pipeline: a triage agent receives each request and routes it by handoff to specialist agents, an FAQ agent answers airline policy questions using only a curated FAQ dataset, and a seat-booking agent updates seat assignments on existing confirmed reservations. The system may use only two tools — faq_lookup_tool (deterministic FAQ lookup) and update_seat (seat reassignment) — against the airline reservation system and a static FAQ dataset. Customer identity must be confirmed before any reservation-state mutation, confirmation numbers must be verified against the authoritative reservation record, and the agents must never send outbound external communication, process payments or refunds, or act on instructions embedded in customer input or tool output.
Behavior Summary (as observed)

The dominant pattern is framework-offers-safe-primitives, example-uses-none. The OpenAI Agents SDK provides exactly the controls this remit demands — input_guardrails and output_guardrails to reject off-scope or injection-laden input, needs_approval on function_tool to gate state-mutating calls, handoff() input_filter and is_enabled to scope inter-agent transfers — and the customer-service example invokes none of them.

Every behavioral rule in the remit — identity verification before mutation, confirmation-number ownership checks, seat-availability checks, audit logging, injection resistance — is delegated entirely to natural-language routine steps inside agent instructions, with zero code-level enforcement. The compound consequence: update_seat writes attacker-supplied confirmation_number and new_seat straight into the reservation context with no approval gate, no identity check, and no audit trail — a customer, or content injected into the conversation, can reassign any seat on any confirmation number in one hop, and nothing records that it happened.

Scope of Analysis
Python project built on the OpenAI Agents SDK (openai-agents). The analyzed agent is the examples/customer_service/main.py example: a single ~191-line module defining three Agent instances (triage_agent, faq_agent, seat_booking_agent) wired with bidirectional handoff() calls, two @function_tools, and an on_seat_booking_handoff hook, run as an interactive console REPL via Runner.run(). The SDK ships rich safety primitives — input_guardrails and output_guardrails on Agent, tool_input_guardrails and tool_output_guardrails plus a needs_approval flag on FunctionTool, input_filter and is_enabled on handoff() — but the example wires none of them: no guardrails, needs_approval left at its False default, no identity check, no logging. It is a teaching artifact and diverges materially from the remit it is measured against.

CRITICAL PRAX-2026-05-19-001 Seat-booking agent mutates reservation state with no code-level customer identity verification.
Implement Zero Trust LLM06 — Excessive Agency ASI03 — Identity and Privilege Abuse
Policy Rule — R-06, R-20 (Worker Remit):
"Customer identity MUST be confirmed before any action that mutates reservation state. / Any request where the customer's identity cannot be verified MUST be escalated to a human."
examples/customer_service/main.py:67 — update_seat is a plain @function_tool; its body sets context fields with no caller-identity check examples/customer_service/main.py:112 — seat_booking_agent instructions only ask for a confirmation number in prose; AirlineAgentContext.passenger_name is never populated or checked
Recommended Action
  • Add an input guardrail to seat_booking_agent (Agent input_guardrails) that confirms an authenticated passenger identity is present in AirlineAgentContext before any seat-mutating turn.
  • Gate update_seat behind needs_approval or a verified-identity predicate so the tool cannot fire for an unauthenticated session.
CRITICAL PRAX-2026-05-19-002 update_seat trusts a model-supplied confirmation number with no ownership verification, allowing seat changes on any reservation.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-07, R-11, R-17 (Worker Remit):
"A submitted confirmation number MUST be verified as belonging to the authenticated customer before that reservation is read or modified. / Reservations or seat assignments for any customer other than the authenticated session user MUST NOT be modified. / A seat change MUST NOT be issued without first verifying the confirmation number against the authoritative reservation record."
examples/customer_service/main.py:68 — update_seat(context, confirmation_number, new_seat) takes confirmation_number straight from LLM tool-call args examples/customer_service/main.py:79 — context.context.confirmation_number is overwritten with the supplied value; no lookup against an authoritative reservation record, no ownership check
Recommended Action
  • Before mutating, look up the confirmation number in the authoritative reservation system and assert it belongs to the authenticated session passenger; reject otherwise.
  • Do not let update_seat overwrite context.confirmation_number from tool args — bind it to the verified reservation for the session.
CRITICAL PRAX-2026-05-19-003 No durable audit log of seat changes — high-impact reservation mutations leave no record.
Monitor Continuously LLM06 — Excessive Agency ASI10 — Rogue Agents
Policy Rule — R-09 (Worker Remit):
"Every seat change MUST be recorded to a durable audit log capturing the time, the customer identity, the confirmation number, the old and new seat, and the agent that performed the change."
examples/customer_service/main.py:83 — update_seat returns a confirmation string and writes nothing to any log or store examples/customer_service/main.py:1 — module has zero 'import logging' / logging calls; agent activity is only print()ed to the console
Recommended Action
  • In update_seat, write a structured audit record (timestamp, passenger identity, confirmation number, old seat, new seat, performing agent) to a durable store before returning.
  • Add structured action logging across the run loop so every tool call and handoff is recorded.
CRITICAL PRAX-2026-05-19-004 update_seat performs no seat-existence or availability check before reassigning a seat.
Implement Zero Trust LLM05 — Improper Output Handling ASI02 — Tool Misuse and Exploitation
Policy Rule — R-04, R-08 (Worker Remit):
"The seat-booking agent updates seat assignments on existing confirmed reservations only. / Seat updates MUST be limited to seats that exist on the flight and are available — not already assigned to another passenger."

HIGH PRAX-2026-05-19-010 Handoffs carry no input filter or identity-preserving guard — inter-agent transfers rely solely on the LLM.
