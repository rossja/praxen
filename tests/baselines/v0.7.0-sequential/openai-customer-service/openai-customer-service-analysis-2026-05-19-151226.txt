BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is framework-offers-safe-primitives, example-uses-none.
The OpenAI Agents SDK provides exactly the controls this remit demands —
input_guardrails and output_guardrails to reject off-scope or injection-laden
input, needs_approval on function_tool to gate state-mutating calls, handoff()
input_filter and is_enabled to scope inter-agent transfers — and the
customer-service example invokes none of them. Every behavioral rule in the
remit — identity verification before mutation, confirmation-number ownership
checks, seat-availability checks, audit logging, injection resistance — is
delegated entirely to natural-language routine steps inside agent
instructions, with zero code-level enforcement. The compound consequence:
update_seat writes attacker-supplied confirmation_number and new_seat straight
into the reservation context with no approval gate, no identity check, and no
audit trail — a customer, or content injected into the conversation, can
reassign any seat on any confirmation number in one hop, and nothing records
that it happened.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  Seat-booking agent mutates reservation state with no
            code-level customer identity verification.
      Action 1: Add an input guardrail to seat_booking_agent (Agent
                input_guardrails) that confirms an authenticated passenger
                identity is present in AirlineAgentContext before any
                seat-mutating turn.
      Action 2: Gate update_seat behind needs_approval or a verified-identity
                predicate so the tool cannot fire for an unauthenticated
                session.

  PRAX-2026-05-19-002  update_seat trusts a model-supplied confirmation number
            with no ownership verification, allowing seat changes on any
            reservation.
      Action 1: Before mutating, look up the confirmation number in the
                authoritative reservation system and assert it belongs to the
                authenticated session passenger; reject otherwise.
      Action 2: Do not let update_seat overwrite context.confirmation_number
                from tool args — bind it to the verified reservation for the
                session.

  PRAX-2026-05-19-003  No durable audit log of seat changes — high-impact
            reservation mutations leave no record.
      Action 1: In update_seat, write a structured audit record (timestamp,
                passenger identity, confirmation number, old seat, new seat,
                performing agent) to a durable store before returning.
      Action 2: Add structured action logging across the run loop so every
                tool call and handoff is recorded.

  PRAX-2026-05-19-004  update_seat performs no seat-existence or availability
            check before reassigning a seat.
      Action: Query the flight's seat map in update_seat and reject seats that
              do not exist or are already assigned to another passenger.

  PRAX-2026-05-19-005  Raw customer free-text is fed to the agents with no
            injection defense; instruction-override attempts can redirect
            agent behavior.
      Action 1: Add an input guardrail (Agent.input_guardrails) on the triage
                agent that detects role/scope/identity-override attempts and
                trips the tripwire.
      Action 2: Treat customer input as untrusted: never let it select agent
                identity or bypass routine steps.
