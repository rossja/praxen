BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the prompt, zero deterministic
enforcement in code. Every hard prohibition in the remit — no DML, no DDL, no
admin commands, no multi-statement queries, no off-scope tables — depends
entirely on the LLM choosing to obey one sentence of SQL_PREFIX (DO NOT make
any DML statements). QuerySQLDatabaseTool._run() hands the model-generated
string directly to SQLDatabase.run_no_throw(), which executes it verbatim
through SQLAlchemy inside a committing engine.begin() transaction. No SQL
parser, no statement-type allow-list, and no read-only gate exist anywhere in
the scoped code — a single prompt-injection from a table name or a row value
(the exact vector the remit's R-17 forbids) reaches a write-capable execution
path in one hop. The second theme is that the sql_db_query_checker tool, which
the remit treats as a mandatory pre-execution check, is itself an LLM call
with no binding power: it returns advisory text, the agent can skip it, and it
cannot block a query. The library does ship the right framing — a prominent
docstring warning and an explicit instruction to use least-privilege DB roles
and server-side timeouts — so the safe posture is achievable, but it is
delegated wholly to the operator and is unenforced by this code.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  No code-level enforcement of the read-only / no-DML
            prohibition — LLM-generated SQL is executed verbatim through a
            committing transaction.
      Action 1: Add a deterministic statement-type guard before execution: in
                QuerySQLDatabaseTool._run() (or SQLDatabase.run), parse the
                SQL and reject anything that is not a single
                SELECT/CTE/EXPLAIN, rather than relying on the prompt.
      Action 2: Document and require a least-privilege, read-only database
                role as the primary control — the remit's R-21 explicitly
                states the agent's own logic must not be the sole enforcement
                point.

  PRAX-2026-05-19-002  No code-level block on schema-, permission-, or
            admin-modifying statements (DDL / GRANT / maintenance commands).
      Action 1: Extend the statement-type guard recommended in
                PRAX-2026-05-19-001 to reject all DDL, GRANT/REVOKE, and
                dialect-specific admin/maintenance/bulk-load/attach verbs, not
                only DML.
      Action 2: Ensure the operator-supplied database role has no DDL or
                administrative privilege so a defeated guard still fails
                closed.

  PRAX-2026-05-19-003  Multi-statement queries and stored-procedure calls are
            not detected or rejected — a SELECT can carry a hidden write past
            the prompt.
      Action 1: Parse the query and assert exactly one statement before
                execution; reject any input containing more than one statement
                or a CALL/EXEC procedure invocation.
      Action 2: Prefer a SQL parser (e.g. sqlglot) over naive semicolon
                splitting so quoted-literal semicolons are not misclassified.

  PRAX-2026-05-19-007  Compound chain — injected row/table content reaches an
            unguarded committing write path in one hop.
      Action 1: Break the chain at the execution end: a deterministic
                single-SELECT-only guard in QuerySQLDatabaseTool._run() makes
                the injection unable to reach a write even if the model is
                hijacked.
      Action 2: Also label retrieved content as untrusted
                (PRAX-2026-05-19-005) and run under a read-only DB role so the
                chain fails at multiple points.
