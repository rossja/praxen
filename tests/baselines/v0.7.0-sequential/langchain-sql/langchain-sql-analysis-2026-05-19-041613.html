Executive Summary
Agent Remit (as declared)
A natural-language interface to a relational database, intended to answer read-only analytical questions for internal data consumers by translating user questions into SQL, executing them, and returning a natural-language answer. The agent may use exactly four tools — sql_db_list_tables, sql_db_schema, sql_db_query_checker, and sql_db_query — against a single pre-configured database, reading only the tables explicitly exposed to it. Data-modifying (DML) and schema-modifying (DDL) statements are forbidden outright, with no approval path inside the agent; the remit also forbids multi-statement queries, off-scope table access, and following instructions embedded in row content. The agent must double-check every query before execution, limit result sets to a small default, operate within a bounded number of tool-call iterations, and produce no outbound network traffic, file writes, or messaging.
Behavior Summary (as observed)

The dominant pattern is policy declared in the prompt, zero deterministic enforcement in code. Every hard prohibition in the remit — no DML, no DDL, no admin commands, no multi-statement queries, no off-scope tables — depends entirely on the LLM choosing to obey one sentence of SQL_PREFIX (DO NOT make any DML statements). QuerySQLDatabaseTool._run() hands the model-generated string directly to SQLDatabase.run_no_throw(), which executes it verbatim through SQLAlchemy inside a committing engine.begin() transaction. No SQL parser, no statement-type allow-list, and no read-only gate exist anywhere in the scoped code — a single prompt-injection from a table name or a row value (the exact vector the remit's R-17 forbids) reaches a write-capable execution path in one hop.

The second theme is that the sql_db_query_checker tool, which the remit treats as a mandatory pre-execution check, is itself an LLM call with no binding power: it returns advisory text, the agent can skip it, and it cannot block a query. The library does ship the right framing — a prominent docstring warning and an explicit instruction to use least-privilege DB roles and server-side timeouts — so the safe posture is achievable, but it is delegated wholly to the operator and is unenforced by this code.

Scope of Analysis
Python library code (langchain-community) — the SQL agent toolkit, not a deployed application. create_sql_agent() in agent_toolkits/sql/base.py assembles a legacy langchain_classic AgentExecutor from an LLM and a SQLDatabaseToolkit, which exposes four tools defined in tools/sql_database/tool.py. QuerySQLDatabaseTool._run() passes the LLM-generated SQL string straight to SQLDatabase.run_no_throw() in utilities/sql_database.py, which wraps it in SQLAlchemy text() and executes it inside an engine.begin() transaction that commits. There is no SQL parsing, no statement-type allow-list, no read-only enforcement, and no logging anywhere in the scoped trees; the only prohibition on writes is one English sentence in the SQL_PREFIX system prompt. Defaults max_iterations=15 and top_k=10 are present; max_execution_time defaults to None. The module docstring carries an explicit security warning that the agent can run arbitrary SQL and that callers must apply server-side controls.
