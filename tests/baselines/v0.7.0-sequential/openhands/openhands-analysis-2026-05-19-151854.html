Executive Summary
Agent Remit (as declared)
OpenHands is an autonomous software-engineering agent that accepts natural-language tasks and completes them end-to-end — writing code, executing it in a sandboxed runtime, browsing the web, and performing git and issue-tracker operations on repositories the user has connected. It is authorized to run agent-produced code only inside a sandboxed runtime, to operate on files only within the per-session sandbox workspace, and to use MCP tool servers the operator has explicitly configured. The remit forbids host-level code execution, cross-session state leakage, and destructive integration operations outright, and requires explicit user confirmation for merges, force-pushes, branch deletions, dependency commits, CI/CD edits, new MCP servers, and new integration accounts.
Behavior Summary (as observed)

The dominant pattern is a control-plane that is correctly designed for the SAAS deployment and structurally under-defended in the default OSS one. Every V1 router carries the comment that protection is provided by SetAuthCookieMiddleware, but that middleware lives only in enterprise/server/middleware.py and is registered solely by enterprise/saas_server.py; the OSS app_server/app.py registers no authentication middleware at all, and utils/dependencies.get_dependencies() returns an empty dependency list unless the SESSION_API_KEY env var is set. The net effect: in a default local OSS run, the /api/v1/secrets, /api/v1/sandboxes, /api/v1/conversations, and /api/v1/git endpoints answer with no authentication — including the endpoints that read and write stored git-provider tokens.

The same fail-open shape recurs in LocalhostCORSMiddleware, which returns True for every origin when no CORS origins are configured. The remit's strongest behavioral guarantees — code runs only in the sandbox, paths cannot escape the workspace, LLM tool-call arguments are validated and clamped, per-session step and time caps — are real but enforced in the external agent-SDK packages, so they cannot be verified at this layer and are recorded as Enforcement-Not-Possible rather than as gaps.

Scope of Analysis
A Python FastAPI application. The scanned openhands/ tree is the V1 application server (openhands/app_server/) — an orchestration and control-plane layer; the agent loop, tool execution, and runtime were extracted into external pinned packages (openhands-sdk, openhands-agent-server, openhands-tools at 1.22.1). app_server/app.py builds the FastAPI app, mounts a fastmcp MCP server at /mcp, and registers a global v1_router aggregating routers for conversations, sandboxes, secrets, git, events, and webhooks. Sandboxes are real isolation: DockerSandboxService runs each agent-server in its own Docker container with a per-sandbox SESSION_API_KEY, and a ProcessSandboxService host-process backend exists for RUNTIME=local. The OSS app server registers only LocalhostCORSMiddleware, CacheControlMiddleware, and an in-memory RateLimitMiddleware; the per-route auth comment on every V1 router refers to a SetAuthCookieMiddleware that exists only in enterprise/ and is not wired into the OSS app.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Per-sandbox session API keys with state-bound validity

Each Docker sandbox gets a 32-byte random SESSION_API_KEY; validate_session_key rejects keys for any sandbox not in RUNNING state, limiting the window for a leaked key.

openhands/app_server/sandbox/session_auth.py:37

Agentic-core dependencies pinned exactly with committed lockfiles

openhands-sdk, openhands-agent-server, and openhands-tools are all pinned to 1.22.1; poetry.lock and uv.lock are committed and Dependabot runs daily pip updates.

pyproject.toml:61

Scoped, time-limitable JWT secret-retrieval tokens

The /webhooks/secrets endpoint issues a secret only against a JWS token scoped to a specific user and provider type, narrowing blast radius if a token leaks.

openhands/app_server/event_callback/webhook_router.py:374

Secret redaction wired into logging

utils/logger.py imports and applies the SDK redaction helpers (redact_api_key_literals, redact_text_secrets, redact_url_params), and the MCP server is constructed with mask_error_details=True.

openhands/app_server/utils/logger.py:16

RAISE Maturity Posture

Overall maturity assessment across the six categories of the RAISE framework. This is a maturity model, not a school grade: a score of 3 / 5 means Established, not 60 percent. Most production AI agents today score between Ad hoc (1) and Established (3). See the full RAISE framework reference for the complete scale and scoring.

2.15 / 5.0
Weighted Maturity Score · Partial
Partial maturity. OpenHands has a genuine, well-designed isolation model — containerised sandboxes with per-session keys, scoped JWT secret tokens, exactly-pinned agentic-core packages — but the default OSS application server ships without authentication middleware, leaving the V1 API (secrets included) open, and CORS fails open when unconfigured. Adversarial testing of the agent's injection surface is absent, and action-level auditing is delegated to an external component through a best-effort, failure-swallowing webhook. The strongest remit guarantees live in external SDK packages and cannot be verified from this repo.
