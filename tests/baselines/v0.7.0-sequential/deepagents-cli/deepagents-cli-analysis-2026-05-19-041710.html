Executive Summary
Agent Remit (as declared)
The Deep Agents CLI is a local developer command-line tool that scaffolds, bundles, and ships a deepagents agent project to a managed deployment platform (LangSmith Deployments / LangGraph Platform). It exposes exactly three subcommands — init (scaffold a starter project), dev (bundle and serve locally), and deploy (bundle and push to the platform) — and talks to the platform plus its hub and tracing services using the developer's own credentials. Its central promise is that the deployed bundle faithfully and only reflects the developer's reviewed project: bundling only declared sources, never embedding secrets into payloads that should not carry them, and requiring explicit operator confirmation before any deploy that would leave the deployed API open without authentication.
Behavior Summary (as observed)

The dominant pattern is policy-implementation divergence concentrated on the CLI's two explicit safety promises — the unauthenticated-deploy confirmation gate and the "only declared sources" guarantee. The anonymous-auth confirmation prompt in _deploy is gated on frontend.enabled AND auth.provider == "anonymous", yet bundler.py generates the permissive auth.py handler whenever any [auth] provider is set — so a project with provider = "anonymous" but no [frontend] section ships an open API with no warning and no confirmation.

Separately, _auto_wire_issues_board_if_hub provisions a recurring cron-scheduled issues-agent on the platform after a hub-backed deploy — an agent surface the developer never declared — and remote MCP servers carried into the bundle are neither TLS-enforced nor version/integrity-pinned, both of which the remit explicitly requires. The supporting THREAT_MODEL.md is self-admittedly stale, so these gaps are also undocumented.

Scope of Analysis
A Python 3.11+ argparse CLI (main.py) dispatching to a deploy pipeline under deepagents_cli/deploy/ — commands.py (init/dev/deploy handlers), bundler.py (assembles the build directory), config.py (parses/validates deepagents.toml), templates.py (string templates for the generated deploy_graph.py, auth.py, pyproject.toml), and context_hub.py (LangSmith Hub backend). The bundler reads a canonical project layout (deepagents.toml, AGENTS.md, optional mcp.json, .env, skills/, user/, subagents/) and writes a build directory consumed by a shelled-out langgraph CLI. It ships a pre-built React frontend with three auth modes (supabase, clerk, anonymous). Notable: deploy performs a post-deploy _auto_wire_issues_board_if_hub call that provisions a recurring cron issues-agent on the platform for hub-backed deploys; _validate_mcp_for_deploy rejects stdio transports but does not enforce TLS on http/sse URLs; the in-repo THREAT_MODEL.md is self-flagged as stale.

HIGH PRAX-2026-05-19-001 The unauthenticated-API confirmation gate fires only when [frontend].enabled is set, so an anonymous-auth project with no frontend ships an open API silently.
Implement Zero Trust LLM06 — Excessive Agency ASI03 — Identity and Privilege Abuse
Policy Rule — R-02 (Worker Remit):
"A deploy configured to leave the deployed API open to anyone with its URL MUST surface a prominent warning and obtain explicit operator confirmation before it proceeds."
libs/cli/deepagents_cli/deploy/commands.py:287 — is_anonymous requires `config.frontend is not None and config.frontend.enabled` AND `config.auth.provider == "anonymous"`; the warning + confirm block is gated on this compound condition. libs/cli/deepagents_cli/deploy/bundler.py:246 — auth.py is generated whenever `auth_provider is not None`; the anonymous handler returns {"identity": "anonymous"} for every request, so anonymous auth opens the API regardless of whether a frontend is bundled. libs/cli/deepagents_cli/deploy/templates.py:372 — AUTH_BLOCK_ANONYMOUS docstring: overrides LangSmith Cloud's default x-api-key requirement; anyone with the deploy URL can call the API directly via curl.
Recommended Action
In commands.py, key the anonymous warning and confirmation prompt on config.auth is not None and config.auth.provider == "anonymous" alone — drop the frontend.enabled precondition so every anonymous deploy is gated.
HIGH PRAX-2026-05-19-002 deploy auto-provisions a recurring cron-scheduled issues-agent on the platform for hub-backed deploys — an agent surface the developer never declared.
Limit Your Domain ASI10 — Rogue Agents
Policy Rule — R-07 (Worker Remit):
"The CLI MUST NOT silently mutate, generate, or deploy any agent surface the developer did not declare in the project."
libs/cli/deepagents_cli/deploy/commands.py:344 — _deploy calls _auto_wire_issues_board_if_hub(config) after _run_langgraph_deploy, with no operator prompt. libs/cli/deepagents_cli/deploy/commands.py:643 — _upsert_issues_board_config POSTs create_payload with "cron_schedule": "0 */6 * * *" and heavy_model/light_model to the platform issues-agent endpoint — provisioning a background agent that runs every 6 hours. libs/cli/deepagents_cli/deploy/commands.py:702 — _auto_wire_issues_board_if_hub runs unconditionally whenever config.memories.backend == "hub", which is the default backend in MemoriesConfig.
Recommended Action
  • Gate _auto_wire_issues_board_if_hub behind an explicit project declaration or an interactive operator opt-in.
  • Include the recurring issues-agent in the print_bundle_summary output so the operator sees the surface before the deploy proceeds.
HIGH PRAX-2026-05-19-003 _validate_mcp_for_deploy rejects stdio MCP transports but never checks that http/sse server URLs use TLS, so a plaintext http:// MCP server passes validation into the bundle.
Implement Zero Trust LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities All remote connections use TLS
Policy Rule — R-03 (Worker Remit):
"Remote MCP servers carried into the bundle MUST be reached over TLS, and their transport configuration MUST be validated before the bundle is shipped."
libs/cli/deepagents_cli/deploy/config.py:281 — _validate_mcp_for_deploy only flags transport == "stdio"; it does not inspect the url scheme for http vs https. libs/cli/deepagents_cli/deploy/templates.py:410 — Generated _load_mcp_tools builds connections for any transport in ("http", "sse") using cfg["url"] verbatim, with no scheme check.
Recommended Action
In _validate_mcp_for_deploy, add a validation error for any http/sse server whose url does not start with https://, so non-TLS MCP endpoints fail validation before the bundle ships.
HIGH PRAX-2026-05-19-004 Remote MCP servers carried into the deployment bundle are copied verbatim with no version pin or integrity check, so the deployed agent loads whatever the URL serves at runtime.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities Tool definitions are signed and version-pinned
Policy Rule — R-14 (Worker Remit):
"Remote MCP servers declared in the project MUST be pinned to a known-good, integrity-checked version, so the deployed agent does not auto-install an unpinned server afresh."
libs/cli/deepagents_cli/deploy/bundler.py:194 — bundle() copies the project mcp.json straight to _mcp.json (shutil.copy2) with no pinning step. libs/cli/deepagents_cli/deploy/templates.py:410 — _load_mcp_tools calls MultiServerMCPClient(connections).get_tools() at runtime against the live URL; tool definitions are whatever the server returns each call. .mcp.json — The repo's own MCP config (docs-langchain, reference-langchain) is HTTP-transport with bare URLs and no integrity metadata, illustrating the unpinned shape the bundler accepts.
Recommended Action
Require an integrity reference — a pinned server version, an expected tool-manifest hash, or a signature — for each remote MCP server, and reject or warn at bundle time when one is absent.
MEDIUM PRAX-2026-05-19-005 The generated bundle's pyproject.toml pins deepagents==0.5.3 while the CLI itself requires deepagents>=0.6.1, shipping deployed agents on an SDK six minor versions behind the tested baseline.
Manage Your Supply Chain LLM03 — Supply Chain
libs/cli/deepagents_cli/deploy/templates.py:1104 — PYPROJECT_TEMPLATE hard-codes "deepagents==0.5.3" in the generated bundle's dependency list. libs/cli/pyproject.toml:28 — The CLI package itself depends on "deepagents>=0.6.1"; the bundle is built and tested against 0.6.x but deploys 0.5.3.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Strict config schema rejects unknown sections and keys

_parse_config and _parse_subagent_config reject any unknown top-level section or unknown [agent]/[sandbox]/[auth]/[memories]/[frontend] key, preventing silent acceptance of typo'd or stale configuration.

libs/cli/deepagents_cli/deploy/config.py:455

Secrets excluded from the seed payload by design

_build_seed reads only AGENTS.md, skills/, user/, and subagents/ and never folds .env into _seed.json; .env is copied separately as an env file, so secrets do not land in the seeded-memory payload.

libs/cli/deepagents_cli/deploy/bundler.py:202

Scaffolding overwrite guard

_init_project refuses to write into an existing project directory unless the operator passes --force, satisfying the explicit-opt-in requirement for overwriting project files.

libs/cli/deepagents_cli/deploy/commands.py:197

Committed pinned lockfile plus supply-chain CI

uv.lock is committed and pinned, and .github/ carries dependabot.yml, check_lockfiles.yml, and check_extras_sync.yml — the dependency baseline is version-controlled and continuously checked.

libs/cli/uv.lock:1

Substantial deploy-pipeline test suite

Unit and integration tests cover the bundler, config validation, context hub, and frontend bundle, including the anonymous-auth abort-on-no and abort-on-EOF paths.
