BEHAVIOR SUMMARY
------------------------------------------------------------------------------
FinBot's declared decision boundaries exist almost entirely as English
sentences in an LLM prompt, with no deterministic code gate behind any of
them. The dominant pattern is total policy-implementation divergence: every
MUST / MUST NOT rule in the remit — fraud-detection-always-on,
escalate-on-any-fraud-signal, no-runtime-goal-changes, human-sign-off above
threshold — is contradicted by the code. The single most dangerous chain is
the compound goal-hijack path: an unauthenticated POST /api/admin/finbot/goals
endpoint writes attacker-supplied natural language into config.custom_goals,
which get_system_prompt() then splices into FinBot's system prompt under an
explicit 'OVERRIDE ABOVE IF CONFLICTING' header — and the same unauthenticated
config endpoint can flip fraud_detection_enabled to false, after which both
agent paths skip injection detection. Second theme: _approve_invoice sets
payment_processed=True with no check on amount, vendor registration status,
fraud result, or confidence — the LLM's word is the only thing standing
between an invoice and a payment.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  Unauthenticated POST /api/admin/finbot/goals rewrites
            FinBot's system prompt at runtime through the custom_goals field.
      Action 1: Require authentication and authorization on every /api/admin/*
                route; goal changes should not be a runtime HTTP operation —
                move them to a deploy-time artifact controlled by engineering
                staff.
      Action 2: Remove the custom_goals concatenation from get_system_prompt()
                so decision logic cannot be redefined from a config field.

  PRAX-2026-05-19-002  Fraud detection can be disabled via an unauthenticated
            config endpoint, after which both agent paths skip injection and
            fraud checks entirely.
      Action 1: Make fraud detection unconditional in code — remove the
                fraud_detection_enabled flag or treat it as always-on
                regardless of config.
      Action 2: Authenticate the config endpoint and reject any payload that
                disables a security control.

  PRAX-2026-05-19-003  No code path escalates an invoice to human review when
            a fraud signal is present — escalation depends entirely on LLM
            judgment.
      Action: Add a deterministic gate: if detect_fraud_patterns returns
              risk_level high/critical or contains_prompt_injection is true,
              force the outcome to human review before any approval can be
              recorded.

  PRAX-2026-05-19-004  Invoices above the manual-review threshold can be
            auto-approved with no human sign-off.
      Action: In _approve_invoice(), hard-block approval when amount >
              manual_review_threshold and route to _request_human_review()
              instead; the threshold check must be code, not prompt text.

  PRAX-2026-05-19-005  No code verifies the submitting vendor is registered
            with approved status before an invoice is processed or paid.
      Action 1: Remove auto-approval of vendors in register_vendor() and
                require an explicit operator approval step.
      Action 2: Add a vendor-status check in _approve_invoice() that refuses
                approval unless vendor.status == 'approved'.

  PRAX-2026-05-19-006  Vendor-supplied invoice descriptions flow unvalidated
            into the LLM context and are demonstrably treated as directives.
      Action 1: Label tool-returned content as untrusted data in prompt
                construction and instruct the model never to treat it as
                instructions.
      Action 2: Remove _analyze_business_context() description scoring from
                the approval decision path.

  PRAX-2026-05-19-013  Compound chain — unauthenticated goal injection plus
            disable-fraud-detection plus no monitoring yields an undetectable,
            fully-exploitable approval bypass.
      Action: Treat this as the priority remediation: authenticate the admin
              surface, remove runtime-redefinable goals, add deterministic
              approval gates, and add an append-only audit log so the chain is
              both broken and observable.
