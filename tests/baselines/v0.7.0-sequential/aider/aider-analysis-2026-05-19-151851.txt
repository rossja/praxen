BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the remit with no code-level
enforcement: every safety boundary the remit names — repository containment,
shell-command confirmation, treating file content as untrusted, secret
redaction — is either absent from the code or implemented as a soft,
prompt-level or default-on, easily-disabled control. The most acute instance
is a confirmation-bypass surface: confirm_ask honors a global --yes-always
flag and a per-question "don't ask again" memory, and several high-impact
paths skip confirmation entirely — cmd_run and the ! alias execute shell
commands with no prompt at all, and auto_lint (default True) runs the
configured lint command as a subprocess after every LLM edit. Compounding
this, /read-only and /add accept arbitrary absolute filesystem paths and pull
their contents straight into LLM context with no path-resolution check, so
external file content and injected instructions reach the model on a one-hop
path to shell execution. There is no injection detection, no secret redaction,
and no structured action log anywhere in the scoped code.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  /read-only and /add accept absolute filesystem paths
            with no repo-root containment, letting any file on the machine be
            read into LLM context.
      Action 1: In commands.py, before adding any path in cmd_read_only /
                _add_read_only_file and cmd_add, resolve it with
                Path.resolve() and reject (do not merely confirm) any path
                whose resolved form is not under coder.root.
      Action 2: Apply the same resolved-root containment check inside
                io.read_text and io.write_text so the boundary holds
                regardless of which call path reaches it.

  PRAX-2026-05-19-002  /run and the ! alias execute arbitrary shell commands
            immediately with no confirmation prompt, defeating the mandatory
            shell-command approval gate.
      Action 1: Add an explicit confirm_ask(..., explicit_yes_required=True)
                gate at the top of cmd_run in commands.py before run_cmd is
                called, showing the verbatim command.
      Action 2: Make --yes-always not apply to shell-execution and
                file-containment confirmations, or exclude
                explicit_yes_required prompts from the --yes-always
                short-circuit in io.confirm_ask.

  PRAX-2026-05-19-003  No injection detection or neutralization exists
            anywhere in scope; untrusted repo content, comments, web pages,
            and command output flow into LLM context verbatim.
      Action 1: Add a content-origin labeling step in base_coder.py prompt
                construction that wraps repo file content, repo-map summaries,
                /web output, and /run output in explicit untrusted-content
                delimiters the system prompt instructs the model never to
                treat as instructions.
      Action 2: Add a screening pass over comments, docstrings, and commit
                messages that flags imperative instruction-like language
                before that content reaches format_messages.

  PRAX-2026-05-19-004  No secret redaction exists; file contents, diffs, and
            LLM-generated commit messages are sent to the provider and written
            to git with no scan for secret-like strings.
      Action 1: Add a secret-pattern scan (high-entropy strings,
                sk-/Bearer/private-key headers) in
                base_coder.get_files_content and repo.get_diffs, redacting
                matches before content reaches the LLM or a commit message.
      Action 2: Refuse or warn-and-confirm in repo.commit when the staged diff
                contains secret-like strings.
