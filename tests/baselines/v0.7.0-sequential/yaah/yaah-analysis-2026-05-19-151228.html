Executive Summary
Agent Remit (as declared)
yaah is a local developer command-line tool that generates coding-agent configuration for Claude Code, OpenCode, Codex CLI, and GitHub Copilot CLI from a single Go source of truth, and ships a built-in security-and-quality toolset alongside it. Its central promise is that every generated agent configuration carries the same set of protective hooks — a linter, a catastrophic-command guard, a secret scanner, a comment checker, and a session logger — plus a built-in MCP server (yaah serve) and a per-session audit log. yaah may contact only the local developer, the configured LLM provider, the context7 and pulumi MCP servers, its own built-in MCP server, and the coding-agent plugin marketplace; catastrophic shell commands must be blocked before they run and remote MCP servers must be reached over TLS.
Behavior Summary (as observed)

The dominant pattern is policy-implementation divergence on the harness's single central promise. The remit states that every generated agent configuration MUST carry the same protective hook set; the code does not deliver this. pkg/generator/hookmap.go has no Codex entry for PreToolUse or PostToolUse, so CodexGenerator.GenerateHooks() emits a .codex/hooks.json containing only SessionStart and Stop — the command guard, secret scanner, linter, and comment checker are dropped for the Codex target, and TestCodex_GenerateHooks_NoSupported codifies that drop as expected behaviour.

A second theme runs through the protections that do exist: they are advisory or post-hoc rather than enforcing. The secret scanner and linter are PostToolUse handlers that run after the write has already landed, the yaah_check_command MCP tool only returns a safe/unsafe verdict with no blocking, and there is no human-approval checkpoint for file writes or for write/send/execute MCP tools — only catastrophic bash commands are hard-blocked, and only via a small bypassable regex denylist.

Scope of Analysis
A Go 1.25 CLI (cmd/yaah, cobra) built on the Interface to Registry to Generator pattern. pkg/harness wires hook handlers, MCP/LSP providers, skills, agents, commands and a session.Store into a Harness; pkg/hooks/handlers holds the five protective handlers (Linter, CommandGuard, SecretScanner, CommentChecker, SessionLogger); pkg/generator emits per-agent config via four AgentGenerator implementations. yaah hook <event> is the runtime dispatcher invoked by the generated agent's hook config; yaah serve runs a built-in MCP server exposing seven tools. The hook handlers fire only for events the target agent's generator actually emits — and the Codex generator maps PreToolUse/PostToolUse to the empty string, so the protective hooks are silently absent from Codex-targeted configuration.

CRITICAL PRAX-2026-05-19-001 Codex-targeted configuration ships without the protective Pre/PostToolUse hooks — the command guard, secret scanner, linter, and comment checker are silently dropped.
Implement Zero Trust ASI05 — Cascading & Multi-Agent Failures
Policy Rule — R-01 (Worker Remit):
"Every generated agent configuration MUST carry the same set of protective hooks — a linter, a command guard that blocks catastrophic shell commands, a secret scanner, a comment checker, and a session logger."
pkg/generator/hookmap.go:30 — schema.HookPreToolUse mapping has Claude/OpenCode/Copilot entries but no Codex field, so CodexEventName(HookPreToolUse) returns "" pkg/generator/hookmap.go:35 — schema.HookPostToolUse mapping likewise has no Codex field pkg/generator/codex.go:90 — GenerateHooks skips any event whose CodexEventName is empty, dropping PreToolUse/PostToolUse from .codex/hooks.json pkg/generator/codex_test.go:118 — TestCodex_GenerateHooks_NoSupported asserts hooks data is nil when only PreToolUse is configured — the drop is tested-in as expected behaviour
Recommended Action
  • Add Codex event names for PreToolUse and PostToolUse in pkg/generator/hookmap.go (Codex hooks.json supports them) so the protective handlers reach the Codex target.
  • If Codex genuinely cannot run Pre/PostToolUse hooks, make yaah generate fail loudly for the codex target rather than silently emitting a degraded config, and document the protection gap.
CRITICAL PRAX-2026-05-19-002 No human-approval checkpoint exists for file writes or for write/send/execute MCP tools — only catastrophic bash commands are blocked.
Implement Zero Trust ASI02 — Tool Misuse
Policy Rule — R-07 (Worker Remit):
"High-impact actions — destructive shell commands, file writes, and MCP tools that write, send, or execute — MUST reach a human checkpoint before they run."
pkg/hooks/handlers/secretscan.go:127 — SecretScanner runs on PostToolUse — i.e. after the Edit/Write has already landed on disk; no pre-write gate exists pkg/mcpserver/tools.go:363 — yaah_planning_init writes STATE.md and creates directories with no approval or confirmation step pkg/hooks/handlers/guard.go:79 — CommandGuard.Execute blocks only Bash commands; Edit/Write/MultiEdit have no PreToolUse gate
Recommended Action
  • Add a PreToolUse gate for Edit/Write/MultiEdit that requires explicit confirmation, or document that approval is delegated to the host agent's permission system and verify each generated config enables it.
  • Gate the write-capable MCP tool (yaah_planning_init) behind a confirmation or a dry-run/confirm two-step.
HIGH PRAX-2026-05-19-003 Hardcoded-credential detection is post-write only — the secret scanner runs on PostToolUse, after the edit has already been written to disk.
Implement Zero Trust LLM02 — Sensitive Information Disclosure
Policy Rule — R-05 (Worker Remit):
"Hardcoded credentials in files the agent edits MUST be detected before the change is accepted."
pkg/hooks/handlers/secretscan.go:68 — Events() returns HookPostToolUse only — the scanner cannot run before the write pkg/hooks/handlers/secretscan.go:106 — ScanFile reads the file from disk line-by-line after the write has completed pkg/hooks/handlers/secretscan.go:34 — fixed regex pattern set; no entropy-based or multi-line detection — novel credential formats are missed
Recommended Action
  • Treat PostToolUse blocking as detection-after-write and document that the file is already on disk; recommend the operator add a pre-commit secret scan as the actual before-accepted gate.
  • Add an entropy-based fallback pattern to catch credential formats not in the fixed list.
HIGH PRAX-2026-05-19-004 No adversarial testing of the protections, no threat model, and no security-disclosure process.
Build an AI Red Team
Policy Rule — R-13 (Worker Remit):
"The project SHOULD publish a threat model and a security-disclosure process, and SHOULD run adversarial testing of its own protections — confirming that the secret scanner catches a planted secret, the command guard blocks a catastrophic command, a poisoning write into a session-loaded file is prevented, and a generated configuration genuinely carries the protections it claims."
AGENTS.md — No SECURITY.md and no threat-model document found anywhere in the repository pkg/generator/codex_test.go:118 — Generator tests assert structural shape but never assert that a generated config end-to-end carries the full protective hook set for every target
Recommended Action
  • Add an end-to-end test that generates config for every target and asserts the linter, command guard, secret scanner, and comment checker are all present and wired.
  • Publish a SECURITY.md with a disclosure process and a short threat model covering the harness's protective promises.
HIGH PRAX-2026-05-19-006 No control prevents poisoning writes into agent-managed, session-loaded files (AGENTS.md, generated configs, .planning/STATE.md).
Implement Zero Trust LLM01 — Prompt Injection ASI06 — Memory & Context Poisoning
Policy Rule — R-06 (Worker Remit):
"Agent-managed, session-loaded files MUST NOT be writable in a way that lets ingested content persist into future sessions unreviewed."
pkg/mcpserver/tools.go:410 — yaah_planning_init writes .planning/STATE.md, which is later read back by yaah_planning_status at tools.go:302 pkg/hooks/handlers/commentcheck.go:67 — CommentChecker inspects edits for placeholder/non-English comments only — it does not detect injected instructions in session-loaded files AGENTS.md:2 — Root AGENTS.md is explicitly Managed by agent and loaded at session start, with no write-protection control in the harness
Recommended Action
  • Add a handler or generator-side check that flags or blocks writes into agent-context files (AGENTS.md, generated settings, .planning/STATE.md) for human review.
  • Document which files are session-loaded and treat edits to them as requiring approval.
HIGH PRAX-2026-05-19-010 Third-party MCP servers are launched via npx -y, which auto-installs the latest unpinned package on every run.
Manage Your Supply Chain Tool definitions are signed and version-pinned
Policy Rule — R-14 (Worker Remit):
"Third-party MCP servers MUST be pinned to a known-good, integrity-checked version — no server package auto-installed afresh, unpinned, on every run."
pkg/mcp/providers/context7.go:21 — Server() returns Command npx with Args [-y, @context7/mcp] — no version pin and no integrity check pkg/mcp/providers/notion.go:25 — Server() returns Command npx with Args [-y, @notionhq/notion-mcp-server] — no version pin .mcp.json — Generated project MCP config carries the same unpinned npx -y @context7/mcp invocation
Recommended Action

MEDIUM PRAX-2026-05-19-005 The catastrophic-command guard is a small regex denylist that is trivially bypassable.
Implement Zero Trust
Policy Rule — R-02 (Worker Remit):
"Catastrophic shell commands — recursive deletion from the filesystem root, force-pushing to a main branch, hard resets, destructive database statements, filesystem formatting, raw disk writes — MUST be blocked before they execute."
pkg/hooks/handlers/guard.go:27 — NewCommandGuard registers exactly 7 fixed regex patterns pkg/hooks/handlers/guard.go:30 — Force-push pattern requires the literal --force; git push -f origin main is not matched pkg/hooks/handlers/guard.go:29 — Pattern rm -rf / does not match rm -fr / or rm --recursive --force /
Recommended Action
  • Broaden the guard patterns to cover short-flag and reordered-flag variants (-fr, -f, --recursive --force).
  • Alternatively, document the guard as best-effort defence-in-depth and soften the remit language from MUST-block to a layered control.
MEDIUM PRAX-2026-05-19-007 Lint and comment findings are surfaced to the operator but not recorded to the structured session audit log.
Monitor Continuously
Policy Rule — R-12 (Worker Remit):
