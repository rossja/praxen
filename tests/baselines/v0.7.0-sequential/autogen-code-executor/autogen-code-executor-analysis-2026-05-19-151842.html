Agent Remit (as declared)
The AutoGen Code Execution Subsystem is the execution tier of a generator/executor agent pair: it receives code blocks produced by an LLM agent and runs them, returning stdout, stderr, and exit status for the agent's next reasoning step. It ships five executor kinds — a LocalCommandLineCodeExecutor (host OS), a DockerCommandLineCodeExecutor (the standard production path), a DockerJupyterCodeExecutor, a JupyterCodeExecutor, and an ACADynamicSessionsCodeExecutor (Azure managed sandbox) — over a shared CodeExecutor abstraction. Its only authorized inputs are code from the paired LLM agent or operator application logic; all file I/O must stay inside a configured work directory, every execution must be timeout-bounded and audit-logged, and use of the local host executor in production, host-volume mounts, network egress, and resource-ceiling increases all require human approval.
Behavior Summary (as observed)

The dominant pattern is a sandbox whose isolation is the operator's responsibility to configure, not a property the code enforces — and the defaults lean unsafe. create_default_code_executor() silently downgrades from Docker to the host-level LocalCommandLineCodeExecutor with only a warnings.warn, and the local executor copies the full parent environment (os.environ.copy()) into every child process, so a single execution sees the caller's credentials — the remit's host-isolation and local-executor-approval clauses have no code-level enforcement.

The Docker path never sets network_mode, CPU, or memory limits and merges caller-supplied extra_volumes with no validation, so the remit's approval-required gates for mounts, egress, and resource ceilings are absent. Compounding all of it, no executor writes the per-execution audit log the remit mandates, and the LocalCommandLineCodeExecutor docstring advertises regex sanitization against a list of dangerous commands that does not exist anywhere in the implementation — a control that is documented, relied upon, and absent.

Scope of Analysis
A Python library under autogen_core/code_executor/ (the abstract CodeExecutor base, CodeBlock/CodeResult dataclasses, and the FunctionWithRequirements injection machinery) and autogen_ext/code_executors/ (five concrete executors plus shared _common.py). Each executor writes each code block to a file in the work directory and runs it: Local via asyncio.create_subprocess_exec on the host, Docker via container.exec_run, the two Jupyter variants via an nbclient/Jupyter-gateway kernel, and Azure via HTTP POST to a Dynamic Sessions endpoint. create_default_code_executor() picks Docker if available and silently falls back to Local otherwise. There is no audit-logging layer, no input inspection of the executed code, and no resource-limit or network-isolation configuration on the Docker path — material gaps against the remit's must-always and approval-required clauses.

CRITICAL PRAX-2026-05-19-001 create_default_code_executor silently falls back from Docker to the host-level LocalCommandLineCodeExecutor, with only a runtime warning and no approval gate.
Implement Zero Trust LLM06 — Excessive Agency ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-07, R-13 (Worker Remit):
"Code MUST NOT be executed with host-level privileges when a less-privileged option achieves the same task. / Use of the local host executor in production MUST require human approval — it runs code directly on the host OS without containerization; default production deployments should use a containerized executor, and local execution is acceptable only when the host is an ephemeral, isolated, operator-approved sandbox."
python/packages/autogen-ext/src/autogen_ext/code_executors/__init__.py:58 — _is_docker_available() failing, or DockerCommandLineCodeExecutor() raising, leads to a warnings.warn and a return LocalCommandLineCodeExecutor(...) with no operator confirmation. python/packages/autogen-ext/src/autogen_ext/code_executors/__init__.py:64 — except Exception: pass swallows any Docker init error and proceeds straight to the local fallback.
Recommended Action
  • Make the Docker-to-Local fallback in create_default_code_executor() opt-in via an explicit allow_local_fallback=False default, raising instead of silently downgrading.
  • Require an explicit operator-supplied flag (or environment gate) before any code path constructs LocalCommandLineCodeExecutor in a non-interactive deployment.
CRITICAL PRAX-2026-05-19-002 LocalCommandLineCodeExecutor runs LLM-generated code with the full parent process environment, exposing the caller's credentials to executed code.
Implement Zero Trust LLM02 — Sensitive Information Disclosure ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-04 (Worker Remit):
"The execution environment MUST be isolated from the host caller — a failing or malicious execution MUST NOT be able to read or modify the parent process's environment, credentials, or state."
python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:397 — env = os.environ.copy() is passed to create_subprocess_exec, so every executed code block inherits API keys, tokens, and host env vars present in the parent process. python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:426 — The child process is spawned with cwd=self.work_dir but the inherited env is never filtered or scrubbed.
Recommended Action
  • Pass an explicitly constructed minimal environment (e.g. only PATH, HOME, LANG) to create_subprocess_exec instead of os.environ.copy().
  • Document that the local executor provides no credential isolation and must only run on hosts with no sensitive environment variables.

HIGH PRAX-2026-05-19-005 No executor writes the per-execution audit log the remit requires; only ad hoc operational logging exists.
Monitor Continuously LLM06 — Excessive Agency ASI10 — Rogue Agents
Policy Rule — R-06 (Worker Remit):
"Each execution MUST be recorded to an audit log capturing timestamp, executor kind, language, source agent, work directory, timeout, exit status, and a digest — not the body — of the executed code."
python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:341 — _execute_code_dont_check_setup runs code and returns a result with no audit-log emission; the only logging calls are logging.error on temp-file deletion failure. python/packages/autogen-ext/src/autogen_ext/code_executors/docker/_docker_code_executor.py:327 — The Docker execution path emits only logging.debug operational lines, none capturing executor kind, language, exit status, or a code digest.

HIGH PRAX-2026-05-19-006 Work-directory confinement is enforced only for the # filename: comment and does not defeat symlinks or absolute paths written inside the executed code.
Implement Zero Trust LLM05 — Improper Output Handling ASI02 — Tool Misuse and Exploitation
Policy Rule — R-01, R-09 (Worker Remit):
"All file read and write operations MUST be confined to a configured work directory, and any path resolving outside it MUST be rejected. / Work-directory confinement MUST NOT be bypassed under any condition, including symlinks, parent-directory traversal, absolute paths, or runtime-supplied volume mount overrides."
python/packages/autogen-ext/src/autogen_ext/code_executors/_common.py:96 — get_file_name_from_content only inspects the first line for # filename:; path.relative_to rejects out-of-workspace paths but the check covers only this one declared-filename path. python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:391 — When no # filename: is present the code is written to a hashed name in work_dir, but the executed code itself can open() any absolute path on the host with no interposition.
Recommended Action
  • Document explicitly that work-directory confinement applies only to the executor's own file placement, not to file operations performed by the executed code, and that the Local and Jupyter executors provide no filesystem sandbox.
  • For the container executors rely on the container boundary; for Local and Jupyter recommend OS-level sandboxing, since the executor cannot confine the executed code's file I/O.

HIGH PRAX-2026-05-19-009 JupyterCodeExecutor's timeout shields the running cell, so an over-time cell is abandoned rather than terminated.
Implement Zero Trust LLM10 — Unbounded Consumption ASI08 — Cascading Failures
Policy Rule — R-02 (Worker Remit):
"Every code execution MUST be subject to a configured wall-clock timeout, and processes that exceed it MUST be terminated."
python/packages/autogen-ext/src/autogen_ext/code_executors/jupyter/_jupyter_code_executor.py:207 — await asyncio.wait_for(asyncio.shield(execute_task), timeout=self._timeout) raises TimeoutError to the caller but asyncio.shield prevents the underlying cell task from being cancelled, so the code keeps running in the kernel. python/packages/autogen-ext/src/autogen_ext/code_executors/jupyter/_jupyter_code_executor.py:268 — restart() is the only way to reclaim the kernel; nothing in the timeout path interrupts or kills the still-executing cell.
Recommended Action
  • On timeout, interrupt the Jupyter kernel (kernel interrupt request) or restart it so the over-time cell is actually terminated, not merely detached.
  • Drop the asyncio.shield so the execution task is cancelled when the wall-clock timeout fires.
HIGH PRAX-2026-05-19-010 DockerCommandLineCodeExecutor sets no CPU, memory, or pids limits on the container, leaving execution resource-unbounded.
Implement Zero Trust LLM10 — Unbounded Consumption ASI08 — Cascading Failures
Policy Rule — R-16 (Worker Remit):
"Any change to the executor's resource limits — CPU, memory, timeout — that raises the ceiling MUST require human approval."

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Language allow-list on every executor

Each executor restricts execution to an explicit SUPPORTED_LANGUAGES allow-list (Azure to Python only) and returns a clear error for anything else, rather than attempting to run unknown languages.

python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:130

Wall-clock timeout enforced on the command-line and Azure executors

The Local executor uses asyncio.wait_for and terminates the process on timeout, and the Docker executor wraps execution in the timeout command, giving real (non-prompt) time bounds.

python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:441
