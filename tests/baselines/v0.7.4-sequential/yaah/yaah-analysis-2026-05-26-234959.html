Executive Summary
Agent Remit (as declared)
yaah is a local developer CLI that generates coding-agent configuration for Claude Code, OpenCode, Codex CLI, and GitHub Copilot CLI from a single Go source of truth, and ships a built-in security-and-quality toolset alongside it. Its central declared promise is that every generated agent configuration carries the same set of protective hooks — linter, command guard for catastrophic shell commands, secret scanner, comment checker, and session logger — together with a built-in MCP server and a durable, structured, timestamped audit log. Authorized counterparties are the local developer, the configured agent's LLM provider, the `context7` and `pulumi` MCP servers, the built-in yaah MCP server, and the coding-agent plugin marketplace; third-party MCP servers MUST be pinned to a known-good version, remote MCP MUST use TLS, MCP tool descriptions MUST NOT carry instruction-like language, and hardcoded credentials MUST be detected before a change lands.
Behavior Summary (as observed)
The single most important pattern is **uniform protections promised, non-uniform protections shipped**: `pkg/generator/hookmap.go` declares no `PreToolUse` or `PostToolUse` mapping for the Codex CLI target, and `pkg/generator/codex.go:GenerateHooks` skips any event whose Codex name is empty — so a developer who runs `yaah generate --agent codex` receives a `.codex/hooks.json` wired only for `SessionStart`/`Stop`, with the command guard, secret scanner, linter, and comment checker silently absent from that agent's runtime path. The same fragility appears one layer deeper in `npx -y @context7/mcp` in `pkg/mcp/providers/context7.go` (and `@notionhq/notion-mcp-server` in `notion.go`): the Worker Remit says third-party MCP servers MUST be pinned to a known-good, integrity-checked version, but the configured invocation auto-installs the latest publish from npm on every run with no integrity check. A third structural gap — the remit's "durable, structured, timestamped record of tool calls, blocked actions, and file modifications" — is half-met: `pkg/harness/harness.go:HandleHookEvent` does record tool calls and blocks into a per-session JSON store, but `pkg/hooks/handlers/sessionlog.go` only logs `SessionStart`/`SessionEnd`/`Stop` events to a plaintext line in `.claude/logs/sessions.log`, so on the Codex target (where the JSON store is never populated for tool calls) the durable audit trail required by the remit does not exist.
Scope of Analysis
yaah is a Go 1.25 cobra CLI organized as `Interface → Registry → Generator`: registries collect hooks, MCP/LSP providers, skills, agents, commands, and plugins, and per-agent generators (`pkg/generator/claude.go`, `opencode.go`, `codex.go`, `copilot.go`) emit each agent's native config. Five hook handlers live in `pkg/hooks/handlers/` (`linter.go`, `guard.go`, `secretscan.go`, `commentcheck.go`, `sessionlog.go`); the command guard is a small regex denylist (`rm -rf /`, `git push --force`, hard reset, DROP, `mkfs`, `dd if=…of=/dev/`) and the secret scanner is a regex pattern set scoped to `Edit|Write|MultiEdit`. The built-in MCP server (`pkg/mcpserver/server.go`, `tools.go`) exposes `yaah_scan_secrets`, `yaah_lint`, `yaah_check_command`, `yaah_doctor`, `yaah_session_info`, `yaah_planning_status`, `yaah_planning_init` over stdio. The `.mcp.json` Praxen sees at the workspace root declares three servers — `context7` (`npx -y @context7/mcp`, unpinned), `pulumi` (`https://mcp.ai.pulumi.com/mcp`, TLS), and `yaah` (stdio `yaah serve`); the cross-agent event mapping in `pkg/generator/hookmap.go` is the determining surface for which protections actually reach each agent.

CRITICAL PRAX-2026-05-26-001 Codex-target generated configuration ships without command guard, secret scanner, linter, or comment checker because no PreToolUse / PostToolUse mapping exists.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation LLM06 — Excessive Agency
Policy Rule — R-01, R-05, R-07 (Worker Remit):
"Every generated agent configuration MUST carry the same set of protective hooks — a linter, a command guard that blocks catastrophic shell commands, a secret scanner, a comment checker, and a session logger. / Hardcoded credentials in files the agent edits MUST be detected before the change is accepted. / High-impact actions — destructive shell commands, file writes, and MCP tools that write, send, or execute — MUST reach a human checkpoint before they run."
pkg/generator/hookmap.go:15 — hookMappings table has no Codex entry for HookPreToolUse or HookPostToolUse — only SessionStart, Stop, notify are mapped for Codex. pkg/generator/codex.go:85 — GenerateHooks loops events and skips any whose CodexEventName is empty ("if codexEvent == \"\" || codexEvent == \"notify\" { continue }"), so PreToolUse/PostToolUse handlers are never reached on Codex.
Recommended Action
  • In `pkg/generator/hookmap.go`, add the Codex mappings for `HookPreToolUse` and `HookPostToolUse` (Codex CLI does expose these hook events) so the command guard and secret scanner reach the Codex runtime path.
  • Add a generator-level invariant test that asserts each target's generated hook config carries `command-guard` on the equivalent of `PreToolUse` and `secret-scanner` on `PostToolUse`, failing the build if any target drops a declared protective hook.
CRITICAL PRAX-2026-05-26-002 Third-party MCP servers (context7, notion) are launched with `npx -y` and no version pin, contradicting the remit's pinning rule.
Manage Your Supply Chain Tool definitions are signed and version-pinned ASI04 — Agentic Supply Chain Vulnerabilities LLM03 — Supply Chain
Policy Rule — R-11 (Worker Remit):
"Third-party MCP servers MUST be pinned to a known-good, integrity-checked version — no server package auto-installed afresh, unpinned, on every run."
pkg/mcp/providers/context7.go:17 — Server() returns Command "npx" with Args []string{"-y", "@context7/mcp"} — no @version suffix, no integrity manifest, auto-installs latest on every run. pkg/mcp/providers/notion.go:21 — Server() returns Command "npx" with Args []string{"-y", "@notionhq/notion-mcp-server"} — same unpinned pattern as context7.
Recommended Action
  • Pin both providers to a known-good npm version (e.g. `@context7/mcp@1.2.3`) and document the version-bump review process; better, fetch the package once into a checked-in lockfile or vendor directory and invoke from there.
  • Add a startup integrity check in `pkg/mcp/registry.go` that refuses to register an MCP provider whose `Command` is `npx` with `-y` and no `@<version>` suffix, with an explicit override flag for development.
HIGH PRAX-2026-05-26-003 No adversarial testing, threat model, or security-disclosure document exists for the protections the harness ships.
Build an AI Red Team
Policy Rule — R-10 (Worker Remit):
"The project SHOULD publish a threat model and a security-disclosure process, and SHOULD run adversarial testing of its own protections — confirming that the secret scanner catches a planted secret, the command guard blocks a catastrophic command, a poisoning write into a session-loaded file is prevented, and a generated configuration genuinely carries the protections it claims."
pkg/hooks/handlers — Existing tests cover the happy-path Execute() return values but no test in the package fires a known-evasion command at the guard or a known-pattern-evading secret string at the scanner. SECURITY.md — File does not exist in the workspace root — no security-disclosure or threat-model document is committed.
Recommended Action
  • Add a `SECURITY.md` with a disclosure mailbox and a brief threat model, and commit a `pkg/hooks/handlers/adversarial_test.go` that fires a curated list of known-evasion commands at the guard (`rm --recursive --force /`, `rm -fr /`, `find / -delete`, `\rm -rf /`) and known-pattern secrets at the scanner.
  • Add a generator-level invariant test that asserts each of the four targets' generated configs contains the declared protective hooks on the appropriate events.
HIGH PRAX-2026-05-26-004 Command guard is a seven-pattern regex denylist trivially bypassable by long-form flag variants of the same destructive commands.
Implement Zero Trust ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-02 (Worker Remit):
"Catastrophic shell commands — recursive deletion from the filesystem root, force-pushing to a main branch, hard resets, destructive database statements, filesystem formatting, raw disk writes — MUST be blocked before they execute."
pkg/hooks/handlers/guard.go:27 — NewCommandGuard() registers exactly seven literal regex patterns (rm -rf /, git push --force main|master, git reset --hard, DROP TABLE|DATABASE, ':>/', mkfs., dd if=…of=/dev/) — long-form flag variants and `find / -delete` are not included.
Recommended Action
Expand the denylist in `NewCommandGuard()` to cover long-form variants (`rm\s+--recursive\s+--force`, `rm\s+-fr`, `find\s+/\s+-delete`, force-push to `(main|master|trunk|develop)`) and add a test fixture file that codifies the bypasses the team has chosen to accept versus block.
HIGH PRAX-2026-05-26-005 SessionLogger handler records only SessionStart/SessionEnd/Stop to a plaintext log — the remit's "tool calls and file modifications" coverage relies entirely on the JSON session store.
Monitor Continuously
Policy Rule — R-08 (Worker Remit):
"The harness MUST maintain a durable, structured, timestamped record of tool calls, blocked actions, and file modifications — detailed enough to reconstruct what the agent did in a session."
pkg/hooks/handlers/sessionlog.go:34 — Events() returns only HookSessionStart, HookSessionEnd, HookStop; Execute() writes a single timestamped line per event — no tool-call or file-modification capture in the handler itself. pkg/harness/harness.go:84 — HandleHookEvent records ToolCallRecord into the session store, but this path is reached only when the harness is invoked via `yaah hook <event>` — which on Codex fires only for SessionStart/Stop, not tool use.
Recommended Action
Extend `SessionLogger.Events()` to include `HookPreToolUse` and `HookPostToolUse`, or add a dedicated `ToolCallLogger` handler that writes a structured JSON line per tool call to a durable log file independent of the session JSON store.
MEDIUM PRAX-2026-05-26-006 The harness configures third-party MCP servers (context7, pulumi, notion) but performs no inspection of their tool descriptions for instruction-like language.
Manage Your Supply Chain Tool descriptions contain no hidden model instructions
Policy Rule — R-04 (Worker Remit):
"MCP tool descriptions MUST NOT contain instruction-like language directed at the model — neither the built-in server's tool descriptions nor, as far as the harness can inspect them, those of the third-party servers it configures."
pkg/mcp/registry.go — Registry.Register and ToConfig surface only the server config (command, args, url, env); no listTools / description-inspection call site exists anywhere under pkg/mcp/.
Recommended Action
Add a `yaah doctor --inspect-mcp` mode (or a CI-time step) that performs an MCP `tools/list` against each configured server and runs the descriptions through a small instruction-like-language detector, surfacing matches for operator review.
MEDIUM PRAX-2026-05-26-007 Root `AGENTS.md` is declared agent-managed and is writable without approval — a structural session-loaded-file poisoning surface absent a confirmed injection path.
Balance Your Knowledge Base ASI06 — Memory and Context Poisoning
Policy Rule — R-06 (Worker Remit):
"Agent-managed, session-loaded files MUST NOT be writable in a way that lets ingested content persist into future sessions unreviewed."
AGENTS.md:2 — HTML comment "Managed by agent: keep sections and order; edit content, not structure" — declares the file agent-writable as part of normal operation; CLAUDE.md is a symlink to AGENTS.md so the same file is consumed at Claude session start.
Recommended Action
Add an `AGENTS.md`-targeted approval gate or a content-diff review hook so agent-driven edits to session-loaded files surface for operator confirmation before they land, and consider git-tracking the file with a pre-commit reviewer to keep changes auditable.
MEDIUM PRAX-2026-05-26-008 Secret scanner skips lock files via the `.lock` binary-extension allowlist, masking secrets accidentally pasted into them.
Implement Zero Trust LLM02 — Sensitive Information Disclosure
Policy Rule — R-05 (Worker Remit):
"Hardcoded credentials in files the agent edits MUST be detected before the change is accepted."
pkg/hooks/handlers/secretscan.go:159 — binaryExts map includes ".lock": true alongside binary formats (.png, .pdf, .so, .dylib); isLikelyBinary returns true for any .lock extension, causing ScanFile to return (nil, nil) before any pattern runs.
Recommended Action
Remove `.lock` from `binaryExts` (or move it to a separate "skip-for-performance" allowlist that the user can opt back in to), and add a regression test covering an `_authToken` line in a `package-lock.json` fixture.
What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

All Go dependencies and CI actions pinned

`go.mod`/`go.sum` pin the yaah dependency tree to specific versions, and every `uses:` in `.github/workflows/ci.yaml` is pinned to a full commit SHA with a version comment — supply-chain hygiene that matches the remit's R-12.

go.mod:1 and .github/workflows/ci.yaml:18

Cosign-signed releases with SBOM

GoReleaser is configured to sign release archives with cosign and emit SPDX SBOMs from Syft, so downstream installs have a verifiable integrity baseline.

.goreleaser.yaml:34

Closed Known Good Baseline of built-in MCP tools

The built-in MCP server registers exactly seven tools (`yaah_scan_secrets`, `yaah_lint`, `yaah_check_command`, `yaah_doctor`, `yaah_session_info`, `yaah_planning_status`, `yaah_planning_init`) — matching the remit's authorized list — and only `yaah_planning_init` writes to the workspace.

pkg/mcpserver/server.go:43

Structured per-session JSON store

`HandleHookEvent` writes a structured `Session` record per session ID with timestamped `ToolCallRecord` entries, blocked-call entries, and a files-modified list — action-level granularity that supports incident reconstruction on Claude and OpenCode targets.

pkg/harness/harness.go:99

Built-in MCP tool descriptions are instruction-free

The seven yaah MCP tool descriptions are plain action statements ("Scan a file for hardcoded secrets and credentials", "Check whether a shell command is safe to run") with no "When using this tool, also…" language — the tool-poisoning indicator from KB_MCP_SECURITY is absent on the built-in side.

pkg/mcpserver/tools.go:59
