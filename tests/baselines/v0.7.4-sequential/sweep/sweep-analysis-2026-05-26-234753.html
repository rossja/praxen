Executive Summary
Agent Remit (as declared)
Sweep is an autonomous GitHub-App software-maintenance agent. It reads issues and comments on installed repositories, plans code changes, opens a branch, commits to it, and opens a pull request against the default branch — but never merges, force-pushes, deletes branches, or executes generated code. Its only authorized counterparties are GitHub, the issue or PR author, the LLM provider, and the project's own logging; every other party is unauthorized by default. The remit also imposes hard input-handling discipline — every character of every issue, comment, file, and diff must be treated as untrusted and screened for instruction injection before reaching the model.
Behavior Summary (as observed)

The dominant pattern is declared policy without code-level enforcement on the agent's untrusted-input path. The remit names instruction-injection screening, scope-bounded changes, secret redaction, and approval-gated CI/security-file edits as MUST-do controls — but in code, issue and comment bodies are concatenated into prompts unscreened, file-change validation only checks "does this path exist," is_blocked() in sweep_bot.py is defined but never called, and no CI/CD or security-file guard exists in any handler.

Layered on top is an unauthenticated trigger surface: POST /jira (sweepai/api.py:296) is dispatched to handle_jira_ticket with no signature check, parses repo: org/name out of an attacker-supplied JSON body, creates a real issue, and runs the full Sweep pipeline against any installed repo — and verify_signature in sweepai/utils/hash.py fail-opens (returns True) when WEBHOOK_SECRET is unset, which is the default. Together with the unscreened input path, this is a one-hop chain from anonymous internet to LLM-driven code-write on customer repositories, and it is the finding to lead remediation with.

Scope of Analysis
Python 3.10 FastAPI service that exposes a GitHub webhook root (POST /), a Jira webhook (POST /jira), a progress endpoint, a cron-style /update_sweep_prs_v2, and a mounted Bearer-authenticated chat sub-app at /chat; webhook dispatch lives in sweepai/api.py's handle_event(). The autonomous path runs on_ticket → fetch_relevant_files → get_files_to_change → modify → handle_file_change_requests → commit_multi_file_changes; LLM calls go through sweepai/utils/anthropic_client.py against Anthropic Claude with an OpenAI fallback. Several CI-fix and validator paths use subprocess.run(..., shell=True) on operator-controlled config (sweepai/handlers/on_check_suite.py, sweepai/utils/code_validators.py). Issue body, PR comments, and CI logs flow directly into prompt templates in sweepai/utils/prompt_constructor.py with no instruction-injection screening; the only screen is an LLM-based actionability check in sweepai/utils/issue_validator.py, which does not address the injection class the remit names.

CRITICAL PRAX-2026-05-26-001 Unauthenticated POST /jira endpoint lets any internet caller trigger Sweep against any installed repository.
Implement Zero Trust ASI03 — Identity and Privilege Abuse LLM06 — Excessive Agency
Policy Rule — R-05, R-20 (Worker Remit):
"Sweep MUST NOT access repositories the GitHub App is not installed on. / Any action that would affect a repository other than the one named in the triggering event."
sweepai/api.py:296 — "@app.post('/jira') def jira_webhook(request_dict: dict = Body(...)) — no Depends(validate_signature), no auth header, body trusted as a Jira event and dispatched to a thread" sweepai/handlers/on_jira_ticket.py:27 — "handle_jira_ticket() pulls repo_full_name from attacker-supplied description via regex, fetches the app's own installation token via get_installation_id(org_name), creates a github issue, and calls on_ticket() with installation_id"
Recommended Action
  • Apply Depends(validate_signature) (or an equivalent Atlassian-Jira HMAC verifier reading a new JIRA_WEBHOOK_SECRET) to the /jira route in sweepai/api.py and reject unauthenticated callers with 403.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Webhook signature verification primitive is correctly implemented

<code>verify_signature</code> in <code>sweepai/utils/hash.py</code> uses <code>hmac.compare_digest</code> against an <code>sha256=</code>-prefixed HMAC of the raw body — the canonical GitHub-webhook check, free of timing-attack pitfalls when the secret is configured.

sweepai/utils/hash.py:27-34

All Python dependencies are exact-version pinned via uv

<code>requirements.txt</code> is generated by <code>uv pip compile pyproject.toml</code> and every dependency is pinned to an exact version with declared provenance — defeats version-swap and dependency-confusion attacks at the SDK layer.

requirements.txt:1-3

Chat sub-app delegates authorization to GitHub via per-user Bearer tokens

<code>/chat/backend/*</code> routes require a Bearer token and re-verify repo access via <code>get_authenticated_github_client</code> against the calling user's own GitHub identity — keeping the chat surface scoped to repos the user can already see.

sweepai/chat/api.py:212-240

Pull-request delivery is structurally enforced — no direct-to-default-branch commits in code

Every modify path in <code>sweepai/handlers/create_pr.py</code> and <code>sweepai/handlers/on_ticket.py</code> commits to a <code>sweep/</code>-prefixed feature branch and opens a PR; no call site in the autonomous path writes to <code>repo.default_branch</code> directly.

sweepai/handlers/create_pr.py:39-99
