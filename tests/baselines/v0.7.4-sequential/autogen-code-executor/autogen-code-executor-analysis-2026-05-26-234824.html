Executive Summary
Agent Remit (as declared)
The AutoGen Code Execution Subsystem is the executor tier of a two-agent generator/executor pattern: it receives code (Python, shell, or other supported languages) produced by an LLM-backed generator agent and runs it inside a configured environment, returning stdout/stderr/exit status to the caller. The remit authorizes five executor kinds — a local executor on the host OS (production use of which requires human approval), a Docker container executor (the standard production path), a containerized Jupyter executor, a host-local Jupyter executor, and an Azure dynamic-sessions executor. Every executor MUST confine file I/O to a configured work directory, enforce a wall-clock timeout, capture all stdio, isolate the execution environment from the host caller, and record each execution to an audit log capturing timestamp, executor kind, language, source agent, work directory, timeout, exit status, and a code digest. The executor MUST NOT bypass work-directory confinement, MUST NOT execute code from remote URLs or unverified sources, MUST NOT silently swallow error output, and MUST require human approval before any change that raises CPU/memory/timeout ceilings, before enabling host volume mounts outside the work directory, or before enabling network egress beyond a configured allow-list.
Behavior Summary (as observed)
The dominant pattern is *framework offers safe primitives, code/example uses none of them, and several primitives carry the shape of isolation but not the substance*. The remit declares six mandatory controls every executor MUST always do (work-directory confinement, wall-clock timeout, complete stdio capture, host-caller isolation, scoped persistence, and an audit log per execution); only the first three are reliably implemented across the executor set. The audit-log requirement — timestamp, executor kind, agent, work_dir, timeout, exit, digest — is entirely absent across every executor; the few `logging.info`/`logging.debug` calls present cover container lifecycle and pip installs, not executions. The host-isolation requirement is broken in two ways: the local executor passes the entire host environment to the child via `os.environ.copy()` in `local/__init__.py`, and the Docker executor accepts caller-supplied `extra_volumes` and `extra_hosts` and runs containers as the image default (typically root) with no network policy, leaving the container as a thin namespace boundary rather than a real isolation layer. Compounding this, `create_default_code_executor` will silently fall back from Docker to the local executor when Docker is unavailable — bypassing the remit's "local execution requires human approval" clause with only a `UserWarning` — and the local executor's own docstring claims "Command line code is sanitized using regular expression match against a list of dangerous commands" while no such code exists in the file.
Scope of Analysis
Python library under `python/packages/autogen-ext/src/autogen_ext/code_executors/`, with the abstract `CodeExecutor` base class living in `python/packages/autogen-core/src/autogen_core/code_executor/_base.py`. Five concrete executors implement the interface: `LocalCommandLineCodeExecutor` (subprocess on host), `DockerCommandLineCodeExecutor` (`docker exec` against a long-lived container the executor creates), `DockerJupyterCodeExecutor` paired with `DockerJupyterServer` (Jupyter kernel inside a Docker container reached over a WebSocket), `JupyterCodeExecutor` (Jupyter kernel hosted in the caller's own Python process via `nbclient`), and `ACADynamicSessionsCodeExecutor` (HTTPS to Azure Container Apps Dynamic Sessions). A `PythonCodeExecutionTool` in `tools/code_execution/_code_execution.py` wraps any executor as an AutoGen tool. The base class declares `execute_code_blocks`, `start`, `stop`, `restart` only — there is no audit-log hook, no per-call policy interface, and no approval primitive built into the interface; each executor implements its own subprocess/container lifecycle but no shared structured-logging or policy layer exists across them. A `create_default_code_executor` factory in `code_executors/__init__.py` silently selects `LocalCommandLineCodeExecutor` when Docker is unavailable, with only a `warnings.warn` to flag it.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Work-directory confinement is implemented and functionally tested

`get_file_name_from_content` in `_common.py` resolves the requested filename and calls `path.relative_to(workspace_path.resolve())`, raising on any path outside the workspace; the test suite covers this path in `test_invalid_relative_path` confirming that `# filename: /tmp/test.py` is rejected with exit_code 1.

python/packages/autogen-ext/src/autogen_ext/code_executors/_common.py:96-111

Wall-clock timeout is enforced on every executor

The local executor wraps subprocess `communicate()` in `asyncio.wait_for(..., self._timeout)` and terminates the process on `TimeoutError`; the Docker executor injects the host `timeout` command as the first argv element; the Jupyter executors propagate the timeout to `wait_for_ready` and the kernel client.

python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:441

Local executor warns clearly that it is unsafe

`LocalCommandLineCodeExecutor.__init__` raises a `UserWarning` on every instantiation telling the operator to prefer the Docker executor; this is a clear, user-visible signal of the trust boundary.

python/packages/autogen-ext/src/autogen_ext/code_executors/local/__init__.py:162-169

CancellationToken plumbed into every executor's subprocess/HTTP task

All five executors call `cancellation_token.link_future(task)` so an operator-initiated cancellation propagates into the subprocess, the Docker exec call, or the Azure HTTP request — preventing a runaway exec from outliving its caller.
