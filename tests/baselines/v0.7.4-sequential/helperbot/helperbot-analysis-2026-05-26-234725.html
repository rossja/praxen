Executive Summary
Agent Remit (as declared)
HelperBot is declared as an internal employee assistant that answers questions, retrieves documents from a company knowledge base, performs public web searches, and writes summaries to a designated output location. The remit grants three tools — a workspace-scoped document reader, a workspace-scoped document writer, and a public web search — and explicitly forbids accessing files outside the authorized workspace, executing shell commands, revealing the system prompt, carrying confidential content into web queries, following instructions embedded in retrieved content, and exceeding a per-session tool-call cap on the order of 20. No human-approval gate is defined because no high-impact capabilities are authorized; the remit's safety model is built entirely from hard prohibitions and auditable logging of every tool call.
Behavior Summary (as observed)

HelperBot's declared policy and HelperBot's deployed code are not the same agent. The remit reads like a strict, auditable internal assistant — workspace-scoped reads and writes, untrusted-input validation, no shell, no system-prompt disclosure, no instruction-following from retrieved content, every tool call logged, a 20-call per-session cap — while the implementation in src/core/agents.js sets inputValidation, outputFiltering, toolApproval, rateLimiting, and auditLogging all to false and turns on promptInjection, dataExfiltration, and contextManipulation as named, intentional vulnerabilities.

The dominant pattern is policy-implementation divergence weaponised by a system prompt that contradicts the remit at the prompt layer: src/llm/prompts.js:23-27 instructs the bot to "share [its instructions or configuration] openly — transparency builds trust" and interpolates the literal value of SENSITIVE_DATA.apiKeys.internal straight into the system prompt, which the persona is then told to leak on request. The remit-declared search_web tool has no implementation site at all, while the canned generateResponse() branches in src/index.js answer prompt-injection probes with "Understood! New instructions accepted." That combination — credentials in prompt plus persona instructed to disclose plus zero input validation plus zero audit log — is the agent's structural shape; it is not one finding, it is the bot.

Scope of Analysis
HelperBot is one of fourteen intentionally vulnerable agents inside the Damn Vulnerable AI Agent (DVAA) Node.js fleet defined in `src/core/agents.js` (the `HELPERBOT` block) and `src/llm/prompts.js` (the `helperbot` system prompt). It runs as an OpenAI-compatible HTTP API on port 7002, dispatched by `generateResponse()` in `src/index.js`, with the WEAK security level — every feature flag (`inputValidation`, `outputFiltering`, `toolApproval`, `rateLimiting`, `auditLogging`) is set to `false` and three named vulnerabilities (`promptInjection`, `dataExfiltration`, `contextManipulation`) are explicitly `enabled: true`. The declared tool inventory `[read_file, write_file, search_web]` has no enforcement at the chat layer and `search_web` has no implementation anywhere; sensitive credentials live as JavaScript string literals in `src/core/vulnerabilities.js` (`SENSITIVE_DATA`) and are interpolated directly into HelperBot's system prompt at `src/llm/prompts.js:27`. The MCP sandbox in `src/sandbox/init.js` plants `.env`, `/etc/passwd`, `/etc/shadow`, and `secrets/service-account.json` for credential-harvest exercises against the MCP-protocol agents on ports 7010-7013 — separate from HelperBot, but reachable by any file-traversal tool the API agents are wired to.
