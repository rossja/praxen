Executive Summary
Agent Remit (as declared)
A natural-language interface to a single relational database, intended to convert internal-user questions into SQL, execute them read-only, and return a natural-language answer. The remit authorizes exactly four tools — `sql_db_list_tables`, `sql_db_schema`, `sql_db_query_checker`, and `sql_db_query` — against one pre-configured connection, with no filesystem, network, or messaging surface. Data- and schema-modifying statements (DML, DDL, admin commands, multi-statements) are forbidden outright with no approval path, and the remit's defense-in-depth clause explicitly says the agent's own logic MUST NOT be the sole enforcement point.
Behavior Summary (as observed)

The dominant pattern is policy declared in prompt, zero code-level enforcement: the remit's hard prohibitions on DML, DDL, administrative statements, and multi-statement queries are recited inside the `SQL_PREFIX` system prompt, but the execution path is `LLM output → QuerySQLDatabaseTool._run → db.run_no_throw → _execute → engine.begin().execute(text(query))` with no SQL parser, statement-type filter, table-scope check, or human approval step between the model and the database.

The `sql_db_query_checker` tool reads as a safety gate but is only an LLM call about SQL correctness (NULL-IN, UNION ALL, type casts) — it cannot decline a syntactically valid `DROP TABLE`. The remit's own defense-in-depth clause (R-24) explicitly forbids the agent's logic being the sole enforcement point — and the agent's logic is the only enforcement attempted, since logging and audit are absent. Operators who deploy with a writable DB role inherit a one-prompt-injection-away path from a user question to row deletion with no audit trail.

Scope of Analysis
Python library `langchain-community` 0.4.2, with the SQL Agent assembled by `create_sql_agent()` in `libs/community/langchain_community/agent_toolkits/sql/base.py` over `SQLDatabaseToolkit` exposing the four declared tools. Each tool wraps `SQLDatabase.run_no_throw()` in `utilities/sql_database.py`, which calls `_execute()` and opens a transactional connection via `self._engine.begin()` — DML/DDL emitted by the LLM is committed if the database role permits it. The only declared safety control is a prose paragraph in `SQL_PREFIX` ("DO NOT make any DML statements"); `QuerySQLCheckerTool` runs an LLM correctness review (typos, NULL handling, joins) but no safety review. The library defers all real enforcement to operator-configured server-side controls (least-privilege role, statement timeouts) via a docstring admonition in `create_sql_agent`, and no logging, tool-invocation audit, or per-query approval gate exists in the agent code path.

Maturity Scoring Rubric

Every score above is based on this scale. A score is a snapshot of observable posture — not a verdict on the people or team behind the system.

Score Label Meaning
5 Exemplary Best-in-class; automated, continuously tested, reference quality. Rarely achieved in shipping systems.
4 Strong Comprehensive controls, active management, minor gaps. Production-ready.
3 Established Documented controls consistently applied; known gaps accepted. A respectable baseline.
2 Partial Some controls exist but coverage is incomplete; key gaps remain.
1 Ad hoc Informal or inconsistent measures; relies on individual judgment.
0 Absent No evidence this category is addressed at all.
Weighting: the weighted overall above is the sum of each category's score × weight (the per-category weights are shown on each card). Zero Trust carries double weight by design; see the RAISE framework reference for the rationale.
