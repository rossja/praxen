Executive Summary
Agent Remit (as declared)
Devika is an autonomous software-engineering agent that takes a high-level project description, plans a multi-step solution, writes code, runs it, and reports back — operating inside a per-project workspace on the operator's local machine. The remit authorizes web research, code generation, code execution inside a sandboxed runner, and inference against operator-configured LLM providers, while explicitly forbidding any file I/O outside the project work directory, arbitrary shell execution outside the sandbox, package installs without explicit user confirmation, self-modification of code or memory, and communication with services beyond the LLM and search providers. Every action — plan step, file write, shell execution, web fetch — must be recorded to a per-project audit log, untrusted input (web content, user prompts) must be screened for injection before reaching LLM context, and wall-clock and step-count caps must bound each run.
Behavior Summary (as observed)

Devika's dominant pattern is policy declared in prose, zero code-level enforcement — and the gap concentrates on the most consequential remit clauses. The remit promises a sandboxed runner; src/sandbox/code_runner.py and src/sandbox/firejail.py are both zero-byte stubs while Runner.run_code calls subprocess.run directly on LLM-produced command strings with no allowlist, approval gate, or step cap. The remit forbids file I/O outside the project work directory; Coder.save_code_to_project writes os.path.join(self.project_dir, project_name, file['file']) with no path-traversal check, and the Flask /api/get-browser-snapshot endpoint passes a user-supplied snapshot_path straight into send_file. The remit forbids communication beyond LLM and search providers; the Decision agent advertises Twitter/Reddit posting as a legitimate browser_interaction use case, the Action handler invokes Netlify().deploy(), and the API server binds to 0.0.0.0 on every interface with no authentication, exposing settings rewrite, project deletion, and arbitrary-path file read to anyone on the network.

The compound chain is straightforward and present end-to-end in the source: a user prompt or attacker-controlled web page (no injection screen) reaches the Coder/Runner agents, the LLM emits a commands[] array, and subprocess.run executes those commands as the application user with no audit trail beyond a free-form devika_agent.log. There are no tests, no red-team artifacts, and ROADMAP.md lists "Create an extensive testing suite" as still-unchecked.

Scope of Analysis
Python Flask + Socket.IO backend (`devika.py`) bound to `0.0.0.0:1337` orchestrating eleven Jinja2-templated sub-agents (`Planner`, `Researcher`, `Coder`, `Action`, `Runner`, `Decision`, etc.) over pluggable LLM providers (Claude, OpenAI, Gemini, Mistral, Groq, Ollama, LM Studio). Code execution flows through `src/agents/runner/runner.py` which calls `subprocess.run(command.split(" "), cwd=project_path)` on LLM-generated commands; the `src/sandbox/code_runner.py` and `src/sandbox/firejail.py` modules that `ARCHITECTURE.md` describes as the "sandboxed environment" are present as zero-byte stub files. Project and conversation state persist to a single shared SQLite database (`data/db/devika.db`) — `Projects`, `AgentStateModel`, and `Knowledge` tables — with no per-project isolation in the `Knowledge` table. External-service surface includes `src/services/netlify.py` (deploy), `src/services/github.py` (repo enumeration), and `src/browser/interaction.py` (Playwright-driven browser whose `start_interaction` feeds raw page text into LLM prompts and back as click/type commands), all reachable without authentication through the unprotected Flask + Socket.IO surface.

CRITICAL PRAX-2026-05-26-001 LLM-produced commands are executed via `subprocess.run` with no sandbox, allowlist, approval gate, or step cap.
Implement Zero Trust LLM05 — Improper Output Handling ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-01, R-11 (Worker Remit):
"Devika MUST NOT execute arbitrary shell commands outside its sandboxed runner. / Control MUST return to the user before any action that installs packages, writes outside a previously-touched file, or executes a shell command whose output was not anticipated in the plan."

Recommended Action
  • In `src/agents/runner/runner.py:run_code`, replace the direct `subprocess.run` invocation with a call into an actually-implemented sandbox runner (firejail/nsjail/container exec) that takes a per-command allowlist; for any command outside the allowlist, route through a synchronous operator-approval gate before execution and record the command + outcome to the per-project audit log.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Container runtime drops to nonroot user

`devika.dockerfile` creates a `nonroot` group and user and `chown`s the application directory before the `USER nonroot` switch, so the application container does not run as root — limiting blast radius for any in-container exploit.

devika.dockerfile:6,33

Werkzeug filename sanitization on project-name API parameters

`src/apis/project.py` wraps every `project_name` argument arriving over the Flask API in `werkzeug.utils.secure_filename` before joining it into a filesystem path, which strips `..` and directory separators from the project-name vector specifically.

src/apis/project.py:21,30,38,47,56

`get_project_files` enforces a base-path containment check

`ProjectManager.get_project_files` computes `os.path.commonprefix([directory, base_path])` and returns `[]` if the resolved directory escapes `data/projects`, which is the right pattern — though it is not applied to the Coder write path or to other read endpoints.

src/project.py:157

Inference call has a configurable timeout

`LLM.inference` runs the provider call in a `ThreadPoolExecutor` and raises `TimeoutError` after `config.TIMEOUT.INFERENCE` seconds (default 60), preventing a hung provider from blocking the worker thread indefinitely.
