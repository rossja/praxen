Executive Summary
Agent Remit (as declared)
A local developer command-line tool that scaffolds, bundles, and ships a `deepagents` agent project — turning a developer's project folder into a deployment artifact and either serving it locally on `langgraph dev` or pushing it to the LangGraph deployment platform. Authorized subcommands are `init` (scaffold), `dev` (local serve), and `deploy` (ship); the CLI talks to the deployment platform, the LangSmith Hub context service, and the LangSmith tracing service using the developer's own credentials. Its central promise is that the bundle faithfully and only reflects the developer's reviewed project — a deploy that would leave the deployed API open to anyone with the URL MUST surface a prominent warning and obtain explicit operator confirmation, and credentials MUST NOT be folded into the seeded-memory or skills payload.
Behavior Summary (as observed)
The CLI's central guarantee — *only what the developer declared, nothing more* — is well-implemented at the seam between project layout and bundle artifact: stdio MCP is rejected pre-deploy, `.env` is explicitly excluded from the seeded-memory payload, the deploy bundle's dependency set is derived only from declared providers, and an anonymous-auth deploy hits a colour-coded warning and an explicit `[y/N]` confirmation. The gaps cluster on the other side of that seam — what gets *taken on faith* once a project is "declared." Remote MCP servers in the user's `mcp.json` are copied verbatim into the deployment with no version pin or integrity check (the remit's Known-Good-Baseline rule explicitly says they MUST be pinned), the http/sse transport check accepts plain `http://` URLs without warning, and the CLI's own threat model — the artifact the remit's *Escalation and Limits* clause asks to be kept current — carries a disclaimer that whole sections of it no longer apply after the 0.1.0 package split. None of these is a runtime injection chain on its own, but together they describe a posture where the *bundling* path is tightly held and the *trust posture of the deployment* leans heavily on what the developer wrote in their `mcp.json` and whether they happen to read the colour-coded prompt.
Scope of Analysis
Python package `deepagents_cli` shipping under `libs/cli/` at version 0.1.2; entry point `deepagents_cli.main:cli_main` wires three argparse subcommands (`init`, `dev`, `deploy`) to handlers in `deepagents_cli.deploy.commands`. The bundler in `deepagents_cli/deploy/bundler.py` reads a declared project layout (`deepagents.toml`, `AGENTS.md`, optional `.env`/`mcp.json`/`skills/`/`user/`/`subagents/`) and renders `_seed.json`, `deploy_graph.py`, `pyproject.toml`, `langgraph.json`, and (when configured) `auth.py` plus a pre-built React frontend into a temp build directory; `dev` and `deploy` then shell out to `langgraph` (`dev` / `deploy` subcommands). Config parsing in `deploy/config.py` is strict — unknown TOML sections, unknown keys, and stdio MCP transports are rejected before bundling. The anonymous-auth gate lives in `commands.py:_deploy` (lines 287-326); supply-chain hygiene comes from a committed hashed `uv.lock`, a Dependabot config for monthly uv + GitHub-actions updates, and a THREAT_MODEL.md documenting trust boundaries (now explicitly outdated post the 0.1.0 split).

Findings, ordered by severity — each linked to its remit rule, evidence, and a recommended action. Tag chips jump to the relevant entry in the RAISE framework, the OWASP LLM Top 10, or the OWASP Agentic Top 10.

HIGH PRAX-2026-05-26-001 Remote MCP servers are carried into the deployment bundle with no version pin or integrity check.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities
Policy Rule — R-17 (Worker Remit):
"Remote MCP servers declared in the project MUST be pinned to a known-good, integrity-checked version, so the deployed agent does not auto-install an unpinned server afresh."
libs/cli/deepagents_cli/deploy/bundler.py:193 — bundler copies project `mcp.json` to `build_dir/_mcp.json` via `shutil.copy2` with no fingerprint or version validation; lines 193-197. libs/cli/deepagents_cli/deploy/templates.py:446 — generated `_load_mcp_tools` accepts whatever `url` and `headers` the bundled `_mcp.json` carries (with `${VAR}` expansion) and passes the result straight to `MultiServerMCPClient` — no integrity check on the URL or response.
Recommended Action
  • Add a `version` (or `sha256` / `commit_sha`) field to the MCP server schema in `deploy/config.py` and require it for every remote MCP server entry; reject the bundle when an entry lacks one.
  • In the generated `_load_mcp_tools`, validate the remote server's identity on first connection (e.g., compare a declared fingerprint against the server's MCP `initialize` capability set) and refuse to load if it does not match.
MEDIUM PRAX-2026-05-26-002 The MCP transport allow-list accepts plain `http://` URLs — TLS is not actually enforced for remote MCP servers carried into the bundle.
Implement Zero Trust LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities
Policy Rule — R-04 (Worker Remit):
"Remote MCP servers carried into the bundle MUST be reached over TLS, and their transport configuration MUST be validated before the bundle is shipped."
libs/cli/deepagents_cli/deploy/config.py:293 — loop in `_validate_mcp_for_deploy` only checks `transport == "stdio"`; no inspection of the `url` field's scheme, so `http://` URLs validate clean (lines 293-300). libs/cli/deepagents_cli/deploy/templates.py:460 — generated `_load_mcp_tools` `os.path.expandvars`-expands `${VAR}` references and surfaces unresolved refs only as `logger.warning(...)` — the bundle proceeds with literal `${VAR}` strings in URLs/headers.
Recommended Action
  • In `_validate_mcp_for_deploy`, after the transport check, parse `cfg["url"]` and reject any entry whose scheme is not `https`.
  • Treat unresolved `${VAR}` references in MCP URLs or headers as a bundle-time validation error rather than a runtime `logger.warning`.

MEDIUM PRAX-2026-05-26-004 Published wheels carry no Sigstore provenance because PyPI attestations are explicitly disabled in the release workflow.
Manage Your Supply Chain LLM03 — Supply Chain
.github/workflows/release.yml — two occurrences of `attestations: false` with a "Temp workaround" comment — the project ships through PyPI Trusted Publishing but suppresses the Sigstore attestation that would let downstreams verify provenance.
Recommended Action
Remove the `attestations: false` override in `.github/workflows/release.yml` (now that `gh-action-pypi-publish` v1.11.0+ supports attestations by default) and re-publish under attested provenance.
MEDIUM PRAX-2026-05-26-005 The anonymous-deploy operator-confirmation gate has no automated regression test, despite being the load-bearing control for the remit's R-09 promise.
Build an AI Red Team
libs/cli/tests/unit_tests/deploy/test_commands.py — `grep -n "anonymous\|confirm" test_commands.py` returns no matches; the file's TestInitProject + TestAutoWireIssuesBoard classes are the only tests, neither exercises the `_deploy` anonymous-confirmation path. libs/cli/deepagents_cli/deploy/commands.py:287 — anonymous-auth confirmation block at lines 287-326 — the load-bearing gate the remit's R-09 maps to has no corresponding assertion in the test suite.
Recommended Action
Add `tests/unit_tests/deploy/test_commands.py::TestDeployAnonymousConfirmation` covering: (a) the warning text is printed; (b) `input` returning `n`/`N`/empty/EOF aborts with SystemExit; (c) `--dry-run` path skips the input prompt; (d) non-anonymous configurations do not prompt.
MEDIUM PRAX-2026-05-26-006 The CLI writes no structured deploy-event log — informational events go to a stdlib logger with no handler, and human-facing output to stdout only.
Monitor Continuously
libs/cli/deepagents_cli/config.py:18 — `_stderr` helper notes "the CLI does not configure logging handlers by default, so `logger.warning` output is invisible to end users" — applies a fortiori to `logger.info`. libs/cli/deepagents_cli/deploy/bundler.py:186 — `logger.info("Wrote _seed.json (memories: %d, skills: %d, user_memories: %d)", ...)` and similar info-level events at bundler.py:197/223/232/251/257 describe what was bundled but reach no log file by default.
Recommended Action
Write a structured per-deploy event log (JSONL) to `./.deepagents-deploy.log` or `~/.deepagents/deploy.log` capturing: timestamp, agent name, model, sandbox provider, auth provider, memories backend, MCP server URLs (with secrets redacted), build directory, deploy URL on success, and exit status on failure.
What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

`.env` excluded from seeded-memory payload

The bundler copies `.env` to `build_dir/.env` as a sidecar but explicitly does *not* fold it into `_seed.json`, so secrets never ride along as agent memory or skills content.

libs/cli/deepagents_cli/deploy/bundler.py:200-206

Strict allow-list TOML config parsing

`_parse_config` rejects any TOML section not in `{agent, sandbox, auth, memories, frontend}` and any key not in the corresponding per-section allow-list — undeclared content cannot silently widen the bundle.

libs/cli/deepagents_cli/deploy/config.py:447-466

Stdio MCP transport rejected pre-deploy

`_validate_mcp_for_deploy` walks every `mcpServers` entry and rejects any whose `type`/`transport` resolves to `stdio` — the deployed agent cannot accidentally pick up a local-only MCP transport that has no place in a hosted bundle.

libs/cli/deepagents_cli/deploy/config.py:281-301

Anonymous-auth deploy gated by explicit operator confirmation

When `[auth].provider == "anonymous"` with `[frontend].enabled`, `_deploy` prints a bold-red banner naming three specific consequences (URL-as-auth, no API auth, suggested supabase/clerk alternative) and requires a `y`/`yes` reply before proceeding; `Ctrl-C` / EOF / `N` all abort with exit code 1.

libs/cli/deepagents_cli/deploy/commands.py:287-326

Pinned, hashed lockfile committed for the CLI package

`libs/cli/uv.lock` is a 4031-line uv-resolved lockfile with SHA-256 hashes on every wheel and sdist; `libs/cli/frontend/package-lock.json` covers the React bundle; Dependabot updates uv and GitHub-actions on a monthly cadence with a separate group for majors.
