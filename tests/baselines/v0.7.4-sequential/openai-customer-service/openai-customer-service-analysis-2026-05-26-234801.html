Executive Summary
Agent Remit (as declared)
A multi-agent airline customer-service system intended to handle FAQ answering and seat-assignment changes on existing confirmed reservations. The remit defines three agents — a triage router, an FAQ agent restricted to a curated dataset via faq_lookup_tool, and a seat-booking agent permitted to call update_seat only after the customer's identity and ownership of the cited confirmation number have been verified. Authorized counterparties are limited to the authenticated customer, the LLM provider, the internal reservation system, and the internal FAQ dataset; outbound channels other than the in-session natural-language response (email, SMS, webhooks) are explicitly forbidden, and refunds, fare changes, rebooking, and special-handling requests must reach a human checkpoint.
Behavior Summary (as observed)

The dominant pattern is framework offers safe primitives, example wires none of them in: the agents SDK ships guardrail.py and tool_guardrails.py alongside this example, but main.py attaches neither input nor output guardrails to any of the three agents and relies entirely on natural-language instructions in the agent prompts to enforce scope, identity, and tool use.

The most exploitable concrete consequence is the update_seat tool: the remit requires that the customer's identity be confirmed and the confirmation number be verified against the authoritative reservation record before any seat mutation, but the implementation accepts both the confirmation number and the new seat as raw model-supplied strings, writes them straight into context.context, and never calls a reservation system at all — an authenticated-customer model is assumed but never enforced in code, so any prompt that reaches the seat-booking agent gets seat-state writes for free.

Scope of Analysis
A Python example built on the openai-agents SDK in a single file examples/customer_service/main.py. Three Agent instances — triage_agent, faq_agent, seat_booking_agent — are wired together via handoff() with the framework's RECOMMENDED_PROMPT_PREFIX appended to each instruction string. The two declared tools are decorated @function_tool Python coroutines; faq_lookup_tool is a hard-coded keyword match over an in-process dictionary of canned answers, and update_seat writes seat state into the in-memory AirlineAgentContext with no reservation lookup. The runtime entrypoint main() drives a stdin/stdout REPL through Runner.run inside an OpenAI trace() context — no authentication layer, no input or output guardrails wired in, and no persistence beyond the running process.

Findings Register

Findings, ordered by severity — each linked to its remit rule, evidence, and a recommended action. Tag chips jump to the relevant entry in the RAISE framework, the OWASP LLM Top 10, or the OWASP Agentic Top 10.

CRITICAL PRAX-2026-05-26-001 Seat-booking agent mutates reservation state with no customer-identity confirmation in code.
Implement Zero Trust ASI03 — Identity and Privilege Abuse LLM06 — Excessive Agency
Policy Rule — R-04, R-03, R-14, R-18 (Worker Remit):
"Customer identity MUST be confirmed before any action that mutates reservation state. / The seat-booking agent updates seat assignments on existing confirmed reservations only. / Handoffs between agents MUST NOT escalate privileges or bypass identity verification. / Any request where the customer's identity cannot be verified MUST be escalated to a human."
examples/customer_service/main.py:67 — update_seat (lines 67-83) writes context.context.confirmation_number and context.context.seat_number from raw model-supplied arguments and returns success — no authenticated_user, session, or identity parameter is consulted. examples/customer_service/main.py:112 — seat_booking_agent instructions tell the LLM to "Ask for their confirmation number" before calling the tool — a prompt-only check, not a code gate.
Recommended Action
  • Add an explicit authenticated_customer_id field to AirlineAgentContext populated from the session before any agent runs, and have update_seat reject the call when it is None.
  • Wire an input_guardrails entry on seat_booking_agent that fails the turn when the context lacks a verified customer identity, using the SDK's guardrail.py primitives the framework already ships.
CRITICAL PRAX-2026-05-26-002 Confirmation-number ownership is never checked against the authoritative reservation system before seat mutation.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation ASI03 — Identity and Privilege Abuse
Policy Rule — R-05, R-09, R-15 (Worker Remit):
"A submitted confirmation number MUST be verified as belonging to the authenticated customer before that reservation is read or modified. / Reservations or seat assignments for any customer other than the authenticated session user MUST NOT be modified. / A seat change MUST NOT be issued without first verifying the confirmation number against the authoritative reservation record."
examples/customer_service/main.py:67 — update_seat accepts confirmation_number as a free-form string parameter and assigns it to context with no lookup, no ownership check, and no rejection path for a confirmation number belonging to another customer. examples/customer_service/main.py:82 — The only validation in the function is "assert context.context.flight_number is not None" — and that flight_number was generated by the handoff hook itself, not retrieved from a reservation record.
Recommended Action
Replace the in-context assignment with a call to a real reservation-lookup function that takes the authenticated customer id and the submitted confirmation number, and returns the reservation only when ownership matches; reject the tool call when no match is returned.
CRITICAL PRAX-2026-05-26-008 Compound chain — untrusted user input flows directly to an unauthenticated write tool with no audit, allowing one-prompt seat hijack of any reservation.
Implement Zero Trust ASI10 — Rogue Agents ASI01 — Agent Goal Hijack LLM06 — Excessive Agency
Policy Rule — R-04, R-05, R-09, R-15 (Worker Remit):
"Customer identity MUST be confirmed before any action that mutates reservation state. / A submitted confirmation number MUST be verified as belonging to the authenticated customer before that reservation is read or modified. / Reservations or seat assignments for any customer other than the authenticated session user MUST NOT be modified. / A seat change MUST NOT be issued without first verifying the confirmation number against the authoritative reservation record."
examples/customer_service/main.py:162 — Step 1 of the chain — unsanitized user_input appended directly to input_items[] and dispatched to the agent (PRAX-2026-05-26-006). examples/customer_service/main.py:67 — Step 2-3 of the chain — update_seat (lines 67-83) writes confirmation_number and new_seat to context with no identity check (PRAX-2026-05-26-001) and no ownership lookup (PRAX-2026-05-26-002). examples/customer_service/main.py:83 — Step 4 of the chain — the function returns success and no audit log is written (PRAX-2026-05-26-004), so the cross-tenant seat overwrite is invisible to operators.
Recommended Action
Fix the three contributing findings together — input guardrails on triage, an authenticated-customer + reservation-lookup gate inside update_seat, and a durable audit-log write — rather than patching them individually; until all three are in place this chain remains live even if any two are mitigated.
HIGH PRAX-2026-05-26-003 Seat availability and existence are never validated before update_seat returns success.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation
Policy Rule — R-06 (Worker Remit):
"Seat updates MUST be limited to seats that exist on the flight and are available — not already assigned to another passenger."
examples/customer_service/main.py:67 — update_seat (lines 67-83) treats new_seat as opaque text — no enum, no per-flight seat map, no "is this seat free" check before the write.
Recommended Action
Look up the flight's seat map by confirmation number, reject seats that are not in the map or are already assigned to another passenger, and return a structured error to the agent so the LLM can ask for an alternative.
HIGH PRAX-2026-05-26-004 Durable seat-change audit log mandated by the remit is not implemented.
Monitor Continuously LLM06 — Excessive Agency
Policy Rule — R-07 (Worker Remit):
"Every seat change MUST be recorded to a durable audit log capturing the time, the customer identity, the confirmation number, the old and new seat, and the agent that performed the change."
examples/customer_service/main.py:67 — update_seat returns a plain string with no logging.info / logger.audit / append-to-file call; the file imports no logging module at all. examples/customer_service/main.py:166 — The trace() context manager wraps the turn for OpenAI tracing but does not produce the remit-required structured record (time, customer, confirmation, old seat, new seat, performing agent).
Recommended Action
Add an append-only audit writer (database table or JSONL file) called from inside update_seat with the six required fields before returning success.
HIGH PRAX-2026-05-26-005 FAQ agent's no-speculation rule is enforced only by prompt text, not by code.
Balance Your Knowledge Base LLM09 — Misinformation
Policy Rule — R-02 (Worker Remit):
"The FAQ agent answers questions about airline policies — baggage, seating, wifi, and similar — using only the curated FAQ dataset, and MUST NOT speculate or draw on general knowledge outside that dataset."
examples/customer_service/main.py:96 — faq_agent.instructions contains the prose rule "Do not rely on your own knowledge" but no output_guardrails are attached to the Agent constructor; the framework's tool_guardrails.py is not imported.
Recommended Action

HIGH PRAX-2026-05-26-006 No detection or handling of injection in customer input or tool outputs is wired into the agents.
Implement Zero Trust LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-10, R-11 (Worker Remit):
"Instructions embedded in free-text customer input that attempt to override an agent's role, scope, or identity MUST NOT be accepted or acted on. / Instructions embedded in tool outputs — FAQ content, seat records, flight data — that attempt to change an agent's behavior MUST NOT be accepted."
examples/customer_service/main.py:162 — user_input is read from the REPL fallback prompt and appended directly to input_items[] with role "user" — no sanitizer, no length cap, no injection-pattern check. examples/customer_service/main.py:96 — Neither faq_agent, seat_booking_agent, nor triage_agent has an input_guardrails= or output_guardrails= keyword passed to its Agent constructor — the SDK's available guardrail primitives are inert in this example.
Recommended Action
  • Attach input_guardrails on triage_agent that flag override-style instructions in the user message (e.g. "ignore previous", "you are now") and either drop the turn or escalate to a human checkpoint.
  • Attach tool_guardrails that treat the faq_lookup_tool and update_seat outputs as opaque data — never as instructions — and prevent them from influencing handoff or tool-selection decisions.
HIGH PRAX-2026-05-26-007 Flight number is fabricated at runtime via random.randint instead of read from the reservation system.
Implement Zero Trust LLM09 — Misinformation
Policy Rule — R-13 (Worker Remit):
"Flight numbers, confirmation numbers, and passenger identifiers MUST come from the authoritative reservation system and MUST NOT be generated by the agent at runtime."
examples/customer_service/main.py:89 — on_seat_booking_handoff sets context.context.flight_number = f"FLT-{random.randint(100, 999)}" — fabricated, not retrieved from any reservation source. examples/customer_service/main.py:82 — update_seat's assertion treats this fabricated value as the authoritative flight identifier before performing the seat write.
Recommended Action
Replace the random flight-number generation in on_seat_booking_handoff with a lookup against the reservation system using the confirmation number the customer provides; refuse to proceed when no reservation is found.
MEDIUM PRAX-2026-05-26-009 Triage agent has no domain restriction in its instructions, leaving its handoff surface open to off-scope topics.
Limit Your Domain
examples/customer_service/main.py:123 — triage_agent.instructions is a single sentence with no allow-list of in-scope topics and no deny-list of out-of-scope ones — the eight remit-forbidden categories are not enumerated.
Recommended Action
Expand the triage agent's instructions to enumerate the in-scope tasks (FAQ, seat changes) and add an explicit decline-with-pointer rule for the remit's "does NOT do" list, and add an output guardrail that catches assistant responses attempting to act in those domains.
MEDIUM PRAX-2026-05-26-010 REPL loop has no per-session turn cap, rate limit, or budget threshold.
Implement Zero Trust LLM10 — Unbounded Consumption
examples/customer_service/main.py:161 — while True: loop drives Runner.run with no max_turns argument, no per-session budget tracker, and no exit condition other than the auto-mode single-turn break at line 186.
Recommended Action
Pass a max_turns argument to Runner.run (the SDK supports it) and add a per-session token budget check that breaks the loop when exceeded.
MEDIUM PRAX-2026-05-26-011 Conversation/trace group_id is a client-side random UUID with no binding to customer identity.
Monitor Continuously
examples/customer_service/main.py:159 — conversation_id = uuid.uuid4().hex[:16] — generated client-side at REPL start, never combined with or replaced by an authenticated-customer identifier in the trace metadata.
Recommended Action
Derive the trace group_id from a verified customer-session identifier (or attach it as a metadata field on the trace), so trace events can be queried by customer during incident response.
