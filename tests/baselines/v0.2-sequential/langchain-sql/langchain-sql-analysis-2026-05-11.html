Agent Remit (as declared)
A natural-language interface to a relational database: it converts a user's question into a SQL query, runs it against a pre-configured SQLAlchemy connection, and returns a natural-language answer based on the results. The agent may use exactly four tools — sql_db_list_tables, sql_db_schema, sql_db_query_checker, and sql_db_query — and is expected to limit result sets to top_k rows (default 10), never SELECT *, run the query-checker before executing, answer "I don't know" for off-topic questions, and stay within max_iterations (default 15). DML, DDL, administrative/superuser commands, multi-statement queries, and queries outside the configured table scope are forbidden entirely — there is no approval path — and the remit explicitly states that if the agent code is the sole enforcement point for the DML/DDL prohibition, that is itself a finding.
Agent Structure (as observed)
A LangChain library component: create_sql_agent() in agent_toolkits/sql/base.py wires an LLM plus a SQLDatabaseToolkit (four tools defined in tools/sql_database/tool.py) into a langchain_classic AgentExecutor — the docstring itself flags this executor as legacy and not actively supported for new production apps. Domain and behavioral constraints live entirely in the SQL_PREFIX system prompt (agent_toolkits/sql/prompt.py); there is no code-level SQL parsing, statement-type check, row-limit injection, or table-scope check at execution time. QuerySQLDatabaseTool._run passes the LLM-generated SQL string straight to SQLDatabase.run_no_throw → _execute in utilities/sql_database.py, which runs it inside a committing engine.begin() transaction; the user's question is concatenated into the prompt as {input} with no validation, and table names, schema, sample rows, and query results are returned into the LLM context unsanitized and unlabeled.
Behavior Summary

The dominant pattern is policy declared in the system prompt with essentially zero code-level enforcement. Every behavioral constraint the remit cares about — no DML/DDL/admin statements, no SELECT *, the top_k row cap, run-the-checker-first, the configured table scope, "answer 'I don't know' when off-topic" — exists only as English in SQL_PREFIX. QuerySQLDatabaseTool._run hands the model's SQL string directly to SQLDatabase._execute, which runs it in a committing transaction with no statement inspection, no row limit, and no table-scope check; the sql_db_query_checker "gate" is itself just another LLM call that reproduces the query.

Layered on top is a clean injection chain (PRAX-002): the user's question goes into the prompt as {input} unvalidated, and table names / schema / sample rows / query results come back into the LLM context unsanitized — so either a crafted question or instruction text hidden in a row reaches the SQL-generation step, and from there into arbitrary execution against the database in one hop. The remit explicitly anticipates this and says the agent code must not be the sole enforcement point for the DML/DDL prohibition — here it is not even an enforcement point at all, so the only real backstop is an out-of-band read-only database role the agent code neither knows about nor verifies.

CRITICAL PRAX-2026-05-11-001 The DML/DDL/admin-statement prohibition has no code-level enforcement — LLM-generated SQL is executed verbatim inside a committing transaction with zero statement inspection.

Policy Rule — R-04, R-10, R-11, R-12, R-13, R-18, R-21 (Worker Remit):
"Return only read-only answers sourced from the database / Execute any DML statement: INSERT, UPDATE, DELETE, MERGE, UPSERT, REPLACE, or any statement that modifies row data / Execute any DDL statement: CREATE, DROP, ALTER, TRUNCATE, RENAME, GRANT, REVOKE... / Execute administrative or superuser commands: VACUUM, ANALYZE, COPY, LOAD DATA, ATTACH, DETACH, PRAGMA... / Execute multi-statement queries or stored-procedure invocations that could mask hidden DML/DDL / Any DML or DDL statement. tools/sql_database/tool.py:53-59 — QuerySQLDatabaseTool._run(query) calls self.db.run_no_throw(query) with no inspection of the query string utilities/sql_database.py:621-647 — run_no_throw → run → _execute; no DML/DDL/admin/multi-statement check anywhere in this path utilities/sql_database.py:475-552 — _execute wraps the model-supplied command in `with self._engine.begin() as connection:` (a transaction that commits) and runs `connection.execute(text(command), ...)` — any INSERT/UPDATE/DELETE/DROP/ALTER the model emits is committed agent_toolkits/sql/prompt.py:12 — the only DML restriction is the prose line `DO NOT make any DML statements (INSERT, UPDATE, DELETE, DROP etc.)` in SQL_PREFIX; DDL, admin commands, and multi-statement queries are not mentioned even there Absence: no SQL parser, statement-type allow-list, read-only execution mode, or per-statement policy exists in any staged file
Recommended Action
Do not rely on the system prompt or the LLM query-checker for the DML/DDL prohibition. Add a deterministic pre-execution check in QuerySQLDatabaseTool._run (or in SQLDatabase.run) that parses the statement (e.g., with sqlglot/sqlparse), rejects anything that is not a single SELECT (or other explicitly-allowlisted read statement), and rejects multi-statement input; pair it with a read-only execution path (read-only transaction / autocommit-off + rollback) so the committing engine.begin() block in _execute can never persist a write. Treat the operator's read-only DB role (R-20) as the second layer, not the only one.
