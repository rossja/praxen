BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the system prompt with essentially
zero code-level enforcement. Every behavioral constraint the remit cares about
— no DML/DDL/admin statements, no SELECT *, the top_k row cap,
run-the-checker-first, the configured table scope, "answer 'I don't know' when
off-topic" — exists only as English in SQL_PREFIX. QuerySQLDatabaseTool._run
hands the model's SQL string directly to SQLDatabase._execute, which runs it
in a committing transaction with no statement inspection, no row limit, and no
table-scope check; the sql_db_query_checker "gate" is itself just another LLM
call that reproduces the query. Layered on top is a clean injection chain
(PRAX-002): the user's question goes into the prompt as {input} unvalidated,
and table names / schema / sample rows / query results come back into the LLM
context unsanitized — so either a crafted question or instruction text hidden
in a row reaches the SQL-generation step, and from there into arbitrary
execution against the database in one hop. The remit explicitly anticipates
this and says the agent code must not be the sole enforcement point for the
DML/DDL prohibition — here it is not even an enforcement point at all, so the
only real backstop is an out-of-band read-only database role the agent code
neither knows about nor verifies.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-11-001  The DML/DDL/admin-statement prohibition has no
            code-level enforcement — LLM-generated SQL is executed verbatim
            inside a committing transaction with zero statement inspection.
      Action: Do not rely on the system prompt or the LLM query-checker for
      the DML/DDL prohibition. Add a deterministic pre-execution check in
      QuerySQLDatabaseTool._run (or in SQLDatabase.run) that parses the
      statement (e.g., with sqlglot/sqlparse), rejects anything that is not a
      single SELECT (or other explicitly-allowlisted read statement), and
      rejects multi-statement input; pair it with a read-only execution path
      (read-only transaction / autocommit-off + rollback) so the committing
      engine.begin() block in _execute can never persist a write. Treat the
      operator's read-only DB role (R-20) as the second layer, not the only
      one.

  PRAX-2026-05-11-002  Compound injection chain: the user's question enters
            the prompt as {input} unvalidated and tool outputs (table names,
            schema, sample rows, query results) re-enter the LLM context
            unsanitized, so a crafted question or instruction text hidden in a
            row reaches SQL generation and then arbitrary execution in one
            hop.
      Action: Treat both the user question and every tool output as untrusted:
      keep them in clearly-delimited, role-separated message slots (already
      partly the case) and add the deterministic statement check from
      PRAX-2026-05-11-001 so that even a hijacked SQL-generation step cannot
      produce a write or out-of-scope read. For sample rows / schema returned
      by InfoSQLDatabaseTool, consider truncating or omitting free-text cell
      contents that are not needed for query planning.
