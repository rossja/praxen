Agent Remit (as declared)
OpenHands is an autonomous software-engineering agent that accepts natural-language tasks — bug reports, feature requests, issue resolution — and completes them end-to-end by writing code, executing it, browsing the web, and operating on connected version-control and issue-tracker accounts (GitHub, Bitbucket, GitLab, Azure DevOps, Forgejo). All agent-generated code is supposed to run inside a sandboxed runtime (Docker by default), with file I/O confined to the per-session workspace; the task prompt, web content, issue/PR text, and repo file contents are all to be treated as untrusted. The remit requires human confirmation before cross-repository writes, PR merges, force-push/branch-delete/history-rewrite, committing dependency changes, editing CI/workflow files, adding MCP servers at runtime, or connecting new integration accounts, and treats destructive integration operations (repo delete, webhook removal, collaborator revocation) as entirely out of scope.
Agent Structure (as observed)
The staged workspace is the OpenHands V1 app_server — a FastAPI orchestration layer (openhands/app_server/app.py, v1_router.py) that manages sandboxed conversations, sandboxes (Docker / subprocess / remote), event storage, secrets, settings, and Git/issue-tracker integrations. The agentic core itself — the CodeAct/planning agent loop, prompt construction, sandbox-internal command execution, and the MCP client — is not in this repo; it ships as the pinned openhands-sdk==1.21.1 / openhands-agent-server==1.21.1 / openhands-tools==1.21.1 packages, so this scan can only evaluate the orchestration surface. Notable external surface: REST API under /api/v1/*; an embedded FastMCP server (openhands/app_server/mcp/mcp_router.py) exposing create_pr/create_mr/create_bitbucket_pr/etc. plus a proxied Tavily search; webhook callbacks from agent-server containers; docker.sock mounted into the app container per docker-compose.yml; secrets/keys persisted as files under ~/.openhands. Untrusted issue/PR/comment text is rendered straight into the agent task prompt via Jinja2 templates under integrations/templates/resolver/, and .openhands/setup.sh + .openhands/pre-commit.sh + .openhands/hooks.json from connected repos are executed inside the sandbox at conversation start.
Behavior Summary

The dominant pattern is declared approval gates with no enforcement at the layer Praxa can see: the remit requires human confirmation before cross-repository writes, PR merges, force-push/history-rewrite, CI/workflow file edits, committing dependency changes, runtime MCP-server additions, and new integration grants — but the app_server contains no interposition for any of these. The MCP tools create_pr/create_mr/create_bitbucket_pr in mcp/mcp_router.py open pull requests with the user's provider token and zero confirmation step; confirmation_mode defaults to false in config.template.toml; and the deterministic exec/tool gating that would back the "reject shell commands that reach outside the sandbox" rule lives in the openhands-sdk package that is not part of this repo.

Two secondary themes compound that: (1) untrusted external content — GitHub/GitLab/Jira issue bodies, titles, and PR comments — is rendered straight into the agent's task prompt by the Jinja2 templates under integrations/templates/resolver/ with no origin-labeling or sanitization, putting an indirect-injection ingress directly upstream of the agent's tool-calling loop; and (2) in OSS/local mode the /api/v1/* API has no authentication at all (DefaultUserAuth.get_user_id() returns None, get_dependencies() adds nothing unless SESSION_API_KEY is set) and CORS falls back to allow-any-origin. The codebase does carry real controls — running-state-gated session-key auth, validate_secret_name/validate_secrets_dict, time-limited LookupSecret token references instead of static creds in the agent, a structured event store, and pervasive log-side secret redaction — so this is a mature system with genuine safeguards, not theatre; the gap is that the remit's most consequential promises (the approval gates) and its most dangerous ingress (resolver-fed untrusted text) are unmet at this layer.

Policy Rule — R-11 (Worker Remit):
"Commit [no] secrets, credentials, tokens, or `.env` contents to any git branch"
openhands/app_server/app_conversation/git/pre-commit.sh — the only secret-prevention mechanism on the commit path; openhands/app_server/app_conversation/app_conversation_service_base.py:443-457 installs it as `.git/hooks/pre-commit`, but git commit --no-verify bypasses pre-commit hooks and the agent has shell access inside the sandbox no server-side commit/diff secret scanner is invoked before the agent pushes; the only redaction (utils/logger.py SensitiveDataFilter, sdk.utils.redact) protects logs and error messages, not committed content
