AutoGen Code Executor · Analysis completed 2026-05-11

Agent Remit (as declared)
The execution tier of a two-agent generator/executor pattern: it receives a code block produced by an LLM agent (or by operator application logic), runs it in a configured environment, and returns stdout, stderr, and exit status for the agent's next reasoning step. Five executor implementations are authorized — LocalCommandLineCodeExecutor (host OS, isolated/ephemeral hosts only), DockerCommandLineCodeExecutor (the standard production path), DockerJupyterCodeExecutor, JupyterCodeExecutor, and AzureContainerCodeExecutor. Every executor must confine file I/O to a configured work directory, enforce a wall-clock timeout, isolate execution from the host caller's environment and credentials, and write a per-execution audit log; production use of the local (non-containerized) executor, runtime host-volume mounts, network egress beyond an allow-list, and any raising of resource ceilings all require human approval.
Agent Structure (as observed)
Python library code under autogen-core/code_executor (the abstract CodeExecutor base, CodeBlock/CodeResult dataclasses, and the FunctionWithRequirements helper) and autogen-ext/code_executors (the five concrete executors plus shared helpers in _common.py). The local executor writes each block to a hashed file in the work dir and runs it via asyncio.create_subprocess_exec with env = os.environ.copy(); the Docker executor creates a container from python:3-slim, bind-mounts the work dir as rw at /workspace, and runs code via container.exec_run wrapped in the timeout command; the Jupyter executors drive an nbclient/kernel-gateway kernel; the Azure executor POSTs code inline to an Azure Container Apps dynamic-sessions endpoint with a bearer token. create_default_code_executor() in code_executors/__init__.py picks the Docker executor if Docker is reachable and otherwise warnings.warn(...) and returns the local executor. There is no per-execution audit log, no CPU/memory resource limits, no network policy on the Docker container, and no approval gate on local-executor use.
Behavior Summary

Across all five executor implementations the subsystem treats LLM-generated code as trusted-to-run — which is its declared job — but the host isolation the remit demands is largely absent or defaulted off. LocalCommandLineCodeExecutor runs code with the parent process's full environment (env = os.environ.copy()) and, when no virtual-env context is configured, pip installs the operator's helper-function requirements into the host interpreter (sys.executable -m pip install); the Docker path runs as root in-container with an rw bind mount of the work dir, unrestricted default network egress, no capability drops and no CPU/memory caps; and create_default_code_executor() silently downgrades to the local (host) executor with only a warnings.warn when Docker is unavailable.

The dominant pattern: containerization is the documented safety story, but the container is shipped insecurely defaulted and the fallback path has no gate — and because no executor writes a per-execution audit log, a host-level execution leaves no record. The other recurring theme is that several remit clauses requiring human approval (runtime host-volume mounts via extra_volumes, raising the timeout ceiling, network egress) are exposed as ungated constructor parameters or simply have no control surface at all.

Each executor takes a `timeout` parameter (default 60s, validated `>= 1`) and applies it: the local executor `asyncio.wait_for`s the subprocess and then `terminate()`s it on timeout, the Docker executor wraps the command in the `timeout` shell builtin, and the Jupyter/Azure executors pass it to the kernel client / HTTP client.
