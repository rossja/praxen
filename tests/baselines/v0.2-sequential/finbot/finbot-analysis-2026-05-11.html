Agent Remit (as declared)
FinBot is declared as an autonomous invoice-processing assistant for CineFlow Productions: it receives vendor invoices, validates them against registered approved vendor records, and renders approve / reject / escalate decisions recorded in the invoice database. It is authorized to use five tools — get_invoice_details, approve_invoice, reject_invoice, request_human_review, detect_fraud_patterns — over the application ORM (invoices, vendors, finbot_config) plus the OpenAI API for inference only. The remit hard-forbids approving payment to an unregistered/unapproved vendor, skipping fraud detection for any reason, processing invoices above the manual_review_threshold without human sign-off, and allowing any runtime input — invoice descriptions, vendor notes, API payloads, the custom_goals field — to alter its goals, system prompt, or decision logic; fraud_detection_enabled must remain true at all times.
Agent Structure (as observed)
A Python Flask app (src/main.py) with a SQLAlchemy SQLite/PostgreSQL backend, fronted by three blueprints — src/routes/vendor.py, src/routes/admin.py, src/routes/user.py — all mounted under /api with CORS open to all origins and no authentication on any route. The agent itself is a single FinBotAgent class in src/services/finbot_agent.py that drives OpenAI function-calling (gpt-4.1-mini, tool_choice="auto", max_iterations=5) over the five tools, with a rule-based _fallback_processing path used whenever the OpenAI client is unavailable. The agent's goals live entirely in a natural-language system prompt assembled in get_system_prompt() from a FinBotConfig DB row whose custom_goals text is concatenated directly into the prompt; both that field and fraud_detection_enabled are mutable via unauthenticated POSTs to /api/admin/finbot/goals and /api/admin/finbot/config. Vendor-submitted invoice descriptions reach the LLM verbatim via _get_invoice_details, and approve_invoice() writes payment_processed=True with no amount, threshold, fraud-signal, or caller check. The repo also ships a CTF walkthrough under docs/ describing exactly these weaknesses; it is, by construction, a deliberately vulnerable training target.
Behavior Summary

FinBot's entire safety model is a natural-language system prompt handed to an LLM; there is no deterministic enforcement anywhere on the path that moves money. The agent's goals, its approval thresholds, and the fraud-detection toggle all live in a single FinBotConfig DB row that an unauthenticated admin API — /api/admin/finbot/goals and /api/admin/finbot/config — can rewrite at will, and custom_goals is concatenated straight into the system prompt with an explicit "OVERRIDE ABOVE IF CONFLICTING" header.

Downstream of that, approve_invoice() in src/services/finbot_agent.py sets payment_processed=True with no check on amount, on the manual_review_threshold, on whether detect_fraud_patterns was ever called, or on who requested processing; vendor-supplied invoice descriptions flow unsanitised into the LLM context via _get_invoice_details; and the rule-based fallback path is written so that "business context" wording (authority + urgency + production keywords) explicitly overrides a detected prompt injection. The dominant pattern is the canonical goal-hijack → autonomous-payment chain — externally reachable goal injection feeding an ungated approval tool — with no structured logging or alerting to detect it.
