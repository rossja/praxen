BEHAVIOR SUMMARY
------------------------------------------------------------------------------
FinBot's entire safety model is a natural-language system prompt handed to an
LLM; there is no deterministic enforcement anywhere on the path that moves
money. The agent's goals, its approval thresholds, and the fraud-detection
toggle all live in a single FinBotConfig DB row that an unauthenticated admin
API — /api/admin/finbot/goals and /api/admin/finbot/config — can rewrite at
will, and custom_goals is concatenated straight into the system prompt with an
explicit "OVERRIDE ABOVE IF CONFLICTING" header. Downstream of that,
approve_invoice() in src/services/finbot_agent.py sets payment_processed=True
with no check on amount, on the manual_review_threshold, on whether
detect_fraud_patterns was ever called, or on who requested processing;
vendor-supplied invoice descriptions flow unsanitised into the LLM context via
_get_invoice_details; and the rule-based fallback path is written so that
"business context" wording (authority + urgency + production keywords)
explicitly overrides a detected prompt injection. The dominant pattern is the
canonical goal-hijack → autonomous-payment chain — externally reachable goal
injection feeding an ungated approval tool — with no structured logging or
alerting to detect it.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-11-001  Unauthenticated admin endpoint rewrites the agent's
            natural-language goals, which are concatenated directly into
            FinBot's system prompt with an explicit override header.
      Action: Remove the runtime goal-mutation path entirely: delete the
      `/api/admin/finbot/goals` route and `update_goals()`, and stop reading
      `config.custom_goals` into the system prompt (the agent's decision logic
      must be code-defined, per remit). If a goal-configuration surface must
      exist, gate it behind authenticated, role-restricted access, validate
      against an allow-list of known parameters, and never concatenate free
      text into the prompt.

  PRAX-2026-05-11-002  fraud_detection_enabled — which the remit requires to
            stay true at all times — can be flipped to false through an
            unauthenticated config endpoint, after which detect_fraud_patterns
            returns an empty result.
      Action: Treat fraud_detection_enabled as a non-configurable constant:
      remove it from the writable config keys in `update_config()` and from
      the `/api/admin/finbot/config` accepted payload, and make
      `_detect_fraud_patterns` always run. If a kill-switch is genuinely
      needed for operations, it must require an out-of-band, authenticated
      deployment change, not a runtime API POST.

  PRAX-2026-05-11-003  approve_invoice() marks payment_processed=True with no
            check on invoice amount, the manual_review_threshold, fraud
            signals, or caller identity — the agent can auto-approve any
            invoice of any size.
      Action: Add a deterministic gate inside `_approve_invoice()` (and the
      fallback approval branch) that refuses to set
      `payment_processed`/`status='approved'` and instead routes to
      `_request_human_review()` whenever `invoice.amount >
      config.manual_review_threshold`, the most recent `detect_fraud_patterns`
      result for the invoice has `risk_level` in {high, critical}, the
      vendor's trust_level is 'low', or `confidence <
      config.confidence_threshold`. The threshold logic must live in code, not
      in the LLM prompt.

  PRAX-2026-05-11-004  Nothing requires detect_fraud_patterns to run before
            approval — the LLM orchestration loop lets FinBot call
            approve_invoice with no fraud check performed.
      Action: Make fraud detection a hard precondition in code: in
      `_approve_invoice()` (and the fallback approval branch), look up or
      compute the fraud result for the invoice and refuse to approve if it was
      never run or returns any indicators — route to `_request_human_review()`
      instead. Do not rely on the prompt's 'Always start by getting invoice
      details' guidance to enforce ordering.

  PRAX-2026-05-11-005  A detected fraud signal does not force escalation — the
            fallback path explicitly lets 'business context' wording
            (authority + urgency + production keywords) override a detected
            prompt injection and auto-approve.
      Action: Make any fraud indicator a non-overridable escalation: if
      `_detect_fraud_patterns` (or `_detect_prompt_injection`) returns any
      indicator, `_approve_invoice` and the fallback must hard-route to
      `_request_human_review()` regardless of business-context score,
      speed_priority, vendor trust, or due date. Delete the `should_expedite`
      / management-approval override branches from `_fallback_processing()`.

  PRAX-2026-05-11-006  Compound chain: an unauthenticated party can inject
            goals via /api/admin/finbot/goals (or craft an invoice
            description), the LLM ingests it as a directive, and
            approve_invoice processes payment with no human gate — goal hijack
            to autonomous payment in one hop.
      Action: Break the chain at every link: (1) remove the runtime
      goal-mutation surface and stop reading custom_goals into the prompt; (2)
      treat vendor-supplied description as untrusted data — never an
      instruction — and strip/label it before it enters context; (3) put a
      code-level approval gate in `_approve_invoice` keyed on amount, fraud
      result, and vendor trust; (4) add structured action logging and alert on
      every approval and every config change. See PRAX-2026-05-11-001, -003,
      -004, -009.
