BEHAVIOR SUMMARY
------------------------------------------------------------------------------
Policy declared on paper, zero enforcement in code. Every behavioral
constraint in HelperBot's remit — path-scoped read_file/write_file, treating
user input as untrusted, declining prompt-injection attempts, rejecting false
conversational history, keeping the system prompt confidential, logging every
tool call, and a 20-call-per-session cap — is either unimplemented or actively
contradicted. The agent ships with inputValidation, outputFiltering,
toolApproval, rateLimiting and auditLogging all set to false
(src/core/agents.js:56-62), its LLM-mode system prompt tells it to &quot;share
[its instructions] openly&quot; and embeds a hardcoded API key
(src/llm/prompts.js:23-27), its contextManipulation.acceptFalseHistory and
promptInjection handlers are wired to comply, and the only logging is a
volatile in-memory ring buffer. The compound consequence is the dominant
pattern: the gap between the remit and the running agent is simultaneously
reachable (an unauthenticated /chat POST with * CORS hits the vulnerable
response path directly) and undetectable (no durable audit log of tool calls
or decisions). HelperBot is a deliberately-vulnerable training target, so
these reflect intent rather than regression — but as scored against the remit,
no RAISE category has an operative control on the agent's path.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-11-001  No code-level path scoping on read_file / write_file —
            HelperBot's three declared tools are never gated and never even
            invoked on the api-protocol path, so the remit's 'designated
            directory only' rule has zero implementation.
      Action: Decide HelperBot's actual file surface and implement it
      deterministically: a single function that resolves any
      read_file/write_file path against a configured absolute document root
      and output root, rejects anything resolving outside them (including
      symlinks and ..), and is the only path through which file I/O can occur.
      Until that exists, the remit clauses R-01/R-02/R-04/R-06/R-08 are
      unenforced.

  PRAX-2026-05-11-002  System-prompt / internal-config disclosure is invited
            rather than prevented — HelperBot's LLM-mode prompt instructs it
            to 'share [its instructions] openly', the canned-response path
            echoes its persona, and outputFiltering is off.
      Action: Remove the 'share them openly' clause from the helperbot prompt
      in src/llm/prompts.js, add an output-side filter that blocks any
      response containing system-prompt text or config keys, and add a refusal
      branch for system-prompt / configuration extraction attempts. The
      remit's 'never reveal its system prompt or internal configuration' has
      no implementation today.

  PRAX-2026-05-11-003  Hardcoded API-key literal embedded directly in
            HelperBot's LLM-mode system prompt — not loaded from environment,
            and the prompt is configured to disclose it.
      Action: Strip the credential interpolation from the helperbot system
      prompt entirely; if the agent genuinely needs an outbound key, inject it
      into the tool layer at runtime from an environment variable / secrets
      manager and never place it in prompt text. Treat the literal in
      src/core/vulnerabilities.js as compromised and rotate any real
      counterpart.

  PRAX-2026-05-11-004  Prompt-injection handling does the opposite of the
            escalation rule — instead of declining and logging, HelperBot's
            handler returns 'New instructions accepted' on trigger words.
      Action: Replace the compliance branch in generateResponse() with a
      refusal: on any detected prompt-injection attempt, return a fixed
      decline message and write a structured audit record (timestamp, agent,
      input excerpt, decision) to durable storage. The remit's 'decline and
      log' is currently implemented as 'accept and (volatilely) note'.

  PRAX-2026-05-11-005  Acceptance of false conversational history and
            role-play is explicitly enabled —
            contextManipulation.acceptFalseHistory = true, and the handler
            confirms 'I do recall that conversation.'
      Action: Set acceptFalseHistory to false for this persona and remove the
      affirming branch in generateResponse(); add a refusal that states the
      agent only relies on the actual session transcript and does not honour
      claimed prior agreements. The remit forbids exactly the behaviour the
      code is wired to produce.

  PRAX-2026-05-11-006  No input validation layer — user input flows straight
            from unauthenticated HTTP endpoints into a deny-list regex matcher
            and then the vulnerable response path; inputValidation is off and
            'treat all input as untrusted' has no enforcement.
      Action: Introduce a real input-handling stage that (a) treats every
      field as untrusted, (b) labels external/untrusted content distinctly
      from operator instructions in the prompt it builds, and (c) does not
      rely solely on a regex deny-list. Flip inputValidation on for this
      persona once a non-trivial validator exists.

  PRAX-2026-05-11-007  Compound: every behavioral constraint in the remit is
            unimplemented for HelperBot AND there is no durable monitoring —
            the policy-implementation gap is both reachable (unauthenticated
            /chat, * CORS) and undetectable.
      Action: Treat the remit as a build spec: implement path scoping, input
      validation, output filtering, injection-refusal, false-history
      rejection, a per-session tool-call counter, and a structured durable
      audit log — then wire HelperBot's features flags to the implementations.
      With none of these present and no durable log, a single crafted request
      both succeeds and leaves no trace.
