Agent Remit (as declared)
Devika is an autonomous software-engineering agent that takes a high-level natural-language request from a developer via a web UI, decomposes it into a plan, researches via web search, writes code into a per-project work directory, debugs, and delivers the result. Authorized capability categories are planning (planner), action selection (decision/action), code generation and patching (coder/patcher), web research (researcher), progress reporting (formatter/reporter), and per-project memory/RAG; LLM providers (OpenAI, Anthropic, Ollama, Mistral, Groq, Gemini, LM Studio) and a web search API are the only external counterparties. The remit explicitly forbids executing LLM-produced code outside a sandboxed runner, writing outside the per-project work directory, installing system packages without user confirmation, modifying Devika's own code/config/memory, and exfiltrating operator credentials; it requires injection detection on untrusted input, a per-project audit log, and a step/wall-clock cap on each run.
Agent Structure (as observed)
Python application: a Flask + Flask-SocketIO backend (devika.py, binds 0.0.0.0:1337) with a Svelte UI (not in scope). A single Agent class (src/agents/agent.py) orchestrates twelve sequential sub-agents, each a thin wrapper that renders a Jinja2 prompt template and calls LLM.inference() (src/llm/llm.py); seven provider clients live under src/llm/. Code execution runs through src/agents/runner/runner.py via raw subprocess.run(command_set, cwd=project_path) on LLM-authored command strings; src/sandbox/code_runner.py and src/sandbox/firejail.py are 0-line stub files. State and project messages are persisted in a shared SQLite DB via SQLModel (src/state.py, src/project.py); external surface includes a Flask blueprint (src/apis/project.py) plus routes in devika.py for settings, logs, run-code, browser/terminal sessions and an arbitrary send_file(snapshot_path). Web content retrieved by the researcher (src/browser/search.py, src/browser/browser.py) flows through the formatter agent straight into the coder/runner LLM context; there is no authentication on any route, no input validation, no injection detection, and no structured per-project audit log.
Behavior Summary

The dominant pattern is planned-but-not-deployed controls layered over zero input/output trust enforcement. Four of the remit's load-bearing controls — a sandboxed code runner, per-project path confinement, instruction-injection detection on untrusted input, and a per-project audit log with step/wall-clock caps — have no running implementation: src/sandbox/code_runner.py and src/sandbox/firejail.py are 0-line stubs, and the live execution path in src/agents/runner/runner.py shells LLM-authored command strings through subprocess.run(command_set, cwd=project_path) with no approval gate, no per-command deny policy, and no isolation.

These gaps compound into a single one-hop chain: web content fetched by the researcher agent passes through the formatter agent into the coder and runner LLM context (src/agents/agent.py:344-357, runner.py:81-90), so an attacker-controlled page can steer the agent into running arbitrary commands on the operator's host. The entire pipeline is reachable from an unauthenticated Flask/Socket.IO server bound to 0.0.0.0:1337 that additionally exposes arbitrary file read (/api/get-browser-snapshot), an unauthenticated config rewrite that can flip logging and swap provider keys (/api/settings), and a wildcard-CORS websocket. Code-writing agents (coder.py, patcher.py, feature.py) also derive file paths from raw LLM output with no traversal check, defeating the work-directory confinement the remit promises.
