Sweep · Analysis completed 2026-05-11

Agent Remit (as declared)
Sweep is an autonomous GitHub issue-to-code agent: installed as a GitHub App on customer repos, it receives webhook events (issue opened, comment added, PR review), reads repository code and issue/comment text, plans a code change, creates a sweep/-prefixed branch, commits to it, and opens a pull request against the default branch for human review. It may reply to the originating issue and call an LLM provider for inference, and nothing else — it must never merge its own PRs, never execute arbitrary shell commands or run generated code, never communicate outside GitHub plus the LLM provider, and never modify CI/CD or security-sensitive files (.github/workflows/*, Dockerfile, docker-compose.yml, SECURITY.md, CODEOWNERS, etc.) without an explicit issue instruction and a prominent PR-description flag. Every character of every issue, comment, file, and diff is to be treated as untrusted input and run through instruction-injection detection before it becomes LLM context.
Agent Structure (as observed)
Python 3.10 FastAPI application: a single webhook receiver (sweepai/api.py, out of staged scope) dispatches GitHub events to handlers that drive an LLM agent loop. The modify loop lives in sweepai/agents/modify.py + sweepai/agents/modify_utils.py with three XML tools — make_change, create_file, submit_task — applied to an in-memory modify_files_dict; sweepai/agents/question_answerer.py and sweepai/agents/search_agent.py add ripgrep/view_file tools. LLM calls go through sweepai/core/chat.py (Anthropic / OpenAI / Bedrock), prompts are assembled by f-string interpolation in sweepai/core/prompts.py, and config is centralized in sweepai/config/server.py (env-driven) and sweepai/config/client.py (per-repo sweep.yaml). External surface: the FastAPI app exposes / (HMAC-verified), /jira (not verified), /health, and several GET endpoints; subprocess calls run rg and github-linguist with shell=True in question_answerer.py, context_pruning.py, dynamic_context_bot.py, and config/client.py; sweepai/core/external_searcher.py fetches arbitrary URLs found in issue text; container runs as root with the repo (including .env) bind-mounted per docker-compose.yml.
Behavior Summary

The dominant pattern is policy declared in the remit, zero code-level enforcement of it, stacked on a fail-open trust gate. The remit's load-bearing controls — treat all issue/comment/file/diff text as untrusted and run instruction-injection detection before it reaches an LLM, never execute shell, never touch CI/CD or security-sensitive files without explicit issue authorization, scope changes to the named issue — have no counterpart in the code. Issue titles and bodies are f-string-interpolated directly into prompts (sweepai/core/prompts.py); no module mentions injection detection, sanitization, or quote-wrapping of instruction-like content. Several agent tools shell out with shell=True on model- and issue-influenced arguments (sweepai/agents/question_answerer.py:270, sweepai/core/context_pruning.py:186, sweepai/core/dynamic_context_bot.py:44, sweepai/config/client.py:338). The make_change/create_file tools in sweepai/agents/modify_utils.py write to any path the model names with no CI/CD blocklist, no security-sensitive-file gate, and no scope check against the originating issue. And sweepai/core/external_searcher.py fetches arbitrary URLs scraped out of issue text straight into LLM context — a channel the remit explicitly forbids.

Compounding all of it: webhook HMAC verification fails open — verify_signature() returns True when WEBHOOK_SECRET is unset (sweepai/utils/hash.py:22, declared at sweepai/config/server.py:204), and the /jira endpoint has no verification at all — so the untrusted-content → prompt-injection → shell-exec / arbitrary-file-write chain is reachable by an unauthenticated caller forging a webhook.

Webhook HMAC verification implemented (when the secret is set)

The primary `/` webhook route is gated by a FastAPI dependency that recomputes an HMAC-SHA256 over the raw body and uses constant-time comparison; when `WEBHOOK_SECRET` is configured this is a correct GitHub-style signature check (the fail-open default when it is unset is filed as PRAX-2026-05-11-012).

Sweep-authored-branch guard on branch deletion and a name-prefixed branch convention

`safe_delete_sweep_branch` will only delete a branch whose PR commits are all authored by the Sweep bot identity and whose name starts with `sweep`, and Sweep consistently creates `sweep/*`-prefixed branches — a partial structural guard against touching non-Sweep branches (tightening needed per PRAX-2026-05-11-009).
