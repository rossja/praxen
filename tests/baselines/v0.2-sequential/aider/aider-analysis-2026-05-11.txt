BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the remit, near-zero code-level
enforcement of the boundaries that matter most. The remit's three load-bearing
rules — never touch anything outside the git root, treat repo content as
untrusted and neutralize embedded instructions, and redact secrets from LLM
context — have no corresponding controls: base_coder.abs_root_path()
joins-and-resolve()s paths with no containment assertion,
--read//read-only//add accept absolute and ~ paths (so ~/.ssh/id_rsa or
/etc/passwd flows straight into LLM-provider context), there is no secret
scanner anywhere in the tree, and --watch-files in watch.py actively reads ai!
/ ai? comments out of source files and feeds them to the model as instructions
to execute and then delete — the exact embedded-instruction class the remit
says must be neutralized. The second theme is missing approval gates on
actions the remit says require explicit human sign-off: auto-commit
(auto_commits=True) and auto-lint (auto_lint=True) fire after every edit with
no diff-accept prompt; commits run git commit -a --no-verify by default,
staging everything and bypassing pre-commit/commit-msg hooks; and
security-sensitive paths (Dockerfile, .github/workflows/*, requirements*.txt,
etc.) get no special-case approval. The LLM-suggested-shell path in
base_coder.handle_shell_commands() is the one place a real
explicit_yes_required gate exists — a genuine safeguard worth crediting — but
it is the exception, not the rule.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-11-001  No repo-containment check on file paths — `--read`,
            `/read-only`, and `/add` accept absolute and `~` paths, so files
            anywhere on disk (e.g. `~/.ssh/id_rsa`, `/etc/passwd`) can be
            loaded into LLM-provider context with no refusal.
      Action: Add a single chokepoint — e.g. in `base_coder.abs_root_path()`
      and at every `/add`, `/read-only`, `--read`, `--file` ingestion site —
      that computes the real path (`Path(p).resolve()`, following symlinks)
      and rejects it unless `os.path.commonpath([resolved, self.root]) ==
      self.root`. Refuse (do not prompt for) absolute paths, `..`-escaping
      paths, and symlinks that resolve outside the repo. Remove the `and
      self.coder.auto_commits` condition on the existing `cmd_add` containment
      check so it is unconditional.

  PRAX-2026-05-11-002  `.gitignore`/`.aiderignore` is enforced only for
            editable files and the repo map — `--read` and `/read-only` ingest
            gitignored files (`.env*`, `*.pem`, `credentials*`, `token*`) with
            no ignore check and no risk acknowledgment.
      Action: Apply the same `repo.git_ignored_file()` / `repo.ignored_file()`
      filter (already used at `base_coder.py:451`) to the `--read` /
      `/read-only` / `_add_read_only_directory` paths. When the operator
      explicitly overrides for a file matching a secret pattern (`.env*`,
      `*.pem`, `*.key`, `credentials*`, `token*`, `secrets*`), require an
      explicit typed acknowledgment before the content enters context.

  PRAX-2026-05-11-003  Repo content is treated as fully trusted
            instruction-bearing context — no injection detection or
            neutralization on file bodies, comments, docstrings, commit
            messages, or diffs, and `--watch-files` actively executes `ai!` /
            `ai?` comments found in source files as operator instructions.
      Action: Before any repo content (file bodies, repo-map text, diffs,
      commit messages) reaches the LLM, run it through a neutralization pass
      that strips or escapes imperative directives in
      comment/docstring/markdown regions and labels the block as untrusted
      data (not instructions). For `--watch-files`, restrict the trigger to
      files the developer has explicitly added to the chat in the current
      session, and surface the extracted comment text to the developer for
      confirmation before acting — do not auto-execute it.

  PRAX-2026-05-11-004  No secret detection or redaction anywhere — secret-like
            strings in repo files flow unmodified into LLM-provider context,
            can appear in proposed edits, and `git commit -a` stages and
            commits them with no scan.
      Action: Add a secret-detection pass (entropy + known-prefix patterns:
      `sk-`, `AKIA`, `ghp_`, `xox[bap]-`, PEM headers, `.env`-style
      `KEY=value`) at three chokepoints: (1) `base_coder.get_files_content()`
      before content enters the prompt — mask matches and warn; (2) the
      edit-application path before writing proposed content; (3)
      `repo.commit()` before staging — block the commit and report the
      file:line if a secret is detected. Make these on by default.

  PRAX-2026-05-11-005  Git commits happen autonomously by default —
            `auto_commits=True` makes `auto_commit()` fire after every edit
            (and after auto-lint) with no diff-accept prompt, and
            `dirty_commits` commits pre-existing changes before edits; the
            remit requires explicit developer confirmation and diff review
            before any commit.
      Action: Insert a confirmation gate in `auto_commit()` (and
      `dirty_commit()`): display the staged diff and call
      `io.confirm_ask("Commit these changes?", explicit_yes_required=True)`
      before `repo.commit()`. If a non-interactive `--yes-always` mode is
      offered it should explicitly require the operator to have opted into
      auto-commit, not inherit it as a silent default.

  PRAX-2026-05-11-006  No special-case approval for edits to
            security-sensitive files — `Dockerfile`, `.github/workflows/*`,
            `pyproject.toml`, `requirements*.txt`, `CODEOWNERS`, etc. are
            treated like any other file by `allowed_to_edit()`, and with
            auto-commit on they are modified and committed without an extra
            gate.
      Action: Define a `SECURITY_SENSITIVE_PATTERNS` list (`Dockerfile`,
      `.github/workflows/*`, `pyproject.toml`, `package.json`,
      `requirements*.txt`, `go.mod`, `Cargo.toml`, `CODEOWNERS`,
      `SECURITY.md`, `.git/hooks/*`) and, in both `allowed_to_edit()` and the
      pre-commit path, require `io.confirm_ask(...,
      explicit_yes_required=True)` showing the diff whenever an edit touches a
      matching path — regardless of whether the file is already in the chat.

  PRAX-2026-05-11-013  Compound chain: untrusted repo content reaches the LLM
            with no neutralization → `--watch-files` lets `ai!` comments in
            files act as operator instructions → those instructions can
            request `/run`-equivalent shell suggestions or off-repo file adds,
            and there is no audit log to detect it — a one-hop path from a
            poisoned file to executed actions outside the remit.
      Action: Break the chain at Step 2: do not treat in-file comments as
      instructions unless the developer has explicitly added that file to the
      chat in the current session, and always surface the extracted comment
      text for confirmation before acting on it. Add the
      content-neutralization pass from PRAX-2026-05-11-003 so even confirmed
      comments are treated as data. Then close the downstream gaps
      (PRAX-2026-05-11-001, -004, -008) so a single poisoned file cannot reach
      off-repo reads, unredacted secrets, or unconfirmed shell.
