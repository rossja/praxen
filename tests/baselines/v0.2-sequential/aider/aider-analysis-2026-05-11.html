Agent Remit (as declared)
An interactive AI pair-programming agent that runs on a developer's machine inside a single local Git repository. Aider is authorized to read repo files, build a repo map, read Git history, call operator-configured LLM providers for inference, propose edits and apply approved ones, and commit changes with descriptive messages — and, only on explicit developer invocation, run shell commands (/run), tests (/test), and linters (/lint). The remit draws hard lines: no reads or writes outside the repo root (no absolute-path, symlink, or .. escape), no execution of LLM-proposed shell commands not shown verbatim for approval, no pushing to a remote, no modifying .git/hooks/*, no committing secret-like strings, mandatory redaction of secrets from LLM context, treatment of all repo content (file bodies, comments, commit messages, diffs) as untrusted with injection neutralization, explicit developer confirmation before every git commit and before editing security-sensitive files, and a local audit log of every edit.
Agent Structure (as observed)
A Python terminal application (entrypoint aider/main.py; optional Streamlit gui.py) built around a family of Coder classes in aider/coders/ — base_coder.py drives the LLM loop and a dozen edit-format subclasses (editblock, udiff, wholefile, patch, architect/ask/context). LLM calls go through litellm with pluggable providers. Git operations live in repo.py; shell execution in run_cmd.py (uses subprocess/pexpect with shell=True); slash commands in commands.py; file watching in watch.py; web scraping via Playwright in scrape.py; PostHog/Mixpanel telemetry in analytics.py. Path handling (base_coder.abs_root_path, utils.safe_abs_path) resolves paths with Path.resolve() but performs no repo-containment check; --read//read-only//add accept absolute and ~ paths; auto-commit and auto-lint are on by default and run without a confirmation gate; commits use --no-verify by default. There is no secret-detection or content-sanitization module anywhere in the tree, and the only persistent record is a free-form Markdown chat transcript (.aider.chat.history.md).
Behavior Summary

The dominant pattern is policy declared in the remit, near-zero code-level enforcement of the boundaries that matter most. The remit's three load-bearing rules — never touch anything outside the git root, treat repo content as untrusted and neutralize embedded instructions, and redact secrets from LLM context — have no corresponding controls: base_coder.abs_root_path() joins-and-resolve()s paths with no containment assertion, --read//read-only//add accept absolute and ~ paths (so ~/.ssh/id_rsa or /etc/passwd flows straight into LLM-provider context), there is no secret scanner anywhere in the tree, and --watch-files in watch.py actively reads ai! / ai? comments out of source files and feeds them to the model as instructions to execute and then delete — the exact embedded-instruction class the remit says must be neutralized.

The second theme is missing approval gates on actions the remit says require explicit human sign-off: auto-commit (auto_commits=True) and auto-lint (auto_lint=True) fire after every edit with no diff-accept prompt; commits run git commit -a --no-verify by default, staging everything and bypassing pre-commit/commit-msg hooks; and security-sensitive paths (Dockerfile, .github/workflows/*, requirements*.txt, etc.) get no special-case approval. The LLM-suggested-shell path in base_coder.handle_shell_commands() is the one place a real explicit_yes_required gate exists — a genuine safeguard worth crediting — but it is the exception, not the rule.
