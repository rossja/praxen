OpenAI Customer Service Agent · Analysis completed 2026-05-11

Agent Remit (as declared)
A multi-agent airline customer-service assistant built on the OpenAI Agents SDK: a Triage Agent routes each customer request via handoff to an FAQ Agent (answers policy questions using faq_lookup_tool only, no general knowledge) or a Seat Booking Agent (updates seat assignments on existing confirmed reservations via update_seat). Authorized data is the authenticated customer's reservation records, the static FAQ dataset, and the read-only flight schedule; the only output is a natural-language reply in the current session — no email, SMS, webhooks, exec, filesystem, or persistent cross-session memory. The remit requires identity verification (passenger name + confirmation number) before any reservation-mutating action, verification of the confirmation number against the authoritative reservation record, a durable audit log of every seat change, and human approval for fare-affecting changes, special-handling reservations, and any request where identity cannot be verified.
Agent Structure (as observed)
Python application using the OpenAI Agents SDK (openai-agents 0.17.1, pinned with bounded ranges in pyproject.toml plus a uv.lock). The agent itself is the ~190-line examples/customer_service/main.py: three Agent instances with instruction strings prefixed by RECOMMENDED_PROMPT_PREFIX, two @function_tool implementations (faq_lookup_tool — deterministic keyword lookup returning canned strings or "I don't know"; update_seat — writes confirmation_number/seat_number straight into the run context), and an on_seat_booking_handoff hook that sets flight_number to f"FLT-{random.randint(100, 999)}". Runner.run is invoked in a loop wrapped by with trace("Customer service", ...). The SDK package shipped alongside it (src/agents/) provides InputGuardrail/OutputGuardrail/ToolInputGuardrail, per-tool needs_approval with RunState.approve()/reject(), handoff input_filter, strict-mode tool/handoff JSON schemas, a DEFAULT_MAX_TURNS loop bound, and built-in tracing — but the example wires in none of the guardrails, no approvals, and no input filters; no reservation database, FAQ dataset file, or audit-log sink is present in the workspace at all.
Behavior Summary

The dominant pattern is framework offers safe primitives, the example uses none of them, and every behavioral guarantee in the remit lives only in LLM instruction text. The OpenAI Agents SDK ships InputGuardrail/OutputGuardrail/ToolInputGuardrail, per-tool needs_approval with RunState.approve()/reject(), handoff input_filter, and strict-schema tool arguments — but examples/customer_service/main.py wires in zero guardrails, zero approval gates, and no input filters. The remit's hard requirements (verify passenger identity before any reservation mutation, verify the confirmation number against the authoritative record, log every seat change to a durable audit trail, escalate to human approval when identity is unverifiable) have no code-level enforcement: update_seat accepts whatever confirmation_number and new_seat the model produces and writes them into the run context with no lookup, no ownership check, no seat-map check, and no logging, and on_seat_booking_handoff manufactures a flight_number with random.randint() instead of reading the reservation system of record — a direct violation of the "do not assign flight/confirmation/passenger identifiers at runtime" clause.

The compound risk to lead with: a successful confused-deputy / IDOR seat change against another customer's reservation would also leave no audit trail, because the remit-mandated seat-change log is absent and the only record is generic SDK execution tracing that does not capture old seat, customer identity, or confirmation number — so the gap is both exploitable and undetectable.
