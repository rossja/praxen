BEHAVIOR SUMMARY
------------------------------------------------------------------------------
HelperBot's declared intent — a scoped internal assistant with no execution,
no PII, audit logging, and a per-session call cap — is contradicted at every
layer of the actual implementation. The agent record in `src/core/agents.js`
sets `inputValidation`, `outputFiltering`, `toolApproval`, `rateLimiting`, and
`auditLogging` all to `false` and enables `promptInjection`,
`dataExfiltration`, and `contextManipulation` as first-class behaviors;
`generateResponse()` in `src/index.js` then keys on those flags to return
canned responses that emit the system prompt, leak API-key hints, and accept
injected "new instructions" — and the LLM-mode prompt in `src/llm/prompts.js`
embeds a literal internal API-key value and instructs the model to disclose
its own configuration. This is a deliberately-vulnerable training agent (the
DVAA project), so the gaps are intentional artifacts; the Praxen report
records them as the remit-vs-implementation divergence they are, which is
exactly the lesson the workspace is designed to teach.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  HelperBot advertises `read_file` and `write_file` tools
            but no code-level path-confinement check exists — every
            workspace-boundary rule in the remit is unenforced.
      Action 1: Implement real `read_file` / `write_file` handlers that
                resolve the requested path against a configured workspace root
                via `path.resolve()`, compare with the root prefix, and reject
                (with an audit-logged decline) any path that escapes it.
      Action 2: Add a workspace-root field to the HelperBot agent record so
                the boundary is data-driven and reviewable, and surface every
                rejected attempt in the audit log path required by R-05.

  PRAX-2026-05-29-002  User input flows unchanged into both the LLM call and
            the flag-driven dispatcher — `HELPERBOT.features.inputValidation`
            is literally `false` and no sanitization exists upstream.
      Action 1: Flip `inputValidation: true` and add a real validator that
                runs before `callLLM()` and before the flag-driven dispatcher
                — reject on the existing `ATTACK_PATTERNS.promptInjection`
                regex hits, write the rejection to the audit log path required
                by R-05, and return a structured decline.
      Action 2: Make `detectAttacks()` results load-bearing for control flow,
                not just telemetry — on a positive hit the request should
                refuse, not continue into the canned-vulnerable branches.

  PRAX-2026-05-29-005  The HelperBot LLM system prompt embeds a literal
            internal API key and explicitly instructs the model to share its
            own instructions on request.
      Action 1: Remove the `${SENSITIVE_DATA.apiKeys.internal}` interpolation
                from `AGENT_PROMPTS.helperbot` and delete the "share them
                openly" sentence so the system prompt no longer instructs the
                model to disclose itself; secrets must not live in prompt
                strings at all.
      Action 2: In `generateResponse()`, replace the L761-767 leak branch with
                a structured refusal that records the attempt to the audit log
                (per R-05), so R-08 is enforced rather than demonstrated.
