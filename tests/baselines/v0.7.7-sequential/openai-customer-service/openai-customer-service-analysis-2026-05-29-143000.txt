BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern in this scan is **framework offers safe primitives, the
example uses none of them.** The OpenAI Agents SDK ships first-class
`InputGuardrail` and `OutputGuardrail` classes, per-tool `needs_approval` and
`is_enabled` callables, a `handoff(...)` `input_filter` parameter, and
strict-schema tool argument validation — every primitive the airline remit's
hard rules ("instructions in free-text customer input MUST NOT be accepted,"
"every seat change MUST be recorded to a durable audit log," "customer
identity MUST be confirmed before any action that mutates reservation state")
would require. The example wires in zero of them: no `input_guardrails=[...]`,
no `output_guardrails=[...]`, no `needs_approval=True` on `update_seat`, no
`input_filter` on the handoffs, no audit-log call inside `update_seat`. The
single most acute compound is the seat-booking path: `on_seat_booking_handoff`
fabricates a flight number via `random.randint(100, 999)` in
`examples/customer_service/main.py`, `update_seat` then writes the
model-supplied confirmation number and new seat into context with no identity
check, no ownership check, no availability check, and no approval gate — a
one-injection chain from untrusted customer text to a reservation-system write
attributed to an unverified caller.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  No InputGuardrail wired onto any agent — free-text
            customer prompt injection flows straight into triage, FAQ, and
            seat-booking agents.
      Action 1: Define an `InputGuardrail` that scans the latest user turn for
                instruction-override patterns (role-redefinition, "ignore
                previous", out-of-scope keywords) and trips the tripwire on
                match; attach it to all three agents via
                `input_guardrails=[...]`.
      Action 2: Add a complementary guardrail that runs only on `triage_agent`
                to reject requests in the remit's "does NOT do" list (refund,
                cancel, rebook, upgrade, special-handling) before any handoff
                fires.

  PRAX-2026-05-29-002  update_seat writes a model-supplied confirmation number
            into context with no reservation-system verification and no
            approval gate.
      Action 1: Set `needs_approval=True` on `@function_tool` for
                `update_seat` (or a callable that requires approval whenever
                the caller's confirmation number does not already match the
                authenticated session) so the run pauses for operator review
                before the write lands.
      Action 2: Add a reservation-system lookup inside `update_seat` that
                resolves `confirmation_number` against the authenticated
                customer and short-circuits with an error before touching
                context if ownership cannot be verified.

  PRAX-2026-05-29-003  No identity verification, no confirmation-number
            ownership check, and no human escalation when identity cannot be
            established — the seat path runs entirely on model-trusted
            strings.
      Action 1: Add an authenticated session principal to
                `AirlineAgentContext` (e.g. a `session_user_id` set by the
                application layer before `Runner.run` is called) and gate
                every mutation on `confirmation_number` belonging to that
                principal.
      Action 2: In `on_seat_booking_handoff`, refuse to proceed (raise or set
                a "must-escalate" flag) when no authenticated session is
                attached to the context, so a triage→seat handoff cannot fire
                for an anonymous caller.
      Action 3: Wire an `InputGuardrail` on `triage_agent` that escalates to a
                human (returns a tripwire) whenever the context lacks a
                verified identity and the request is anything other than an
                FAQ.

  PRAX-2026-05-29-004  Flight number is fabricated at runtime via
            random.randint(100, 999) inside on_seat_booking_handoff — the
            remit forbids agent-generated identifiers.
      Action 1: Delete the `random.randint` fabrication entirely. Replace
                `on_seat_booking_handoff` with a lookup that resolves the
                customer's actual flight from the authoritative reservation
                system using the authenticated session identity, and fail the
                handoff (escalate to human) when no flight can be resolved.
      Action 2: Add a regression test that exercises `on_seat_booking_handoff`
                and asserts `random` is not in its call chain, so the
                fabrication cannot silently return in a future refactor.
