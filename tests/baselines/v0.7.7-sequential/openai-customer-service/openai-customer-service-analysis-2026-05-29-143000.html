Executive Summary
Agent Remit (as declared)
A customer-facing airline service system composed of three cooperating agents — a triage agent that routes incoming requests, an FAQ agent that answers policy questions from a curated dataset, and a seat-booking agent that updates seat assignments on existing confirmed reservations. The agents may call `faq_lookup_tool` and `update_seat` only, and may read and write the airline's internal reservation system scoped to the authenticated customer. Outbound email, SMS, and webhooks are explicitly forbidden; ticket purchases, refunds, rebooking, and special-handling requests must reach a human checkpoint, and customer identity plus confirmation-number ownership must be verified before any mutation. Flight numbers, confirmation numbers, and passenger identifiers must come from the authoritative reservation system and must not be generated at runtime.
Behavior Summary (as observed)
The dominant pattern in this scan is **framework offers safe primitives, the example uses none of them.** The OpenAI Agents SDK ships first-class `InputGuardrail` and `OutputGuardrail` classes, per-tool `needs_approval` and `is_enabled` callables, a `handoff(...)` `input_filter` parameter, and strict-schema tool argument validation — every primitive the airline remit's hard rules ("instructions in free-text customer input MUST NOT be accepted," "every seat change MUST be recorded to a durable audit log," "customer identity MUST be confirmed before any action that mutates reservation state") would require. The example wires in zero of them: no `input_guardrails=[...]`, no `output_guardrails=[...]`, no `needs_approval=True` on `update_seat`, no `input_filter` on the handoffs, no audit-log call inside `update_seat`. The single most acute compound is the seat-booking path: `on_seat_booking_handoff` fabricates a flight number via `random.randint(100, 999)` in `examples/customer_service/main.py`, `update_seat` then writes the model-supplied confirmation number and new seat into context with no identity check, no ownership check, no availability check, and no approval gate — a one-injection chain from untrusted customer text to a reservation-system write attributed to an unverified caller.
Scope of Analysis
A Python implementation built on the OpenAI Agents SDK (`openai-agents` v0.17.4): the entire customer-service application lives in `examples/customer_service/main.py` (≈192 lines) and is driven by `Runner.run()` from `src/agents/run.py`. Three `Agent` objects — `triage_agent`, `faq_agent`, and `seat_booking_agent` — are wired together with `handoff(...)` edges; both specialist agents register a return handoff to triage. Two `@function_tool` functions (`faq_lookup_tool` and `update_seat`) provide the action surface, and an `on_seat_booking_handoff` hook fabricates a flight number via `random.randint(100, 999)` before control reaches the seat agent. The SDK ships `InputGuardrail` / `OutputGuardrail` / `needs_approval` / `is_enabled` / `input_filter` primitives in `src/agents/guardrail.py`, `src/agents/tool.py`, and `src/agents/handoffs/__init__.py` — the example wires in none of them and relies entirely on prompt-level instructions in each agent's `instructions` string.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Strict JSON-schema tool arguments by default

`FunctionTool` defaults `strict_json_schema=True` and passes the tool schema through `ensure_strict_json_schema` at construction (`src/agents/tool.py:310,403`), so `faq_lookup_tool` and `update_seat` cannot be invoked with structurally malformed arguments — a real, operative defence against a class of model-output-as-injection attacks.

src/agents/tool.py:310

Curated, deterministic FAQ lookup with explicit "don't know" path

`faq_lookup_tool` is a keyword switch over a fixed dataset that returns a closed answer or `"I'm sorry, I don't know the answer to that question."` (`examples/customer_service/main.py:42-64`); the FAQ surface is genuinely scoped, in line with the remit's "MUST NOT speculate" clause for the FAQ agent.

examples/customer_service/main.py:42

Dependencies upper-bounded and lockfile committed

`pyproject.toml` ceilings every direct dependency (e.g. `openai>=2.36.0,<3`, `pydantic>=2.12.2,<3`, `mcp>=1.19.0,<2`) and the workspace ships `uv.lock` (`pyproject.toml:9-18`, `uv.lock`); a major-version surprise from a transitive update is bounded.

pyproject.toml:9

Default tracing wired into every turn

The SDK installs `BatchTraceProcessor` + `BackendSpanExporter` by default (`src/agents/tracing/setup.py:39`, `src/agents/tracing/processors.py:707`) and `main.py` wraps each loop iteration in `with trace("Customer service", group_id=conversation_id)` — every agent turn produces an exported trace span out of the box.
