Executive Summary
Agent Remit (as declared)
OpenHands is an autonomous software-engineering agent that accepts natural-language tasks and completes them end-to-end by writing code, executing it, browsing the web, and interacting with version control. Agent-generated code must run only inside a sandboxed runtime, file I/O must be confined to a per-session workspace, and untrusted inputs — including web pages, issue text, repo file contents, and micro-agent or memory content — must be treated as prompt-injection vectors. High-impact actions such as cross-repo writes, PR merges, force-push, dependency changes, CI/CD edits, new MCP servers, and new integration connections each require user confirmation, and every action must be recorded to a durable session record.
Behavior Summary (as observed)
The dominant pattern in this control plane is policy declared in the remit and at the runtime boundary, almost no code-level enforcement in the OSS default: every v1 router calls get_dependencies(), but that helper returns an empty list unless an operator sets SESSION_API_KEY or runs in SAAS mode, so a default docker compose up deployment exposes /api/v1/sandboxes, /api/v1/secrets, /api/v1/settings, /api/v1/git, /api/v1/skills, and the /mcp tool plane to any client that can reach port 3000 — and LocalhostCORSMiddleware logs a warning then allows the request anyway when no origins are configured.

Two compound chains amplify this: the Process sandbox back-end spawns the agent-server as a same-user subprocess inheriting os.environ.copy(), defeating the remit's "MUST run inside the sandboxed runtime / MUST NOT run on the host" clause whenever RUNTIME=local or RUNTIME=process is selected; and the Docker compose deployment bind-mounts the host Docker socket into the OpenHands container, so any code path that gains control of that container (including the unauthenticated v1 API above) can drive arbitrary host-level containers. The repo also ships durable event persistence and a structured secret-redacting log filter — those are real controls — but the unauthenticated surface in front of them is what a security lead should read first.

Scope of Analysis
The in-scope workspace is the V1 application server — a FastAPI app under openhands/app_server/ with the V0 server under openhands/server/ reduced to deprecation shims that re-export from app_server. The agentic core (controller, runtime, LLM client, MCP client) has moved out of this repository into a separate software-agent-sdk package, so the app_server here is a control plane that mounts a FastMCP server at /mcp, exposes a v1 REST API at /api/v1/* (events, sandboxes, secrets, settings, webhooks, git, config, skills), provides three sandbox back-ends (Docker — the documented default, Process — same-host subprocess, Remote — managed cluster), and persists conversation events to filesystem, S3, or GCS. CORS, rate limiting, and cache-control middleware are wired in app.py; no authentication middleware is added by default, and per-router get_dependencies() only enforces an API key when the SESSION_API_KEY environment variable is set or when running in SAAS app mode. Docker compose mounts /var/run/docker.sock into the OpenHands container, and skills are loaded over HTTP from the agent-server inside the sandbox without any signature or hash check.
