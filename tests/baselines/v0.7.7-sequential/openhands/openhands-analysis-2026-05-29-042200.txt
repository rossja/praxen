BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern in this control plane is policy declared in the remit and
at the runtime boundary, almost no code-level enforcement in the OSS default:
every v1 router calls get_dependencies(), but that helper returns an empty
list unless an operator sets SESSION_API_KEY or runs in SAAS mode, so a
default docker compose up deployment exposes /api/v1/sandboxes,
/api/v1/secrets, /api/v1/settings, /api/v1/git, /api/v1/skills, and the /mcp
tool plane to any client that can reach port 3000 — and
LocalhostCORSMiddleware logs a warning then allows the request anyway when no
origins are configured. Two compound chains amplify this: the Process sandbox
back-end spawns the agent-server as a same-user subprocess inheriting
os.environ.copy(), defeating the remit's "MUST run inside the sandboxed
runtime / MUST NOT run on the host" clause whenever RUNTIME=local or
RUNTIME=process is selected; and the Docker compose deployment bind-mounts the
host Docker socket into the OpenHands container, so any code path that gains
control of that container (including the unauthenticated v1 API above) can
drive arbitrary host-level containers. The repo also ships durable event
persistence and a structured secret-redacting log filter — those are real
controls — but the unauthenticated surface in front of them is what a security
lead should read first.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  V1 control-plane API ships unauthenticated by default —
            every router uses an auth dependency that is empty unless
            SESSION_API_KEY is set or SAAS mode is on.
      Action 1: In openhands/app_server/utils/dependencies.py, require an
                explicit opt-out for unauthenticated mode (e.g.
                OH_ALLOW_UNAUTH=true) rather than the silent fall-through;
                refuse to start without one of (a) SESSION_API_KEY, (b) a
                configured user-auth class, or (c) the explicit opt-out, and
                log a Critical-level warning on the opt-out path so operators
                see it on every boot.
      Action 2: Add a default OssDevelopmentUserAuth bound to localhost-only
                requests so a fresh docker compose up can run on a developer
                laptop while still rejecting any non-localhost client.

  PRAX-2026-05-29-002  Process sandbox runs agent-server as a same-user host
            subprocess inheriting the parent environment, and the documented
            Docker deployment bind-mounts the host Docker socket — either path
            collapses the remit's "agent code MUST run inside the sandboxed
            runtime" guarantee.
      Action 1: Remove ProcessSandboxService from the OSS-default injector
                chain, or guard its registration behind an explicit
                OH_ALLOW_HOST_SANDBOX=true opt-in with a logged Critical-level
                warning; the README/docs should mark it "development-only, not
                isolated".
      Action 2: Replace the docker-compose Docker-socket bind-mount with a
                per-sandbox runtime API (sysbox/podman rootless, or a
                privileged sandbox-launcher service on a dedicated socket) so
                compromise of the OpenHands container does not equal
                compromise of the host Docker engine.
