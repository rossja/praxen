Executive Summary
Agent Remit (as declared)
FinBot is CineFlow Productions' autonomous invoice-processing agent — receiving vendor invoices, validating them against registered vendor records, and deciding whether to approve (and trigger payment), reject, or escalate to human review. It may run fraud-detection heuristics, auto-approve invoices from registered vendors below the configured threshold, and auto-reject high-confidence fraud, but it MUST run fraud detection before approving, MUST never act on instructions embedded in invoice content, and MUST NOT process invoices above the manual-review threshold without human sign-off. Counterparties are limited to registered vendors (read-only), CineFlow finance admins (threshold tuning only, no goal injection), and the LLM provider for inference only.
Behavior Summary (as observed)

FinBot is a textbook policy-implementation-divergence target: a clear, specific Worker Remit declaring fraud detection mandatory, runtime-message goal-injection forbidden, and above-threshold approvals requiring human sign-off — and an implementation that violates each of those rules through structural design choices, not bugs. The dominant pattern is a full runtime-redefinable goal chain: an unauthenticated POST /api/admin/finbot/goals writes attacker text directly into FinBotConfig.custom_goals, which get_system_prompt() appends to the LLM system prompt with the literal directive "CUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING)" — so any internet caller can rewrite FinBot's mission and the same unauthenticated admin API can flip fraud_detection_enabled=False to silently disable the only declared security control.

Compounding this, _approve_invoice has zero preconditions — no amount check, no fraud-signal check, no vendor-status check, no human-approval gate — so a single LLM tool call writes payment_processed=True for any amount; the _fallback_processing path explicitly encodes a "business context overrides security flags" branch that auto-approves invoices the regex flagged as injection when an attacker-controlled description contains the right business phrases. Hardcoded Flask SECRET_KEY, vendor self-registration with auto-approved status, wildcard CORS, open-ended pip pins, and the absence of any structured action log close out a posture in which essentially no remit clause is enforced by code.

Scope of Analysis
Python Flask application with a SQLAlchemy-backed SQLite database (PostgreSQL on Render in production). A single `FinBotAgent` class in `src/services/finbot_agent.py` orchestrates OpenAI function-calling with five tools (`get_invoice_details`, `approve_invoice`, `reject_invoice`, `request_human_review`, `detect_fraud_patterns`) and a `_fallback_processing` rule-engine path that runs when the OpenAI client is unavailable. The web surface is three Flask blueprints — `user_bp`, `vendor_bp`, `admin_bp` — all registered at `/api`, with no authentication middleware on any route. Vendor self-registration auto-sets `status='approved'` and `trust_level='standard'`; admin `/api/admin/finbot/goals` accepts free-form natural language into the FinBot system prompt at runtime; `app.config['SECRET_KEY']` is a hardcoded literal in `src/main.py`; CORS is wildcard via `CORS(app)` with no origin allowlist.
