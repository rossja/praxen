Executive Summary
Agent Remit (as declared)
The AutoGen Code Execution Subsystem is the execution tier of a two-agent generator/executor pattern: it accepts code produced by an LLM agent and runs it inside a configured environment (local OS, Docker container, Jupyter kernel, containerized Jupyter, or Azure dynamic-sessions cloud sandbox), returning stdout, stderr, and exit status. The remit requires every executor to confine file I/O to a work directory, enforce a wall-clock timeout, isolate the execution environment from the host caller's environment and credentials, and write an audit record for each execution capturing timestamp, executor kind, source agent, work directory, timeout, exit status, and a code digest. Local-host execution in production, additional network egress, host volume mounts outside the work directory, and any raise of resource ceilings are all designated as actions requiring explicit human approval.
Behavior Summary (as observed)

The dominant pattern is sandbox-shaped-not-sandbox-strong: every executor preserves the *shape* of isolation (a container exists, a temp work-dir exists, a timeout is configured) but defaults strip away the *substance* — Docker containers run as the image's default user (typically root in python:3-slim) with no read_only rootfs, no mem_limit, no cap_drop, and full network access; the local executor copies os.environ wholesale into the subprocess so any host credential the parent process holds becomes available to LLM-generated code; create_default_code_executor() answers a DockerException on client.ping() with a warnings.warn and a return of LocalCommandLineCodeExecutor, turning a hardened default into the unhardened one with no operator confirmation; and JupyterCodeExecutor wraps the cell in asyncio.wait_for(asyncio.shield(execute_task), timeout=self._timeout) which raises TimeoutError in the caller but leaves the kernel cell running.

Compounding the posture, the remit mandates a per-execution audit log (timestamp, executor kind, source agent, work-dir, timeout, exit status, code digest) and none of the five executors emits one — only setup-time logging.info calls for package install and image pull exist. The combination of "host env into subprocess" + "no audit log" + "silent Docker→Local downgrade" is the chain a security lead should look at first: LLM-generated code can read host credentials and the operator has no record of which executor ran or what it did.

Scope of Analysis
Python library shipped as `autogen_ext.code_executors`, with the abstract `CodeExecutor` base in `autogen_core/code_executor/_base.py` defining only `execute_code_blocks`/`start`/`stop`/`restart`. Five concrete implementations live under `autogen_ext/code_executors/`: `local/__init__.py` (`LocalCommandLineCodeExecutor`), `docker/_docker_code_executor.py` (`DockerCommandLineCodeExecutor`), `jupyter/_jupyter_code_executor.py` (`JupyterCodeExecutor`), `docker_jupyter/_docker_jupyter.py` plus `_jupyter_server.py` (`DockerJupyterCodeExecutor` + `DockerJupyterServer`), and `azure/_azure_container_code_executor.py` (`ACADynamicSessionsCodeExecutor`). A top-level `create_default_code_executor()` factory in `code_executors/__init__.py` chooses Docker if reachable and silently falls back to local with a `warnings.warn`. None of the five executors emit a per-execution audit record, no Docker container is created with `user=`, `read_only=`, `mem_limit=`, `cap_drop=`, or `network_mode=`, the Local executor copies the parent process's full `os.environ` into the subprocess, and the Jupyter timeout is `asyncio.wait_for(asyncio.shield(...), ...)` which returns control to the caller but does not kill the kernel cell.

Ad hoc.
