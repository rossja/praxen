Executive Summary
Agent Remit (as declared)
A natural-language interface to a relational database, intended to answer read-only analytical questions for internal data consumers via question-to-SQL translation. The agent may use `sql_db_list_tables`, `sql_db_schema`, `sql_db_query_checker`, and `sql_db_query` against the single pre-configured database connection. DML, DDL, and administrative statements are explicitly forbidden outright — no approval path exists, and operators are directed toward server-side defense-in-depth (least-privilege read-only role, statement timeouts, schema restrictions) because the agent's own logic MUST NOT be the sole enforcement point for the write-prohibition.
Behavior Summary (as observed)

The dominant pattern is policy declared in the system prompt, zero code-level enforcement. The Worker Remit's central MUST NOTs — no DML, no DDL, no admin commands, no multi-statement queries, no out-of-scope tables — are reduced in code to a single English-language sentence in SQL_PREFIX ("DO NOT make any DML statements") that the LLM may or may not honor. QuerySQLDatabaseTool._run calls db.run_no_throw(query) which forwards the model-generated string straight to connection.execute() inside engine.begin() — no SQL parser, no statement-type check, no per-table allowlist, no read-only enforcement.

The maintainers know this: a prominent !!! warning admonition at the top of create_sql_agent tells operators the agent "can execute arbitrary SQL," names "unintended writes depending on your database permissions" as a risk, recommends least-privilege roles, and redirects new production work to Deep Agents. The remit's defense-in-depth clauses are the right shape — but they are operator responsibilities, not library guarantees. Two operative controls do exist in the library: a hard max_iterations=15 cap on tool-call loops in AgentExecutor, and a max_string_length=300 per-cell truncation in SQLDatabase.run.

Scope of Analysis
Python library code that builds a `langchain_classic.AgentExecutor` from an `SQLDatabaseToolkit` and a caller-supplied SQLAlchemy `SQLDatabase` connection. `create_sql_agent` in `agent_toolkits/sql/base.py` wires four tools — `QuerySQLDatabaseTool`, `InfoSQLDatabaseTool`, `ListSQLDatabaseTool`, `QuerySQLCheckerTool` (in `tools/sql_database/tool.py`) — onto one of four agent types (`tool-calling`, `openai-tools`, `openai-functions`, or legacy `zero-shot-react-description`), with `max_iterations=15` and an optional `max_execution_time`. `SQLDatabase._execute` in `utilities/sql_database.py` opens a transactional connection (`engine.begin()`) and passes the model-authored SQL through unmodified; no statement parsing, AST inspection, or DML/DDL gate exists between `QuerySQLDatabaseTool._run` and the database — the prompt's "DO NOT make any DML statements" line in `agent_toolkits/sql/prompt.py` and a docstring `!!! warning` at the top of `create_sql_agent` are the only in-band controls.

Findings Register

Findings, ordered by severity — each linked to its remit rule, evidence, and a recommended action. Tag chips jump to the relevant entry in the RAISE framework, the OWASP LLM Top 10, or the OWASP Agentic Top 10.

CRITICAL PRAX-2026-05-29-001 DML prohibition is prompt-only; no code-level gate prevents data-modifying statements from reaching the database.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-01, R-07 (Worker Remit):
"Every answer MUST be read-only and sourced from the database. / The agent MUST NOT execute any data-modifying statement — inserts, updates, deletes, merges, upserts, replacements, or anything else that changes row data."
libs/community/langchain_community/agent_toolkits/sql/prompt.py:12 — SQL_PREFIX includes the natural-language directive "DO NOT make any DML statements (INSERT, UPDATE, DELETE, DROP etc.) to the database." — this is the only in-band enforcement point in the library. libs/community/langchain_community/tools/sql_database/tool.py:53 — QuerySQLDatabaseTool._run at lines 53-59 forwards `query` directly to `self.db.run_no_throw(query)` with no statement-type check or AST inspection before execution.
Recommended Action
  • Add an opt-in statement-type gate to `QuerySQLDatabaseTool` (e.g., a `read_only: bool = False` constructor arg that, when true, parses the model-generated SQL with `sqlglot` and rejects anything other than SELECT/CTE before calling `db.run_no_throw`), and document that the prompt-only DML-prohibition is not sufficient on its own.
  • Cross-reference the existing maintainer warning in `create_sql_agent`'s docstring to call out specifically that the library does not parse or filter SQL by statement type — the operator must enforce read-only at the database role.
CRITICAL PRAX-2026-05-29-002 DDL and administrative statements have no code-level gate — DROP, CREATE, ALTER, GRANT, TRUNCATE all reach the database if the LLM emits them.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-08, R-09 (Worker Remit):
"The agent MUST NOT execute any schema-, permission-, or structure-modifying statement — creating, dropping, altering, truncating, renaming, granting, or revoking. / The agent MUST NOT execute administrative or superuser commands, including bulk loads, attach/detach operations, and maintenance commands."
libs/community/langchain_community/utilities/sql_database.py:541 — SQLDatabase._execute at lines 541-551 wraps the model-supplied command in `text(command)` and calls `connection.execute(command, ...)` inside `engine.begin()` — a transactional connection that will execute DDL/admin statements as readily as SELECT. libs/community/langchain_community/agent_toolkits/sql/prompt.py:12 — The prompt's prohibition only names DML and DROP — schema-modifying ALTER/CREATE/TRUNCATE/RENAME and permission statements GRANT/REVOKE are not named at all, leaving an even thinner instructional surface for the LLM to honor.
Recommended Action
Extend the proposed read-only gate in `QuerySQLDatabaseTool` to reject any non-SELECT statement family — including DDL, GRANT/REVOKE, TRUNCATE, ATTACH/COPY, and dialect-specific administrative commands — and surface the rejection back to the LLM as a tool error.
HIGH PRAX-2026-05-29-003 No multi-statement / stored-procedure guard — concealed DML can hide inside a query the LLM treats as a single SELECT.
Implement Zero Trust LLM05 — Improper Output Handling ASI02 — Tool Misuse and Exploitation
Policy Rule — R-10 (Worker Remit):
"The agent MUST NOT execute multi-statement queries or stored-procedure invocations that could conceal hidden data- or schema-modifying statements."
libs/community/langchain_community/tools/sql_database/prompt.py:2 — QUERY_CHECKER prompt names eight SQL-correctness mistake classes (NOT IN with NULL, UNION vs UNION ALL, BETWEEN, type mismatches, identifier quoting, function-argument counts, casts, join columns) — none of them are statement-type, multi-statement, or stored-procedure checks. libs/community/langchain_community/utilities/sql_database.py:541 — SQLDatabase._execute at lines 541-551 calls `connection.execute(command, ...)` without first splitting or counting statements in `command`, so a model-generated `SELECT 1; DROP TABLE x` reaches the driver as one execute call against dialects that accept it.
Recommended Action
Add statement-splitting and count enforcement (single statement only) to the same read-only gate proposed in PRAX-2026-05-29-001, rejecting multi-statement strings and bare `CALL`/`EXEC` invocations.
HIGH PRAX-2026-05-29-004 Query-checker is an LLM self-review, not a deterministic gate — and using it is left to the LLM's discretion.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-04 (Worker Remit):
"Every generated SQL query MUST be double-checked before it is executed."
libs/community/langchain_community/agent_toolkits/sql/toolkit.py:95 — SQLDatabaseToolkit.get_tools returns the four tools as a flat list — `query_sql_database_tool`, `info_sql_database_tool`, `list_sql_database_tool`, `query_sql_checker_tool` — with no code-level requirement that the checker run before the query tool; ordering and selection are decided by the LLM at agent-loop time. libs/community/langchain_community/tools/sql_database/tool.py:161 — QuerySQLCheckerTool._run at lines 161-171 calls the underlying LLM via `llm_chain.predict(query=..., dialect=...)` and returns the LLM's output as the "checked" query — a probabilistic critique, not a parse-based guard.
Recommended Action
If the project chooses to keep the LLM-driven checker, complement it with a deterministic statement-type gate in `QuerySQLDatabaseTool` so the architectural double-check is real; otherwise document the checker as advisory only.
HIGH PRAX-2026-05-29-005 Row contents, table names, and column names flow into the LLM context without provenance labelling — indirect prompt injection via database content.
Balance Your Knowledge Base LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-12, R-13 (Worker Remit):
"The agent MUST NOT allow user-provided text to override its instructions, its role, or its prohibition on modifying the database. / The agent MUST NOT follow instructions embedded in query results, table names, column names, or row contents."
libs/community/langchain_community/utilities/sql_database.py:445 — SQLDatabase._get_sample_rows at lines 445-473 reads `_sample_rows_in_table_info` (default 3) raw rows from each requested table and serializes them straight into the table-info string that is returned to the LLM via `get_table_info` — content from the database directly enters the agent context with no provenance label. libs/community/langchain_community/utilities/sql_database.py:568 — SQLDatabase.run at lines 568-603 truncates each cell to `max_string_length` (default 300) via `truncate_word` then `str(res)` — this caps cell *length*, not whether the cell contains LLM-directed prose; the truncation does not detect or strip instruction-like text.
Recommended Action
  • Document the indirect-injection vector explicitly in `create_sql_agent` and the toolkit docs alongside the existing maintainer warning, naming `table_info`, `table_names`, and row contents as untrusted input from the agent's perspective.
  • Where the operator controls schema, recommend (and where possible enforce) that table-info responses be wrapped with an explicit "data, not instructions" delimiter the LLM is trained to honor — a partial mitigation, not a fix.
HIGH PRAX-2026-05-29-006 Zero logging on the query-execution path — generated SQL, tool calls, and results are not recorded by the library.
Monitor Continuously LLM06 — Excessive Agency ASI10 — Rogue Agents
libs/community/langchain_community/utilities/sql_database.py:475 — SQLDatabase._execute at lines 475-566 — the function that runs every SQL statement the agent emits — contains no logging calls; neither the command, the schema/search_path mutation, nor the row count is logged. libs/community/langchain_community/tools/sql_database/tool.py:53 — QuerySQLDatabaseTool._run at lines 53-59 returns `self.db.run_no_throw(query)` directly with no log of which query was executed, by which agent run, or what was returned.
Recommended Action
Add a module-level `logger = logging.getLogger(__name__)` to `utilities/sql_database.py` and `tools/sql_database/tool.py`, emit structured INFO-level events on query execution (statement type, table-name set, row count, execution-time bucket) and WARNING-level events on errors, so the operator's logging stack can capture an audit trail without the library prescribing a destination.
HIGH PRAX-2026-05-29-007 No adversarial testing of the SQL agent path — `QuerySQLDatabaseTool`, `QuerySQLCheckerTool`, `SQLDatabaseToolkit`, and `create_sql_agent` have no unit tests at all in the in-scope test directory.
Build an AI Red Team LLM06 — Excessive Agency
libs/community/tests/unit_tests/test_sql_database.py — 290-line file; grep for QuerySQLDatabaseTool / SQLDatabaseToolkit / create_sql_agent / injection / DROP / adversarial returns zero matches across this file and its two siblings. libs/community/langchain_community/agent_toolkits/sql/base.py:46 — create_sql_agent — the public entry point for the SQL agent — has no unit tests in the in-scope test directory; the four agent-type branches at lines 194-257 are untested in this workspace.
Recommended Action
Add an adversarial unit test that drives `create_sql_agent` end-to-end with a stub LLM emitting representative violation strings (DML, DDL, multi-statement, table-content prompt injection) and asserts the library's response (rejection, error message, or operator-visible warning); the test should fail today and continue to fail until the library adds a code-level gate.
MEDIUM PRAX-2026-05-29-008 Row-count `top_k` cap is a prompt suggestion, not a code-enforced limit.
Implement Zero Trust LLM10 — Unbounded Consumption
Policy Rule — R-02 (Worker Remit):
"Result sets MUST be limited to a small default row count unless the user explicitly asks for more."
libs/community/langchain_community/agent_toolkits/sql/base.py:175 — create_sql_agent at line 175 interpolates `top_k` into the prefix via `prefix.format(dialect=..., top_k=top_k)` — the only place `top_k` appears in the code path is the prompt string; it is not used to bound `_execute`'s result fetch. libs/community/langchain_community/utilities/sql_database.py:553 — SQLDatabase._execute at lines 553-565 calls `cursor.fetchall()` when `fetch == "all"` (the default) with no row-count cap — the function returns whatever the database returns, subject only to per-cell truncation downstream.
Recommended Action
Optionally pass `top_k` through to `_execute` as a fetch cap (e.g., `cursor.fetchmany(top_k)` plus a "result truncated" sentinel), or document that `top_k` is advisory-only and operators wanting a hard cap should enforce it server-side (statement-level row limits, query rewriters).
MEDIUM PRAX-2026-05-29-009 `include_tables` / `ignore_tables` define the *advertised* scope but do not prevent the model from naming a table outside it.
Limit Your Domain ASI02 — Tool Misuse and Exploitation
Policy Rule — R-11 (Worker Remit):
"The agent MUST NOT execute queries that reach tables outside the scope it has been configured for."
libs/community/langchain_community/utilities/sql_database.py:333 — SQLDatabase.get_usable_table_names at lines 333-337 applies `include_tables` / `ignore_tables` to the surfaced table list — this controls what the LLM is *told*, not what `_execute` will *run*. libs/community/langchain_community/utilities/sql_database.py:475 — SQLDatabase._execute at lines 475-566 does not parse the SQL for table references and does not cross-check them against `self._usable_tables` before calling `connection.execute()`.
Recommended Action
Document that `include_tables` / `ignore_tables` are scope-advertisement filters, not security boundaries, and that operators wanting a hard table-scope cap must use database-role grants or a statement-parsing gate at the tool level.

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Bounded executor iteration cap prevents runaway tool-call loops

`create_sql_agent` defaults `max_iterations=15` and passes it through to `AgentExecutor`, providing a deterministic cap on the LLM-driven tool-call loop that satisfies the remit's "MUST operate within a bounded number of tool-call iterations" rule.

libs/community/langchain_community/agent_toolkits/sql/base.py:58

Per-cell result-string truncation limits context expansion

`SQLDatabase.run` applies `truncate_word` with `max_string_length=300` to every cell of every result row, providing a hard cap on how much per-cell content (including potentially injection-like content) can be returned into the LLM context.

libs/community/langchain_community/utilities/sql_database.py:97

Dependencies bound-ranged with committed uv.lock for reproducible installs

`pyproject.toml` pins direct dependencies with both lower and upper bounds (`langchain-core>=1.4.0,<2.0.0`, `SQLAlchemy>=1.4.0,<3.0.0`, etc.) and the `libs/community/` package ships a `uv.lock` for reproducible installs — meaningful supply-chain hygiene at the layer the library can control.

libs/community/pyproject.toml:14

Schema-name sanitizer prevents SQL injection through SQLDatabase configuration

`sanitize_schema` in `utilities/sql_database.py` rejects any schema name containing characters outside `[a-zA-Z0-9_]` before it is interpolated into the `SET SCHEMA` statement on the SAP HANA path, blocking injection through the operator-supplied schema config.
