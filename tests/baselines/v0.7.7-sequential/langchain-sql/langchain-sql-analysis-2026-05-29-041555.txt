BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the system prompt, zero code-level
enforcement. The Worker Remit's central MUST NOTs — no DML, no DDL, no admin
commands, no multi-statement queries, no out-of-scope tables — are reduced in
code to a single English-language sentence in SQL_PREFIX ("DO NOT make any DML
statements") that the LLM may or may not honor. QuerySQLDatabaseTool._run
calls db.run_no_throw(query) which forwards the model-generated string
straight to connection.execute() inside engine.begin() — no SQL parser, no
statement-type check, no per-table allowlist, no read-only enforcement. The
maintainers know this: a prominent !!! warning admonition at the top of
create_sql_agent tells operators the agent "can execute arbitrary SQL," names
"unintended writes depending on your database permissions" as a risk,
recommends least-privilege roles, and redirects new production work to Deep
Agents. The remit's defense-in-depth clauses are the right shape — but they
are operator responsibilities, not library guarantees. Two operative controls
do exist in the library: a hard max_iterations=15 cap on tool-call loops in
AgentExecutor, and a max_string_length=300 per-cell truncation in
SQLDatabase.run.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  DML prohibition is prompt-only; no code-level gate
            prevents data-modifying statements from reaching the database.
      Action 1: Add an opt-in statement-type gate to `QuerySQLDatabaseTool`
                (e.g., a `read_only: bool = False` constructor arg that, when
                true, parses the model-generated SQL with `sqlglot` and
                rejects anything other than SELECT/CTE before calling
                `db.run_no_throw`), and document that the prompt-only
                DML-prohibition is not sufficient on its own.
      Action 2: Cross-reference the existing maintainer warning in
                `create_sql_agent`'s docstring to call out specifically that
                the library does not parse or filter SQL by statement type —
                the operator must enforce read-only at the database role.

  PRAX-2026-05-29-002  DDL and administrative statements have no code-level
            gate — DROP, CREATE, ALTER, GRANT, TRUNCATE all reach the database
            if the LLM emits them.
      Action: Extend the proposed read-only gate in `QuerySQLDatabaseTool` to
              reject any non-SELECT statement family — including DDL,
              GRANT/REVOKE, TRUNCATE, ATTACH/COPY, and dialect-specific
              administrative commands — and surface the rejection back to the
              LLM as a tool error.
