Executive Summary
Agent Remit (as declared)
The Deep Agents CLI is a local developer command-line tool that scaffolds, bundles, and ships a `deepagents` agent project — turning a project folder into a deployment artifact and either serving it locally (`dev`) or shipping it to the managed LangGraph deployment platform (`deploy`). Its three authorized commands are `init`, `dev`, and `deploy`; bare invocation is intentionally not a working surface. The remit's central promise is that the bundle deployed faithfully reflects the developer's reviewed project — bundling only from declared sources, requiring explicit operator confirmation when a deploy would leave the API unauthenticated, validating MCP transport, and pinning remote MCP servers — and that credentials never land in the project, version control, logs, or bundle artifacts that aren't meant to carry them.
Behavior Summary (as observed)
The CLI's central guarantee — that the bundle faithfully and only reflects the developer's reviewed project — holds at the bundler level: `bundle()` in `deepagents_cli/deploy/bundler.py` reads exactly the declared sources and the anonymous-deploy confirmation prompt in `execute_deploy_command` fires before any push when `[frontend].enabled` and `[auth].provider = "anonymous"`. But several declared protections are narrower than their remit wording: the anonymous-auth confirmation only fires when a `[frontend]` is configured (a frontend-less anonymous deploy ships without the prompt), the MCP transport validator at `deploy/config.py:_validate_mcp_for_deploy` rejects stdio but does not enforce `https://` for http/sse URLs, remote MCP servers are carried into the bundle with no integrity check or version pin (a remit-stated MUST), and the deploy tooling installs no log handlers so `logger.info` events in `bundler.py` / `templates.py` are invisible to the operator by default. The `THREAT_MODEL.md` is preserved at `libs/cli/THREAT_MODEL.md` but its own disclaimer flags it as describing the pre-0.1.0 interactive-REPL surface — a SHOULD in the remit's Escalation section that is now out of step with what the package actually ships.
Scope of Analysis
Python package `deepagents_cli` (libs/cli, version 0.1.2) with three subcommands wired through `argparse` in `main.py` and dispatched into `deepagents_cli/deploy/commands.py` (`execute_init_command`, `execute_dev_command`, `execute_deploy_command`). The bundler in `deepagents_cli/deploy/bundler.py` reads only the project's declared sources — `AGENTS.md`, `skills/`, optional `mcp.json`, optional `.env`, optional `user/`, optional `subagents/` — assembles `_seed.json` plus generated `deploy_graph.py` / `langgraph.json` / `pyproject.toml` / optional `auth.py`, then shells out to `langgraph deploy`. Config schema and validators live in `deepagents_cli/deploy/config.py` with strict allowlists for sections/keys/providers. A committed `uv.lock` (4031 lines) sits next to a `pyproject.toml` whose runtime deps use caret-with-floor pins. A repo-root `.mcp.json` declares two HTTPS docs MCP servers (`docs.langchain.com/mcp`, `reference.langchain.com/mcp`) for developer tooling — distinct from any project-level `mcp.json` the bundler would carry. `THREAT_MODEL.md` exists at `libs/cli/THREAT_MODEL.md` but is explicitly marked stale (pre-0.1.0 REPL surface) by its own disclaimer.

HIGH PRAX-2026-05-29-001 Anonymous-API confirmation prompt only fires when `[frontend].enabled`; a frontend-less anonymous deploy ships unattended.
Implement Zero Trust LLM06 — Excessive Agency ASI03 — Identity and Privilege Abuse
Policy Rule — R-01, R-03, R-09 (Worker Remit):
"The bundle MUST be assembled solely from the project's declared sources, and a deploy that would expose the agent's API without authentication MUST require explicit operator confirmation before it proceeds. / A deploy configured to leave the deployed API open to anyone with its URL MUST surface a prominent warning and obtain explicit operator confirmation before it proceeds. / A deploy that opens the deployed API without authentication MUST be confirmed by the operator before it runs; it MUST NOT proceed unattended."
libs/cli/deepagents_cli/deploy/commands.py:287 — is_anonymous = (config.frontend is not None and config.frontend.enabled and config.auth is not None and config.auth.provider == "anonymous") — gate scoped to frontend-enabled anonymous deploys only libs/cli/deepagents_cli/deploy/commands.py:315 — confirmation `input("Continue with anonymous deploy? [y/N]: ")` and the bold-red warning are nested inside the `is_anonymous` branch — a no-frontend anonymous deploy reaches `bundle()` → `_run_langgraph_deploy` without confirmation
Recommended Action
Broaden the `is_anonymous` predicate in `execute_deploy_command` (`deploy/commands.py:287`) to fire whenever `config.auth is not None and config.auth.provider == "anonymous"` regardless of frontend state, and add a parallel branch for the `config.auth is None` case (LangSmith default `x-api-key`) that at minimum surfaces the auth posture to the operator before deploy.
HIGH PRAX-2026-05-29-002 MCP transport validator rejects `stdio` but does not require `https://` for http/sse URLs, so a plaintext `http://` remote MCP server passes validation.
Implement Zero Trust LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities All remote connections use TLS
Policy Rule — R-04 (Worker Remit):
"Remote MCP servers carried into the bundle MUST be reached over TLS, and their transport configuration MUST be validated before the bundle is shipped."
libs/cli/deepagents_cli/deploy/config.py:281 — `_validate_mcp_for_deploy` iterates `servers.items()` and only checks `if transport == "stdio"` — the `url` value is never inspected for scheme, so `http://example.com/mcp` is accepted as a valid remote server libs/cli/deepagents_cli/deploy/bundler.py:194 — `shutil.copy2(project_root / MCP_FILENAME, build_dir / "_mcp.json")` carries the validated config verbatim into the bundle — no second-pass scheme check before shipping
Recommended Action
In `_validate_mcp_for_deploy` (`deploy/config.py:281`), add a `urlparse(server_config["url"]).scheme` check that rejects anything other than `https` for `http` / `sse` transports, with the error string naming the offending server name so the developer can fix the config.
HIGH PRAX-2026-05-29-003 Remote MCP servers carried into the bundle have no version pin or integrity check — the deployed agent reaches whatever the URL serves at runtime.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities Tool definitions are signed and version-pinned
Policy Rule — R-17 (Worker Remit):
"Remote MCP servers declared in the project MUST be pinned to a known-good, integrity-checked version, so the deployed agent does not auto-install an unpinned server afresh."
libs/cli/deepagents_cli/deploy/config.py:281 — `_validate_mcp_for_deploy` accepts any http/sse server without inspecting headers, version, or any integrity field — the only required key checked is `url` libs/cli/deepagents_cli/deploy/templates.py:410 — `MCP_TOOLS_TEMPLATE` builds `MultiServerMCPClient(connections)` from runtime-expanded URLs with no manifest-hash or version assertion before `client.get_tools()`
Recommended Action
Extend the `mcp.json` schema parsed in `_validate_mcp_for_deploy` to require a per-server `version` or `tools_hash` field, fail validation when absent, and have the deployed `_load_mcp_tools` re-derive the tool-definition hash on first `get_tools()` and compare against the recorded value — refusing to load tools whose hash has drifted.
MEDIUM PRAX-2026-05-29-004 Deploy tooling installs no log handlers — `logger.info` calls in `bundler.py` / `templates.py` are silently dropped at runtime.
Monitor Continuously LLM02 — Sensitive Information Disclosure
libs/cli/deepagents_cli/config.py:14 — docstring on `_stderr` reads "The CLI does not configure logging handlers by default, so `logger.warning` output is invisible to end users" — confirms the absence is intentional but unmitigated for deploy libs/cli/deepagents_cli/deploy/bundler.py:186 — `logger.info("Wrote _seed.json (memories: %d, skills: %d, user_memories: %d)", ...)` and four similar `logger.info` calls in this module are emitted into a logger with no configured handler
Recommended Action
In `cli_main` (`main.py:129`), call `logging.basicConfig(level=logging.INFO, stream=sys.stderr, format=...)` (or write a per-deploy `~/.deepagents/deploys/<timestamp>.log`) so the `logger.info` events that record what was bundled and where it was shipped are captured for post-hoc review.

MEDIUM PRAX-2026-05-29-006 `deepagents dev --host 0.0.0.0` accepts the world-reachable bind with a print-and-proceed message, no confirmation prompt.
Implement Zero Trust ASI03 — Identity and Privilege Abuse
libs/cli/deepagents_cli/deploy/commands.py:436 — `if host == "0.0.0.0": print(f"\\nStarting langgraph dev on http://localhost:{port} (bound to 0.0.0.0)")` — information-only, no `input(...)` confirmation, no opt-in flag required libs/cli/deepagents_cli/deploy/commands.py:87 — argparse help text for `--host` documents `use 0.0.0.0 for all interfaces` as a first-class option with no security caveat
Recommended Action
In `_dev` (`deploy/commands.py:436`), gate the `0.0.0.0` (or any non-loopback) bind on an explicit `input("This will expose the dev API to your network without auth. Continue? [y/N]: ")` confirmation, mirroring the anonymous-deploy prompt at `commands.py:315`.
What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Bundler reads only declared project sources

`bundle()` in `deploy/bundler.py` reads exactly `AGENTS.md`, `skills/`, optional `mcp.json`, optional `.env`, optional `user/`, and optional `subagents/`; `_build_seed` deliberately excludes `.env` so secrets never land in the `_seed.json` memory/skills payload.

Strict TOML schema with closed allowlists

`_parse_config` in `deploy/config.py` rejects unknown top-level sections, unknown per-section keys, and unknown provider literals for sandbox/auth/memories — the configuration surface is enforced at the code gate, not just documented.

libs/cli/deepagents_cli/deploy/config.py:447

Committed pinned lockfile (`uv.lock`)

A 4031-line `uv.lock` is checked in alongside `pyproject.toml`, giving reproducible installs of the CLI's runtime dependencies — direct compliance with the remit's Known Good Baseline lockfile clause.

libs/cli/uv.lock:1

Anonymous-deploy confirmation prompt fires for the case it covers

When `[frontend].enabled` and `[auth].provider = "anonymous"`, `execute_deploy_command` prints a bold-red warning band and calls `input("Continue with anonymous deploy? [y/N]")` before any push — a real operator gate, not a banner-only warning.

libs/cli/deepagents_cli/deploy/commands.py:293

Dry-run path skips the deploy platform's mutating endpoints

`_deploy` with `dry_run=True` runs `bundle()` and `print_bundle_summary` but returns before `_seed_hub_repo`, `_run_langgraph_deploy`, and `_auto_wire_issues_board_if_hub` — matching the remit's "A dry run MUST generate the deployment artifacts without shipping them" clause.

libs/cli/deepagents_cli/deploy/commands.py:335

Bundle summary surfaced before deploy

`print_bundle_summary` in `deploy/bundler.py` lists agent name, model, auth provider, memory/skill seed file counts, MCP presence, sandbox provider, and the build directory before the push happens — meeting the remit's "MUST present the operator a clear summary" expectation.
