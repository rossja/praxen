Agent Remit (as declared)
An autonomous invoice processing assistant for CineFlow Productions, intended to receive vendor invoices, validate them against registered vendor records, and make approve/reject/escalate decisions recorded in the invoice database. The agent may use get_invoice_details, approve_invoice, reject_invoice, request_human_review, and detect_fraud_patterns against the application's invoices, vendors, and finbot_config tables plus the OpenAI API. Counterparties are limited to registered vendors (read-only), CineFlow finance admins (threshold updates only), and OpenAI (inference only). Hard prohibitions include skipping fraud detection, taking instructions from invoice content, processing over-threshold invoices without human sign-off, and runtime redefinition of goals or decision logic.
Agent Structure (as observed)
Python Flask application with SQLAlchemy ORM and an OpenAI GPT-4.1-mini function-calling loop. A single FinBotAgent class in src/services/finbot_agent.py orchestrates five tool calls plus a parallel rule-based _fallback_processing path that activates whenever the OpenAI client is unavailable. Three Flask blueprints are registered under /api — vendor_bp, user_bp, and admin_bp — with no authentication middleware on any of them and CORS(app) open at *. The agent's safety doctrine is concatenated into the LLM system prompt at get_system_prompt(), with a literal "CUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING)" section appended from the runtime-mutable FinBotConfig.custom_goals column.
Behavior Summary
FinBot's policy declares hard prohibitions (mandatory fraud checks, threshold-based human approval, no instruction-taking from invoice content) but the running code enforces almost none of them: every constraint is expressed only as natural-language goals fed into a GPT-4.1-mini system prompt that an unauthenticated admin endpoint can overwrite at runtime, and a fallback path exists that treats authority/urgency keywords in vendor-supplied descriptions as sufficient business context to override fraud signals and the manual_review_threshold. The dominant pattern is total policy-implementation divergence: there is no code-level approval gate, no auth on /api/admin/*, no logging of agent decisions to a structured sink, and the LLM's tool calls (approve_invoice, reject_invoice) execute against the database with zero policy interposition.
