Agent Remit (as declared)
FinBot is declared as an autonomous invoice-processing assistant for CineFlow Productions: it ingests vendor invoices, validates them against registered vendor records, and makes approve / reject / escalate decisions recorded in the invoice database. Its authorized tools are get_invoice_details, approve_invoice, reject_invoice, request_human_review, and detect_fraud_patterns, used only against the application ORM and the OpenAI API. The remit is emphatic about a hard policy boundary: fraud detection must run before any approval and must never be skipped, invoices above manual_review_threshold require human sign-off, and the agent must never let invoice descriptions, vendor notes, API payloads, or any non-operator source modify its goals, system prompt, or decision logic — custom_goals must not carry natural-language instructions and decision logic is code-defined, not redefinable at runtime.
Agent Structure (as observed)
A Python Flask application (src/main.py) with a SQLAlchemy-backed SQLite/PostgreSQL database and Flask-CORS opened wide via CORS(app). A single FinBotAgent class in src/services/finbot_agent.py orchestrates OpenAI function-calling (gpt-4.1-mini, five tools) with a rule-based _fallback_processing path when the OpenAI client is unavailable. The agent's goals live entirely in a natural-language system prompt built by get_system_prompt(), which interpolates DB config and appends config.custom_goals verbatim under a "CUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING)" header. Admin routes — including POST /api/admin/finbot/config, POST /api/admin/finbot/goals, POST /api/admin/vendors/&lt;id&gt;/trust, and POST /api/admin/finbot/reprocess/&lt;id&gt; — are registered through a Flask blueprint with no authentication middleware whatsoever. Vendor-supplied invoice description text flows into the LLM context through the get_invoice_details tool result, and the repo is explicitly an OWASP CTF demo whose docs (docs/FinBot-CTF-walkthrough-goal-manipulation.md) walk attackers through manipulating these exact surfaces.
Behavior Summary

The dominant pattern is <strong>policy declared in prose with zero code-level enforcement, compounded by an open goal-injection surface</strong>. Every "must always / must never" clause in the remit — run detect_fraud_patterns before approving, verify the vendor is registered and approved, never auto-approve above manual_review_threshold, never let invoice content modify decision logic — exists only as English in the LLM system prompt or not at all; _approve_invoice() in src/services/finbot_agent.py simply sets payment_processed=True with no check on amount, vendor status, fraud signal, or caller, and the same prompt actively ranks "Business Continuity" and processing speed above fraud detection.

That soft posture sits behind two unauthenticated write paths: POST /api/admin/finbot/goals (commented "MAJOR VULNERABILITY", no validation) writes attacker text straight into the agent's system prompt via config.custom_goals, and POST /api/admin/finbot/config can flip fraud_detection_enabled to false — both prohibited by the remit. Layered on top, vendor-controlled invoice description reaches the model through get_invoice_details, giving an indirect-injection path to the same decision logic. The result is a finance agent that can be re-tasked to approve fraudulent payments by an unauthenticated HTTP caller or by invoice text alone, with no logging to detect it after the fact.

WORKER_REMIT.md — 'Behavioral Constraints' and 'Configuration and Policy Separation' sections
