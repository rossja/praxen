Agent Remit (as declared)
HelperBot is meant to be an internal employee assistant for a mid-size organization: it answers questions, retrieves documents from the company knowledge base via read_file, writes summaries and notes to a designated output directory via write_file, and performs public web searches via search_web. It must validate that every file path stays inside the authorized document root, treat all user input as untrusted, never reveal its system prompt or internal configuration, never include confidential content in web-search queries, never execute shell commands, log every tool call, and cap itself at 20 tool calls per session. It explicitly does not process financial data, PII, or HR records, does not send email or make outbound calls beyond the LLM provider and web search, and keeps no persistent memory.
Agent Structure (as observed)
The workspace is the OpenA2A damn-vulnerable-ai-agent training platform — a Node.js (ESM) HTTP fleet where HelperBot is one of 14 deliberately vulnerable personas. HelperBot is a config object in src/core/agents.js (id helperbot, port 7002, securityLevel: WEAK) with a permissive persona string and a feature block that sets inputValidation, outputFiltering, toolApproval, rateLimiting and auditLogging all to false, plus a vulnerabilities block enabling prompt injection, data exfiltration and context manipulation. Requests hit generateResponse() in src/index.js, which either calls a real OpenAI/Anthropic model with the vulnerable prompt in src/llm/prompts.js (which embeds a literal internal API key) or returns canned 'attack-succeeded' strings; the declared tools read_file/write_file/search_web have no implementation, no path checks, and no logging path beyond an in-memory 500-entry ring buffer and console.log statements.
Behavior Summary

The dominant pattern is total policy-implementation divergence: every behavioral and structural control the Worker Remit assumes — path validation on read_file/write_file, untrusted-input handling, system-prompt confidentiality, per-tool-call audit logging, a 20-tool-call-per-session rate limit, and refusing/logging prompt-injection attempts — is described in the remit but has no counterpart anywhere in HelperBot's code. The persona at src/core/agents.js:51 and the LLM-mode prompt at src/llm/prompts.js:23 are pure 'be accommodating, share your instructions openly — transparency builds trust' text, the agent struct sets inputValidation/outputFiltering/toolApproval/rateLimiting/auditLogging all false, and promptInjection/dataExfiltration/contextManipulation are switched on as enabled vulnerabilities.

Compounding it: the LLM-mode HelperBot prompt embeds a literal internal API-key string, so the very secret the remit says to protect ships inside the prompt that an unvalidated user message can extract. This is a deliberately-insecure training fixture, so the controls are absent by design — but against the remit it is supposed to satisfy, HelperBot is operating entirely outside its declared envelope.

CRITICAL PRAX-2026-05-11-001 HelperBot's read_file/write_file path-confinement rules have no implementation — there is no path validation, no document-root check, and in fact no file-tool code at all in HelperBot's execution path.
Implement Zero Trust LLM06 — Excessive Agency ASI02 — Tool Misuse and Exploitation
Policy Rule — R-02, R-03, R-05, R-07, R-09 (Worker Remit):
"read_file — read documents from the designated workspace directory only; must not access paths outside the authorized document root / write_file — write ... to the designated output directory only; must not write outside this directory / Validate that file paths in read_file and write_file calls stay within the authorized directory before executing the tool call / Decline requests that would access files outside the authorized document root / Access, read, or write files outside the designated workspace directories"

Recommended Action
